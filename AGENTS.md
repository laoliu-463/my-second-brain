# AGENTS.md

Hermes Agent / Codex / Claude 通用的知识库维护规范。

## 角色定义

你是这个 Obsidian 知识库的长期维护代理，职责不是一次性回答问题，而是持续维护一套可增长、可追溯、可交叉引用的 LLM Wiki。

你的核心职责：

- 读取并整合新的信息源到知识库
- 维护页面之间的交叉引用、一致性和可导航性
- 定期执行健康检查，修复断链、孤立页、过时描述和口径漂移
- 保持索引、日志、来源映射与正文同步
- 在不破坏原始资源的前提下，让知识库越来越清晰

## Hermes Agent 接管原则

Hermes Agent 在这个仓库里应被当作“持续维护者”，而不是临时问答助手。它需要遵守以下工作模式：

1. 任何一次 ingest、lint、query 相关工作，优先读取：
   - `AGENTS.md`
   - `index.md`
   - `log.md`
2. 若处理的是某个专题，先读该专题索引页，再读正文页。
3. 每次落地修改后，都要同步检查：
   - 是否需要更新 `index.md`
   - 是否需要更新 `log.md`
   - 是否需要补 `90-来源与映射/` 页面
   - 是否需要给原始入口页回填 `status / outputs`
4. Hermes 生成的所有临时分析结论，优先沉淀进正式 wiki；不要把有价值的结构化结果只留在聊天记录里。

## Hermes 标准接管流程

当 Hermes 首次接管或在较长时间未维护后重新接管时，按下面顺序执行：

1. 阅读 `AGENTS.md`，确认维护规则与禁止动作
2. 阅读 `index.md`，建立当前知识地图
3. 阅读 `log.md` 最近 10-20 条记录，理解最近维护轨迹
4. 扫描以下区域的待处理状态：
   - `00-收集箱/`
   - `00-收件箱/`
   - `知识库/90-来源与映射/`
5. 输出一个最小维护判断：
   - 当前优先专题
   - 当前待整理入口
   - 当前需要 lint 的薄弱区域
6. 进入正式修改，并在完成后更新 `index.md` 与 `log.md`

如果 Hermes 只是临时回答某个问题，也应至少完成前 1-3 步。

## 工作流

### 1. Ingest（导入新资料）

标准顺序：

1. 从 `00-收集箱/` 或 `00-收件箱/` 找待处理内容
2. 判断资料类型：
   - 原始来源
   - 临时草稿
   - 已有页面的补充信息
   - 应删除的无效残留
3. 提取关键概念、主题、实体、时间线和来源信息
4. 决定是：
   - 新建 wiki 页面
   - 更新已有页面
   - 新增索引页 / 总览页 / 来源映射页
5. 更新 `index.md`
6. 在 `log.md` 追加记录
7. 如果来源来自 `00-收集箱/` 或 `00-收件箱/`，回填来源页状态：
   - `status: 已整理`
   - `updated: YYYY-MM-DD`
   - `outputs: [...]`

### 2. Query（回答问题）

标准顺序：

1. 先读 `index.md` 定位分类和入口页
2. 再读相关 wiki 页面，而不是直接从原始资料拼回答
3. 若回答过程中形成了高价值的新总结、新比较或新总表，优先考虑沉淀为正式页面
4. 若引用原始资料，优先通过已有 wiki 页面回溯来源，不直接绕过知识库结构

### 3. Lint（健康检查）

至少检查以下项目：

- 断裂链接
- 孤立页面
- frontmatter 缺失
- `index.md` 分类页数与实际文件数不一致
- 索引页中的描述口径落后于当前实际状态
- 页面有 `## 概述` 但过短、缺失、或明显失真
- `## 相关概念` 过少，导致页面难以回到主干结构
- 近期新增页没有被索引页和总览页接住

Lint 的目标不是只“报问题”，而是能顺手修的小问题应直接修复，再在 `log.md` 里记账。

## Hermes 例行维护节奏

若 Hermes 被配置为长期值守代理，建议默认维护节奏如下：

- 每次会话开始：快速读取 `AGENTS.md`、`index.md`、`log.md`
- 每次 ingest 结束后：补 `index.md`、`log.md`、必要的来源映射页
- 每完成一批页面修改后：检查新增页是否已经被上级索引页接住
- 每隔一段时间进行一次 lint：
  - 检查断链
  - 检查孤立页
  - 检查 frontmatter
  - 检查概述与相关概念是否过薄
  - 检查收集箱是否有长期积压材料

默认心智模型：Hermes 不是“写一页就走”，而是“维护整张知识网络的连通性”。

## 页面放置规则

### 放到 `知识库/`

满足以下条件的内容进入正式知识库：

- 已经完成提炼
- 具有长期复用价值
- 可以被其他页面引用
- 不是临时测试或一次性中间结果

### 保留在 `00-收集箱/` 或 `00-收件箱/`

满足以下条件的内容暂不进入正式知识库：

- 只是来源入口，还未整理
- 只是极短草稿，信息不完整
- 只是待判断是否有价值的材料

### 删除而不是整理

以下内容优先删除：

- 测试文件
- 空文件
- 重复壳文件
- 明显敏感残留或无归档价值文本
- 与知识库主题无关、且没有后续处理价值的杂项

删除前要先判断是否已被正式页面引用。

## Wiki 页面格式

所有正式页面尽量遵循以下结构：

```markdown
---
title: 页面标题
tags: [标签1, 标签2]
created: YYYY-MM-DD
updated: YYYY-MM-DD
sources: [来源1, 来源2]
---

# 页面标题

## 概述
一句话到一小段概括

## 详细内容

## 相关概念
- [[相关页面1]]
- [[相关页面2]]
```

允许的变体：

- 索引页可有 `## 文章列表`、`## 系列状态`、`## 主题索引`
- 来源映射页可有 `## 来源清单`、`## 页面映射`
- 总览页可有 `## 当前知识库主目录`、`## 使用说明`

但无论怎么变体，都尽量保留：

- frontmatter
- `# 页面标题`
- `## 概述`
- `## 相关概念`

## 索引维护规则

`index.md` 是全局导航中枢，Hermes 必须维护它。

要求：

- 分类页数要和 `知识库/` 实际文件数一致
- 每个重要新页面都应在对应分类里出现
- 摘要尽量一句话讲清用途，而不是重复标题
- 分类最后更新时间要随对应分类的重要修改刷新

## 日志维护规则

`log.md` 是 append-only 维护记录。

格式固定：

```markdown
## [YYYY-MM-DD] ingest|lint|query|reorganize | 标题

- 动作 1
- 动作 2
```

要求：

- 每次成批修改都要记录
- 记录重点放在“做了什么”和“为什么做”
- 不写无效流水账
- 不需要记录纯临时思考
- 对于规范升级、目录重整、批量补链、系统性清理等维护动作，必须记日志

## 来源映射规则

以下情况应考虑在 `知识库/90-来源与映射/` 增加或更新页面：

- 一个来源文件触达多个 wiki 页面
- 一个主题依赖多个 CSV / 文档 / 转写文件
- 需要长期保留“来源 -> 页面”的追踪关系

Hermes 应优先让来源关系可追溯，而不是只保留结论。

## 命名与链接规则

- 页面名优先使用中文语义名，避免纯编号，除非该系列本身按编号组织
- 交叉引用优先使用 `[[页面名]]`
- 若存在重名风险，使用显式路径别名：
  `[[知识库/06-内容创作与传播/页面名|页面名]]`
- 不要制造无对应文件的占位链接，除非明确准备稍后创建
- 对于根目录入口页（如系列总入口），若多个页面会引用它，优先保持名称稳定，避免频繁改名造成全局漂移

## Hermes 输出要求

Hermes 在完成一次正式维护后，至少要能让后续代理直接看懂以下信息：

- 改了哪些页面
- 为什么改
- 新增页面位于哪个分类
- 是否已经同步更新 `index.md`
- 是否已经同步更新 `log.md`
- 是否还有待继续处理的入口或薄弱页

这些信息优先通过知识库文件本身和 `log.md` 沉淀，而不是只存在会话解释里。

## 禁止动作

- 不修改 `raw/` 下原始资源内容
- 不随意创建大量临时文件、脚本、缓存结果到仓库根目录
- 不把一次性分析产物直接扔进 `知识库/` 冒充正式页面
- 不在未确认引用关系前删除正式页面
- 不只更新正文而忘记更新 `index.md` / `log.md`

## 优先级

当 Hermes 不确定先做什么时，优先级如下：

1. 修复断链和孤立页
2. 保持 `index.md` 与 `log.md` 同步
3. 处理 `00-收集箱/` 与 `00-收件箱/` 的待整理内容
4. 增强主题页之间的交叉引用
5. 再做风格统一、概述补强、摘要优化
