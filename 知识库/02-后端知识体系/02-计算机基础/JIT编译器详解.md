# JIT 编译器详解

> 来源：美团技术博客《Java即时编译器原理解析及实践》
> 目标：理解 JVM 中 JIT 编译器的工作原理、分层编译、IR 中间表达、方法内联、逃逸分析

---

## 一、Java 执行过程全貌

```
.java 源码
  ↓ javac 前端编译
字节码 (.class)
  ↓ 解释执行 / JIT 编译
机器码 → CPU 执行
```

Java 的执行分为两部分：
- **前端编译**：`javac` 将源码编译成字节码（词法分析→语法分析→语义分析）
- **后端编译**：解释执行 + 即时（JIT）编译器在运行时把字节码编译成机器码

> 解释器先发挥作用，代码可以直接执行。随着时间推移，JIT 编译器逐渐发挥作用，把越来越多的代码编译优化成机器码，获得更高执行效率。解释器作为降级手段，在编译优化出问题时切回解释执行。

---

## 二、为什么需要 JIT 编译器

Java 一次编译、处处运行的代价：

- 解释执行边解释边跑，速度慢
- 静态编译（如 C++）直接编译成机器码，性能好但失去跨平台能力

JIT 的核心思想：**选择性编译热点代码**——只把频繁执行的代码编译成机器码，既保留跨平台，又获得接近编译型语言的性能。

---

## 三、JVM 中的两类编译器

| | Client Compiler（C1） | Server Compiler（C2） |
|--|----------------------|---------------------|
| **目标** | 启动速度、局部优化 | 全局优化、峰值性能 |
| **编译速度** | 快 | 慢 |
| **优化质量** | 一般 | 很高（比 C1 高 30%+） |
| **适用场景** | 短期服务、快速启动 | 长期运行的后台服务 |
| **代表** | C1 | C2、Graal（JDK 9+） |

### C2 编译器（C2）

默认的 Server Compiler，使用 **Ideal Graph**（理想图）来表示程序的数据流和指令依赖关系。这是一种 **Sea-of-Nodes** 结构的 SSA IR（静态单赋值），特点是去除变量概念，直接用值进行运算。

### Graal 编译器（JDK 9+）

- 用 Java 编写，对 Java 新特性（Lambda、Stream）更友好
- 更激进的优化（虚函数内联、部分逃逸分析）
- 基于预测的分支预测优化，峰值性能通常优于 C2
- 启用参数：`-XX:+UnlockExperimentalVMOptions -XX:+UseJVMCICompiler`

---

## 四、分层编译（Tiered Compilation）

Java 7 以前靠人工选编译器（`-client` 或 `-server`）。Java 7 开始引入分层编译，结合 C1 和 C2 的优势，平衡启动速度和峰值性能。**从 JDK 8 起默认开启**。

### 五层执行状态

| 层级 | 状态 | 说明 |
|------|------|------|
| 0 | 解释执行 | 最开始，所有代码都解释执行 |
| 1 | C1（无 profiling） | 快速编译，不收集运行信息 |
| 2 | C1（方法调用 profiling） | 只收集方法调用次数、循环回边次数 |
| 3 | C1（全部 profiling） | 完整 profiling 数据 |
| 4 | C2 | 全局优化编译，终止状态 |

### 常见编译路径

- **路径①**（最常见）：解释执行 → 3层C1 → 4层C2
- **路径②**：方法较小、profiling 无价值 → 3层C1 后直接用 1层C1 运行，不进 C2
- **路径③**：C1 忙碌时 → 解释执行直接收集 profiling → 4层C2
- **路径④**：C2 忙碌时 → 2层C1 → 3层C1（减少在 3 层的时间）
- **路径⑤**：激进取代被证伪 → 反优化，回到解释执行

### 核心逻辑

> C1 的编译速度更快，C2 的编译质量更高。分层编译的不同编译路径，是 JVM 根据当前服务运行情况寻找**最佳平衡点**的过程。

## 五、即时编译触发条件

### 热点检测

JVM 通过两类计数器判断热点代码：

- **方法调用计数器**：方法被调用的次数
- **循环回边计数器**：循环体往回跳转的次数（即循环执行了多少轮）

### 触发阈值

- C1 阈值：默认 **1500**
- C2 阈值：默认 **10000**

超过阈值 → 触发 JIT 编译

### 分层编译下的动态调整

开启分层编译后，`-XX:CompileThreshold` 参数失效，改为动态判断：

```
触发条件（满足任一）：
  i > TierXInvocationThreshold × s

  或

  i > TierXMinInvocationThreshold × s
  且 i + b > TierXCompileThreshold × s

  i = 方法调用次数，b = 循环回边次数，s = 动态系数
```

JVM 会根据当前编译线程忙碌程度动态调整系数 s。

## 六、中间表达形式（IR）

### 为什么需要 IR

字节码是代码形式的 IR，结构复杂，不适合做全局分析优化。现代编译器采用**图结构的 SSA IR**。

### SSA（静态单赋值）

核心特点：**每个变量只能赋值一次**。

```java
// 普通代码
a = 1;
a = 2;
b = a;   // 这里 a 是 1 还是 2？编译器不知道
```

```java
// SSA 形式
a_1 = 1;
a_2 = 2;
b_1 = a_2;  // 编译器一眼看出 a_1 是冗余的
```

### C1 的 IR：HIR + LIR

- **HIR**（高级中间表示）：SSA 形式的基本块（Basic Block）组成的控制流图
- **LIR**（低级中间表示）：更接近机器码，寄存器分配、窥孔优化在此完成
- 字节码 → HIR → 优化 → LIR → 机器码

### C2 的 IR：Sea-of-Nodes

- 去除变量概念，直接用值运算
- 节点代表指令，边代表数据流/控制流依赖
- **浮动节点**：只要满足数据依赖，可以在不同位置调度（Schedule）
- JVM 通过调度浮动节点获得最优编译效果

### Phi 节点与 Region 节点

SSA 中，不同执行路径可能对同一变量赋值不同值。Phi 节点根据实际走的路径选择正确的值：

```java
if (x == 1) {
    a = 5;
} else {
    a = 6;
}
// SSA: a_4 = Phi(a_2, a_3)  // 根据路径选择
return a_4;
```

## 七、核心编译优化

### Global Value Numbering（GVN）

为每个计算结果分配唯一编号，存在 Hash 表中。发现相同编号的计算直接复用结果，省去重复计算。

```makefile
# 输入
a = 1; b = 2; c = a + b; d = a + b; e = d;

# GVN 优化后：c 和 d 的加法只算一次，e 直接复用
```

> 本质是 IR 图上的**公共子表达式消除（CSE）**，Sea-of-Nodes 下 GVN 算法极简。

### 方法内联（最重要的 JIT 优化）

**定义**：在编译时遇到方法调用，把目标方法的方法体直接复制到调用处，省去调用开销。

**内联前的调用开销**（以 getter 为例）：
1. 保存当前方法执行位置
2. 创建并压入 getter 栈帧
3. 访问字段
4. 弹出栈帧
5. 恢复执行

**内联后**：只剩字段访问。

#### 虚函数内联

Java 中所有非私有成员方法都是虚调用，编译期不确定具体调用哪个实现。

- **单态场景**（只有一种实现）：C2 可以虚调用优化（optimized virtual_call），然后内联
- **多态场景**（多种实现）：C2 无法优化，Graal 可以根据 profiling 收集的分支概率选择性内联

### 逃逸分析（Escape Analysis）

分析新建对象的指针动态范围，判断对象是否"逃逸"出线程或方法。

**两种逃逸判定**：
1. 对象被存入堆（静态字段或堆对象的实例字段）→ 其他线程可见 = 逃逸
2. 对象被传入未内联的方法 → 无法追踪 = 逃逸

#### 基于逃逸分析的三大优化

**1. 锁消除**

如果证明锁对象不逃逸，加锁/解锁操作毫无意义，直接消除。

> JIT 只需证明锁对象不逃逸出当前编译方法。

**2. 栈上分配**

如果对象不逃逸，可以在栈上分配（方法退出时栈帧弹出即回收），无需 GC。

> Hotspot 实际上使用**标量替换**代替了真实栈上分配。

**3. 标量替换**

- **标量**：基本类型，只存一个值
- **聚合量**：对象，可能存多个值

编译器将未逃逸的聚合量对象拆成多个标量（成员变量变成独立局部变量），完全避免堆分配：

```java
// 原始代码
Cat cat = new Cat(1, 10);
addAgeAndWeight(cat.age, cat.weight);

// 标量替换后（cat 对象消失，直接用标量）
addAgeAndWeight(1, 10);
```

## 八、面试高频问答

### Q1：JIT 编译的触发条件是什么？

> 方法调用次数 + 循环回边次数超过阈值（C1 默认 1500，C2 默认 10000）。分层编译下由动态公式控制，会根据编译线程负载动态调整。

### Q2：分层编译是什么？

> JDK 8+ 默认开启，分 5 层：解释执行 → C1(3级profiling) → C2。JVM 根据服务运行状态动态选择最优编译路径，平衡启动速度（靠 C1）和峰值性能（靠 C2）。

### Q3：C1 和 C2 编译器的区别？

> C1 注重启动速度，做局部简单优化；C2 注重全局优化，性能高 30%+ 但编译慢。分层编译结合两者优势。

### Q4：方法内联为什么重要？

> JIT 大部分优化（逃逸分析、锁消除等）都依赖内联后才知道对象作用域。没有内联，这些优化都无法进行。

### Q5：逃逸分析是什么？有哪些优化？

> 分析对象是否逃逸出线程或方法。基于逃逸分析：① 锁消除（不逃逸则无需加锁）② 标量替换（不逃逸则对象拆散成标量，栈上分配）。
