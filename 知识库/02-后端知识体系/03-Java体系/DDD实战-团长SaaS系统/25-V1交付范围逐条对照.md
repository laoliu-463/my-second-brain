# V1交付范围 × 系统落地差距 × 验收证据

## 定位说明

本文档是 **V1 交付范围与当前系统落地的对齐结论**，不只回答"做了什么"，重点回答"哪些已验收、哪些缺证据、哪些需要拍板"。

文档分四层：
1. **V1 范围冻结** — 只回答"做不做"
2. **当前系统验收状态** — 只回答"有没有证据"
3. **P0 业务链验收** — 最关键，逐项需要证据
4. **对齐会拍板** — 只放真正需要业务决策的争议点

---

## 状态说明

| 状态 | 含义 |
|------|------|
| ✅ 已验收 | 有完整业务链证据（页面/接口/数据/QA报告） |
| 🟡 待验收 | 代码存在，但缺完整业务链证据 |
| ⚠️ 不阻断 | 有底座/简化版，不作为 V1 P0 阻断项 |
| ❌ 缺口 | V1 必做但当前缺失或未联调 |
| 🚫 不进 V1 | 明确不是 V1 验收范围 |

## 五、核心缺口说明

以下缺口是当前项目已知的、真实阻塞 V1 验收的关键问题：

### 缺口 1（最关键）：P-06 转链 → O-03 归因链路断裂

**现象**：`ProductService.createPromotionLink()` 生成推广链接时，写入 `pick_source_mapping` 表的 `colonelBuyinId` 为 null。

**影响**：real-pre E2E 报告（2026-05-17）显示订单归因 unattributed=100%，导致：
- P0-1 渠道链端到端无法验收
- P0-4 pick_source 渠道归因失败
- P0-6 寄样交作业无法触发

**根因**：写入 mapping 表时 colonelsBuyinId 字段为空，导致后续订单归因无法匹配。

**定位方法**：
```sql
-- 检查 mapping 表 colonelBuyinId 是否有值
SELECT id, pick_source, colonel_buyin_id, user_id, source_type, status
FROM pick_source_mapping
WHERE source_type = 'NATIVE' AND colonel_buyin_id IS NULL;

-- 检查真实订单归因结果分布
SELECT attribution_status, attribution_remark, COUNT(*)
FROM colonelsettlement_order
GROUP BY attribution_status, attribution_remark;
```

---

### 缺口 2：U-09 一人多角色验收证据缺失

**现象**：`SysUserRole` 表支持多角色写入，但未在 `docs/04` 或 `docs/10` 中找到对应的 UI 验证或 QA 报告。

**建议**：用 `channel_leader` 账号（同时有 biz_leader 角色）登录，验证权限是否取并集、数据范围是否取最宽。

---

### 缺口 3：技术栈声明与当前项目不完全一致

V1 交付范围表来源文档建议的技术栈是：FastAPI + SQLAlchemy + Celery + Python 爬虫 + PostgreSQL + Redis。

当前项目实际技术栈是：Spring Boot + MyBatis-Plus + PostgreSQL + Redis + Vue3 + Vite。

**结论**：技术栈与 V1 设计建议**不完全一致**，但业务能力可等价承载 V1 范围。本次对齐以业务验收为准，不因技术栈差异推翻现有 Spring Boot 实现。
