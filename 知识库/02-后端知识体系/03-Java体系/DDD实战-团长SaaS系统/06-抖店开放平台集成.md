# 06-抖店开放平台集成

## 1. 概述

本章记录团长 SaaS 系统与抖音开放平台的集成架构，包括：OAuth2.0 授权流程、Access Token 管理、抖店 SDK 使用、订单同步机制、订单归因逻辑。

**核心矛盾**：抖店开放平台返回的订单数据本身包含推广来源（`pick_source`）和团长归因（`colonel_order_info`）两种信息，系统需要正确识别并将订单归到正确的渠道用户（channel_user）。

## 4. OAuth2.0 授权流程

### 4.1 抖店授权流程

```
商户/团长在抖店开放平台注册应用
         ↓
获取 client_key + client_secret
         ↓
用户访问授权 URL（包含 client_key + redirect_uri + scope）
         ↓
用户确认授权，抖店生成一次性 authorization_code（有效期 10 分钟）
         ↓
后端用 code 换取 access_token + refresh_token
         ↓
access_token 存入 Redis，TTL = expiresIn（抖店返回，通常 2h）
         ↓
refresh_token 存入 Redis，TTL = 14 天
```

### 4.2 Code 换 Token

`DouyinTokenService.exchangeCodeAndBootstrap()` 执行 OAuth2.0 Code 交换：

```java
// DouyinTokenService.java:180-225
public void exchangeCodeAndBootstrap(String appId, String authorizationCode, ...) {
    DouyinAuthGateway.TokenCreateCommand command = new DouyinAuthGateway.TokenCreateCommand(
            authorizationCode,  // 一次性 Code
            "authorization_code",
            testShop, shopId, authId, authSubjectType
    );
    DouyinAuthGateway.TokenPayload payload = douyinAuthGateway.createToken(command);
    cacheTokenPayload(finalAppId, payload);  // 写入 Redis
}
```

对应抖店 API：`POST /token/create`

**异常处理**：若 `sub_code == 31005`，说明 code 不匹配当前 app 或已过期，提示用户重新授权。

### 4.3 Token 缓存

| Key | 内容 | TTL |
|---|---|---|
| `douyin:token:<appId>` | Access Token | = `expiresIn`（抖店返回，通常 2h） |
| `douyin:refresh:<appId>` | Refresh Token | 14 天 |
| `douyin:token:expire_at:<appId>` | 过期时间戳（Unix 秒） | = `expiresIn` |
| `douyin:token:lock:<appId>` | 分布式锁 | 5 分钟 |
| `douyin:token:reauthorize_required:<appId>` | 重新授权标记 | 1 天 |

## 5. Token 自动刷新

### 5.1 双 Token 机制

抖店 Access Token 有效期短（通常 2h），系统采用双 Token 机制：

```
Access Token（短期，2h）→ 用于 API 调用
Refresh Token（长期，14 天）→ 用于刷新 Access Token
```

### 5.2 提前异步刷新

在 Access Token 距离过期不足 5 分钟时，`getValidToken()` 触发异步刷新：

```java
// DouyinTokenService.java:103-105
if (isTokenExpiringSoon(finalAppId)) {
    triggerAsyncRefresh(finalAppId);  // CompletableFuture.runAsync，不阻塞当前请求
}
return token;  // 旧 Token 直接返回，不等待刷新完成
```

刷新本身用 `setIfAbsent` 加 Redis 分布式锁，防止多实例同时刷新：

```java
// DouyinTokenService.java:112
Boolean locked = redisTemplate.opsForValue().setIfAbsent(
        lockKey, "1", Duration.ofMinutes(redisLockMinutes));  // 5 分钟自动过期
if (!Boolean.TRUE.equals(locked)) {
    throw new TokenRefreshInProgressException();  // 抛异常，不阻塞
}
```

### 5.3 Token 刷新流程

```
getValidToken(appId)
    ↓
Redis GET douyin:token:<appId>
    ├── 命中 → 检查过期时间
    │          ├── 距过期 ≥ 5分钟 → 直接返回（快路径）
    │          └── 距过期 < 5分钟 → 触发异步刷新 → 返回旧 Token
    └── 未命中 → refreshToken()
                 ├── setIfAbsent 加锁（分布式锁）
                 │    ├── 加锁失败 → TokenRefreshInProgressException
                 │    │              → waitForToken() 等待 600ms
                 │    │              → 重试 3 次
                 │    └── 加锁成功 → 调用抖店 /token/refresh
                 │                    → 写入 Redis
                 │                    → 删除锁
                 └── refreshToken 缺失 → BusinessException
```

### 5.4 Token 状态查询

`DouyinTokenService.getTokenStatus()` 用于管理后台展示 Token 状态，返回脱敏后的 Token 片段：


## 6. Gateway 模式（本地与三方调用分离）

### 6.1 接口设计

项目采用 **Gateway 模式** 分离本地调用与外部三方调用，符合 DDD 防腐层（ACL）思想：

```
业务层（Service）
    ↓
Gateway 接口（如 DouyinAuthGateway、DouyinOrderGateway）
    ↓
RealDouyinAuthGateway    ← 真实调用抖店 SDK
MockDouyinAuthGateway     ← 测试/开发时返回模拟数据
```

**关键接口**：

| 接口 | 职责 |
|---|---|
| `DouyinAuthGateway` | Token 管理（创建/刷新/查询） |
| `DouyinOrderGateway` | 订单查询（结算订单列表） |
| `DouyinActivityGateway` | 活动管理（创建/查询活动） |
| `DouyinProductGateway` | 商品管理（物料查询/入库） |

### 6.2 条件注入（Test 模式切换）

```java
// RealDouyinAuthGateway.java:17
@ConditionalOnProperty(name = "douyin.test.enabled", havingValue = "false", matchIfMissing = true)
```

`douyin.test.enabled=true` 时，`RealDouyinAuthGateway` 不注入，业务层得到的是 `Mock` 实现，所有抖店 API 调用返回模拟数据。

## 7. 订单同步机制

### 7.1 同步入口

| 触发方式 | 方式 |
|---|---|
| 定时同步 | `OrderSyncJob`（`@Scheduled`，每分钟执行） |
| 手动触发 | 前端调用 `POST /api/orders/sync/trigger` |

### 7.2 同步时间窗口

```
每次同步范围：[lastSyncTime - 60s, now - 60s]

WINDOW_SECONDS   = 600   （窗口 10 分钟）
OVERLAP_SECONDS  =  60   （重叠 1 分钟，防止跨窗口漏单）
LAG_SECONDS      =  60   （延迟 1 分钟，等抖店数据稳定）

lastSyncTime = Redis KEY "order:sync:last_time"
```

**关键**：下次从 `lastSyncTime - 60s` 开始，而非从 `lastSyncTime`，确保重叠区域不漏单。

### 7.3 分页拉取

```java
// OrderSyncService.java:167-231
String cursor = "0";
boolean hasMore = true;
int pages = 0;
while (hasMore) {
    DouyinOrderGateway.OrderListResult response =
            douyinOrderGateway.listSettlement(new DouyinOrderQueryRequest(startTime, endTime, 100, cursor));
    // cursor 游标翻页，不依赖 offset
    hasMore = response.hasMore() && pages < MAX_PAGES;  // MAX_PAGES = 200，上限 2 万条/次
    pages++;
}
```

### 7.4 同步锁

`OrderSyncService` 使用 Redis `setIfAbsent` 防止 Job 并发：

```java
// OrderSyncService.java:113-127
Boolean locked = redisTemplate.opsForValue().setIfAbsent(
        SYNC_LOCK_KEY, "1", Duration.ofMinutes(10));
if (!Boolean.TRUE.equals(locked)) {
    return new SyncResult(..., locked=true);  // 跳过本次
}
```

**已知 Bug**：`OrderSyncJob` 中 `result.locked()` 返回 true 时，Job 并不重试，直接跳过，存在漏同步风险。

### 7.5 同步后处理

每条订单经过：
1. `AttributionService.resolveAttribution()` — 归因
2. `OrderSyncPersistenceService.persistOrder()` — 写入数据库（insert 或 update）
3. 人名补全（通过 `channelUserId` / `colonelUserId` 查 `SysUser`）

## 8. 订单归因逻辑（核心）

### 8.1 归因本质

订单归因是判断"**这笔订单是谁带来的**"的过程。团长 SaaS 系统存在两类归因通道：

**通道一：外部推广链接（`pick_source`）**
用户通过团长分享的推广链接下单，订单中带有 `pick_source` 参数。

**通道二：抖店原生归因（`colonel_order_info`）**
用户通过视频/直播间直接下单，抖店在 `colonel_order_info` 中记录 `colonel_buyin_id`。

### 8.2 数据来源

抖店 API `buyin.instituteOrderColonel` 返回原始 JSON，字段结构如下：

```json
{
  "order_id": "1234567890",
  "pick_source": "https://haibian.com/p/abc123",
  "colonel_order_info": {
    "colonel_buyin_id": "12345",
    "colonel_activity_id": "67890"
  },
  "colonel_order_info_second": {
    "second_colonel_buyin_id": "...",
    "second_colonel_activity_id": "..."
  },
  "talent_uid": "987654321",
  "product_id": "111222333"
}
```

### 8.3 数据规范化（AttributionSourceNormalizer）

`RealDouyinOrderGateway.toOrderItem()` 在构建 `DouyinOrderItem` 前，先调用 `AttributionSourceNormalizer.normalize()` 将嵌套字段拍平：

```java
// AttributionSourceNormalizer.java:13-43
public static Map<String, Object> normalize(Map<String, Object> source) {
    Map<String, Object> normalized = new LinkedHashMap<>(source);
    // colonel_order_info 拍平到根级别
    flattenColonelInfo(normalized, asMap(source.get("colonel_order_info")),
            "colonel_buyin_id", "colonel_activity_id");
    // colonel_order_info_second 拍平
    flattenColonelInfo(normalized, asMap(source.get("colonel_order_info_second")),
            "second_colonel_buyin_id", "second_colonel_activity_id");
    return normalized;
}
```

这样后续归因逻辑可以直接从根级别取 `colonel_buyin_id`，无需处理嵌套。

### 8.4 归因决策树（resolveAttribution 完整流程）

```
resolveAttribution(order, source)
│
├─ productId 为空 → UNATTRIBUTED（REASON: PRODUCT_NOT_FOUND）
│
├─ 从 source 提取 colonelsBuyinId / secondColonelsBuyinId
│
├─ colonelsBuyinId 或 secondColonelsBuyinId 有值？
│   ├─ 是 → resolveNativeColonelAttribution()  【通道二优先判断】
│   │        ├─ 三级匹配：colonelBuyinId + activityId + productId（最精确）
│   │        ├─ 二级匹配：activityId + productId（fallback）
│   │        └─ 一级匹配：仅 colonelBuyinId（兜底，allowGenericFallback）
│   │   返回 ATTRIBUTED 或 UNATTRIBUTED
│   │
│   └─ 否 → pickSource 和 pickExtra 都为空？
│            ├─ 是 → UNATTRIBUTED（REASON: NO_PICK_SOURCE）
│            │         【抖店自然流量，无法归因】
│            └─ 否 → findPickSourceMapping()  【通道一】
│                     ├─ pickSource 精确匹配 → 命中
│                     ├─ pickExtra 精确匹配 → 命中
│                     └─ pickExtra 末 20 位短 ID 匹配 → 命中
│                  返回 ATTRIBUTED 或 UNATTRIBUTED
│
└─ 独家归属（商品维度）
   ├─ 商家独家归属（merchantId → exclusiveMerchantOwner）
   │   └─ 有归属商家 → ATTRIBUTED
   └─ 达人独家归属（talentUid → exclusiveTalentOwner）
       └─ 有归属达人 → ATTRIBUTED
```

### 8.5 通道一：pick_source 归因（外部推广链接）

当订单通过团长分享的推广链接产生时，`pick_source` 字段携带链接标识。

**匹配优先级**：

```java
// AttributionService.java:359-387
PickSourceMapping findPickSourceMapping(String pickSource, String pickExtra) {
    // 1. pickSource 精确匹配
    PickSourceMapping byPickSource = pickSourceMappingMapper.selectOne(
            .eq(PickSourceMapping::getPickSource, pickSource)
            .eq(PickSourceMapping::getStatus, 1));
    if (byPickSource != null) return byPickSource;

    // 2. pickExtra 精确匹配
    PickSourceMapping byPickExtra = pickSourceMappingMapper.selectOne(
            .eq(PickSourceMapping::getPickExtra, pickExtra)
            .eq(PickSourceMapping::getStatus, 1));
    if (byPickExtra != null) return byPickExtra;

    // 3. pickExtra 末 20 位短 ID 匹配
    String normalized = pickExtra.substring(pickExtra.length() - 20);
    return pickSourceMappingMapper.selectOne(
            .eq(PickSourceMapping::getShortId, normalized)
            .eq(PickSourceMapping::getStatus, 1));
}
```

**`PickSourceMapping` 表结构关键字段**：

| 字段 | 说明 |
|---|---|
| `pick_source` | 推广链接全量（URL） |
| `pick_extra` | 推广链接扩展参数 |
| `short_id` | 链接短 ID（pick_extra 末 20 位） |
| `user_id` | 归属的渠道用户 ID（最终归因到这个用户） |
| `dept_id` | 归属部门 ID |
| `activity_id` | 关联活动 ID |
| `source_type` | `NATIVE`（抖店原生）或普通 |
| `status` | 1=有效 |

### 8.6 通道二：colonel_order_info 归因（抖店原生）

当用户通过视频/直播间下单（无外部推广链接），抖店在 `colonel_order_info` 中携带团长身份信息。

**匹配三级精确度**：

```java
// AttributionService.java:270-326
// Level 1：colonelBuyinId + activityId + productId（三级全匹配）
PickSourceMapping exactNative = selectList(
    .eq(PickSourceMapping::getColonelBuyinId, colonelsBuyinId)
    .eq(PickSourceMapping::getActivityId, activityId)
    .eq(PickSourceMapping::getProductId, productId)
    .eq(PickSourceMapping::getSourceType, "NATIVE")
    .eq(PickSourceMapping::getStatus, 1));

// Level 2：仅 activityId + productId（无 colonelBuyinId）
PickSourceMapping exactActivityProduct = selectList(
    .eq(PickSourceMapping::getActivityId, activityId)
    .eq(PickSourceMapping::getProductId, productId)
    .eq(PickSourceMapping::getSourceType, "NATIVE")
    .eq(PickSourceMapping::getStatus, 1));

// Level 3：仅 colonelBuyinId（兜底）
PickSourceMapping generic = selectList(
    .eq(PickSourceMapping::getColonelBuyinId, colonelsBuyinId)
    .eq(PickSourceMapping::getSourceType, "NATIVE")
    .eq(PickSourceMapping::getStatus, 1));
```

**双团长场景**（`colonel_order_info_second`）：
系统同时支持主副两个团长的归因，`resolveNativeColonelAttribution()` 先解析主团长，主未命中则解析副团长。

### 8.7 归因结果

```java
// AttributionService.java:450-461
public record AttributionResult(
        UUID channelUserId,    // 渠道用户（团长小助手账号）
        UUID deptId,           // 部门
        UUID userId,           // 用户（通常等于 channelUserId）
        UUID talentId,         // 达人 UUID（通过 talentUid 查 Talent 表）
        String talentUid,      // 达人抖店 UID（原始值）
        String activityId,     // 活动 ID
        UUID colonelUserId,    // 团长用户 ID（来自 ProductOperationState）
        String attributionStatus,  // "ATTRIBUTED" 或 "UNATTRIBUTED"

```

**归因原因码**（`attributionRemark`）：

| 原因码 | 含义 |
|---|---|
| `ATTRIBUTED` | 成功归因 |
| `NO_PICK_SOURCE` | 无推广链接，抖店自然流量，无法归因 |
| `MAPPING_NOT_FOUND` | pick_source 在映射表中未找到 |
| `COLONEL_MAPPING_NOT_FOUND` | colonel_buyin_id 在映射表中未找到 |
| `COLONEL_MAPPING_AMBIGUOUS` | 映射有多条，模糊匹配 |
| `PRODUCT_NOT_FOUND` | 订单中 productId 为空 |
| `ACTIVITY_NOT_FOUND` | activityId 在 ProductOperationState 表中不存在 |
| `CHANNEL_NOT_FOUND` | 映射到的人 channelUserId 为空 |
| `COLONEL_ORDER_INFO` | 通过 colonel_order_info 归因成功（成功归因） |

## 9. 已知归因问题

### 9.1 colonelBuyinId 写入 Bug（已确认）

**问题**：`ProductService.createPromotionLink()` 在创建推广链接时，`PickSourceMapping.colonelBuyinId` 字段被写为 `null`，导致映射表中有 `activityId + productId` 但缺少 `colonelBuyinId`。

**影响**：归因走不通 Level 1 和 Level 2，只能走 Level 3（仅靠 colonelsBuyinId），大量订单无法正确归因到对应团长。

**修复方向**：在创建推广链接时，补全 `colonelBuyinId` 字段（从抖店授权信息或用户选择中获取）。

### 9.2 NO_PICK_SOURCE 订单无法归因（业务限制）

**问题**：202 个 NO_PICK_SOURCE 订单是用户在抖店内自然流量产生（无推广链接），系统无法归因。

**影响**：这是**业务限制**而非接口 Bug，无法通过代码修复。

**可能的业务解决方向**：
- 引导用户在推广链接中下单（增强用户教育）
- 与抖店协商获取更多归因字段
