# 05-认证授权体系

## 1. 架构总览

项目采用 **JWT 无状态认证 + Spring Security 过滤器链 + AOP 鉴权** 三层架构，Token 过期后用 Refresh Token 无感刷新，Redis 集中管理黑名单。

```
请求 → JwtAuthenticationFilter（Spring Security 链）
              ↓（通过）
       → JwtAuthInterceptor（WebMvc HandlerInterceptor）
              ↓（通过）
       → Controller
              ↓（如有 @RequireRoles）
       → RoleGuardAspect（AOP 拦截方法级角色校验）
              ↓（如有 @DataScope）
       → DataScopeAspect（AOP 拦截 Mapper 级数据范围注入）
```

| 层次 | 组件 | 职责 |
|---|---|---|
| 认证层 | `JwtAuthenticationFilter` | Spring Security 链中的过滤器，解析 JWT、写入 SecurityContext、拦截未认证请求 |
| 认证层 | `JwtAuthInterceptor` | MVC HandlerInterceptor，与 Filter 功能重复，作为双重保险 |
| 授权层 | `RoleGuardAspect` | `@RequireRoles` 注解 AOP，方法/类级别角色校验 |
| 数据层 | `DataScopeAspect` | `@DataScope` 注解 AOP，Mapper 参数自动追加数据范围条件 |
| Token 管理 | `JwtTokenProvider` | HS256 签名生成与解析，Access Token + Refresh Token 双 Token |
| 会话管理 | `Redis` | Token 黑名单、Refresh Token 吊销列表 |
| 密码 | `BCryptPasswordEncoder` | 密码哈希 |

**源码**：`AuthService.login()`（`auth/service/AuthService.java:52-113`）

```java
public LoginResponse login(LoginRequest request) {
    // ① 查用户
    SysUser user = sysUserMapper.findByUsername(request.getUsername())
            .orElseThrow(() -> new BusinessException(ResultCode.UNAUTHORIZED.getCode(), "用户名或密码错误"));

    // ② 账号状态
    if (user.getStatus() == null || user.getStatus() != 1)
        throw new BusinessException(ResultCode.UNAUTHORIZED.getCode(), "账号已停用");

    // ③ 密码校验（BCrypt）
    if (!matchesPassword(request.getPassword(), user.getPassword()))
        throw new BusinessException(ResultCode.UNAUTHORIZED.getCode(), "用户名或密码错误");

    // ④ 查角色列表
    List<SysRole> roles = sysRoleMapper.findByUserId(user.getId());

    // ⑤ 计算 dataScope（取角色中最大值；OPS_STAFF/ADMIN 强制 3）
    int dataScope = roles.stream()
            .map(SysRole::getDataScope)
            .filter(scope -> scope != null && scope > 0)
            .max(Integer::compareTo).orElse(1);
    if (roleCodes.contains(RoleCodes.OPS_STAFF) || roleCodes.contains(RoleCodes.ADMIN))
        dataScope = 3;

    // ⑥ 生成双 Token
    String accessToken = jwtTokenProvider.generateAccessToken(
            user.getId(), user.getDeptId(), dataScope, roleCodes, user.getUsername());
    String refreshToken = jwtTokenProvider.generateRefreshToken(user.getId());

    // ⑦ 更新最后登录时间
    sysUserMapper.updateById(update);

    return LoginResponse.builder()
            .token(accessToken)       // Access Token（默认 2 小时有效期）
            .refreshToken(refreshToken)  // Refresh Token（默认 7 天）
            .userId(user.getId())
            .deptId(user.getDeptId())
            .dataScope(dataScope)
            .roleCodes(roleCodes)
            .username(user.getUsername())
            .realName(user.getRealName())
            .expiresIn(jwtTokenProvider.getExpireSeconds())
            .refreshExpiresIn(jwtTokenProvider.getRefreshExpireSeconds())
            .build();
}
```

**源码**：`AuthService.refreshToken()`（`auth/service/AuthService.java:115-175`）

```java
public RefreshResponse refreshToken(RefreshRequest request) {
    String refreshToken = request.getRefreshToken();

    // ① 解析 JWT
    Claims claims = jwtTokenProvider.parseClaims(refreshToken);
    if (!"refresh".equals(claims.get("type", String.class)))
        throw new BusinessException(ResultCode.UNAUTHORIZED.getCode(), "Token 类型错误，需要 refresh token");

    // ② 检查 Redis 黑名单（Refresh Token 吊销后不可再用）
    String tokenHash = jwtTokenProvider.getTokenHash(refreshToken);
    if (Boolean.TRUE.equals(redisTemplate.hasKey(REDIS_REFRESH_PREFIX + tokenHash)))
        throw new BusinessException(ResultCode.UNAUTHORIZED.getCode(), "Refresh Token 已吊销");

    // ③ 查用户状态
    UUID userId = UUID.fromString(claims.getSubject());
    SysUser user = sysUserMapper.selectById(userId);
    if (user == null || user.getStatus() != 1)
        throw new BusinessException(ResultCode.UNAUTHORIZED.getCode(), "账号已停用");

    // ④ 生成新 Access Token（Refresh Token 不刷新，返回原 token）
    String newAccessToken = jwtTokenProvider.generateAccessToken(...);

    return RefreshResponse.builder()
            .accessToken(newAccessToken)
            .refreshToken(refreshToken)  // 原 Refresh Token 继续使用
            .build();
}
```

**源码**：`AuthService.logout()`（`auth/service/AuthService.java:177-210`）

```java
public void logout(LogoutRequest request) {
    String accessToken = request.getAccessToken();

    // Access Token 黑名单（TTL = 剩余有效期，过期后自动失效）
    String accessTokenHash = jwtTokenProvider.getTokenHash(accessToken);
    long remainingSeconds = jwtTokenProvider.getRemainingSeconds(accessToken);
    redisTemplate.opsForValue().set(
            REDIS_BLACKLIST_PREFIX + accessTokenHash, "1", remainingSeconds, TimeUnit.SECONDS);

    // Refresh Token 吊销
    String refreshToken = request.getRefreshToken();
    if (refreshToken != null) {
        String refreshTokenHash = jwtTokenProvider.getTokenHash(refreshToken);
        long refreshRemaining = jwtTokenProvider.getRemainingSeconds(refreshToken);
        redisTemplate.opsForValue().set(
                REDIS_REFRESH_PREFIX + refreshTokenHash, "1", refreshRemaining, TimeUnit.SECONDS);
    }
}
```

public JwtTokenProvider(
        @Value("${security.jwt.secret}") String secret,      // 必须 ≥ 32 字符
        @Value("${security.jwt.expire-seconds:7200}") long expireSeconds,        // 默认 2h
        @Value("${security.jwt.refresh-expire-seconds:604800}") long refreshExpireSeconds  // 默认 7d

    this.refreshExpireSeconds = refreshExpireSeconds;

### 4.2 JwtAuthenticationFilter（OncePerRequestFilter）

```java
// JwtAuthenticationFilter.java:66-133
protected void doFilterInternal(HttpServletRequest request,
        HttpServletResponse response, FilterChain filterChain) {

    // ① 提取 Bearer Token
    String authorization = request.getHeader("Authorization");
    if (authorization == null || !authorization.startsWith("Bearer ")) {
        writeUnauthorized(response, "缺少或非法的 Authorization 头");
        return;
    }
    String token = authorization.substring("Bearer ".length()).trim();

    // ② 解析 JWT 并校验 type
    Claims claims = jwtTokenProvider.parseClaims(token);
    if (!"access".equals(claims.get("type", String.class))) {
        writeUnauthorized(response, "Token 类型错误，请使用 access token");
        return;
    }

    // ③ 检查黑名单
    String tokenHash = jwtTokenProvider.getTokenHash(token);
    if (authService.isTokenBlacklisted(tokenHash)) {
        writeUnauthorized(response, "Token 已吊销");
        return;
    }

    // ④ 写入 SecurityContext（Spring Security 依赖）
    UsernamePasswordAuthenticationToken authentication =
            new UsernamePasswordAuthenticationToken(userId, null, Collections.emptyList());
    SecurityContextHolder.getContext().setAuthentication(authentication);

    // ⑤ 写入 Request Attribute（Controller 直接获取）
    request.setAttribute("userId", userId);
    request.setAttribute("deptId", deptId);
    request.setAttribute("dataScope", dataScope);
    request.setAttribute("roleCodes", roleCodes);
    request.setAttribute("username", username);

    filterChain.doFilter(request, response);
}
```

**设计说明**：`userId`、`deptId`、`dataScope`、`roleCodes` 同时写入 SecurityContext 和 Request Attribute 两处——前者满足 Spring Security 的接口约束，后者方便 Controller 直接用 `@RequestAttribute` 获取，无需每次从 SecurityContext 解析。

### 5.3 RoleGuardAspect 切面逻辑

```java
// RoleGuardAspect.java:29-46
@Around("execution(* com.colonel.saas.controller..*(..))")
public Object guard(ProceedingJoinPoint point) throws Throwable {
    RequireRoles requireRoles = resolveRequireRoles(point);
    if (requireRoles == null || requireRoles.value().length == 0)
        return point.proceed();  // 无注解，直接放行

    Set<String> currentRoles = resolveCurrentRoles();  // 从 request attribute 读

    // ADMIN 永远是绿灯
    if (currentRoles.contains(RoleCodes.ADMIN))
        return point.proceed();

    // 检查是否持有任一必需角色
    for (String role : requireRoles.value()) {
        if (currentRoles.contains(normalize(role)))
            return point.proceed();
    }

    throw new ForbiddenException("无权限访问该接口");
}
```

### 5.4 菜单权限分配流程

**Admin 初始化**：建表时通过 `CROSS JOIN` 自动为 admin 角色分配全部菜单。

// SysMenuService.assignMenusToRole()（核心逻辑）
public void assignMenusToRole(UUID roleId, List<UUID> menuIds, UUID operatorId) {
    // ① 删除旧关联
    sysRoleMenuMapper.deleteByRoleId(roleId);

    // ② 批量插入新关联
    List<SysRoleMenu> newMenus = menuIds.stream()
            .map(menuId -> SysRoleMenu.builder()
                    .roleId(roleId)

### 6.1 三级数据范围

### 6.2 @DataScope 注解

```java
// annotation/DataScope.java
@Target(ElementType.METHOD)
@Retention(RetentionPolicy.RUNTIME)
public @interface DataScope {
    /**
     * PERSONAL 范围对应的用户字段名，默认 user_id。
     * 用于自动 eq() 追加条件。
     */
    String userField() default "user_id";
}
```

**使用方式**（标注在 Mapper 方法上）：

```java
@Select("SELECT * FROM colonelsettlement_order")

```

### 6.3 DataScopeAspect 切面逻辑

```java
// DataScopeAspect.java:25-51
@Around("@annotation(dataScope)")
@SuppressWarnings({"rawtypes", "unchecked"})
public Object around(ProceedingJoinPoint point, DataScope dataScope) throws Throwable {
    AbstractWrapper<?, ?, ?> wrapper = findWrapper(point.getArgs());
    if (wrapper == null) return point.proceed();  // 无 wrapper，不处理

    ScopeContext context = resolveContext();  // 从 request attribute 读 userId/deptId/dataScope
    if (context == null || context.scope() == null) return point.proceed();

    if (scope == DataScope.PERSONAL) {
        wrapper.eq(dataScope.userField(), context.userId());   // WHERE user_id = ?
    } else if (scope == DataScope.DEPT) {
        wrapper.eq("dept_id", context.deptId());              // WHERE dept_id = ?
    }
    // ALL：不追加任何条件

    return point.proceed();
}
```

## 9. 关键设计决策

### 9.1 双 Token 而非单一 Access Token

| | Access Token | Refresh Token |
|---|---|---|
| 有效期 | 短（默认 2h） | 长（默认 7d） |
| 存储位置 | 内存（前端不持久化） | HttpOnly Cookie 或 localStorage |
| 用途 | 访问接口 | 静默刷新 Access Token |
| 吊销粒度 | 可精确吊销（Redis 黑名单 TTL） | 可精确吊销（Redis 黑名单 TTL） |

### 9.2 黑名单存 Redis 而非 JWT 注销列表

选择 Redis 黑名单而非"Token 注销列表"（数据库存所有已签发 Token）的理由：

| | Redis 黑名单 | Token 注销列表 |
|---|---|---|
| 存储量 | 仅已吊销 Token | 所有已签发 Token |
| 校验复杂度 | O(1) Redis GET | O(n) 数据库扫描 |
| 过期处理 | TTL 自动失效 | 需定期清理 |
| 适用场景 | 无状态 JWT + 短 Access Token | 有状态 Token |

### 9.3 JwtAuthInterceptor 与 Filter 功能重复

两者逻辑几乎完全一致，是有意为之的纵深防御（defense in depth）：
- Filter 在 Spring Security 链最外层，所有请求都经过
- Interceptor 在 DispatcherServlet 之后，在 Controller 之前
- 只要其中一个能拦截无效 Token，非法请求就无法到达业务层

### 9.4 dataScope 写入 JWT 而非每次查库

用户登录时计算 `dataScope` 并写入 JWT Payload，验证时直接从 JWT 解析，无需每次请求都查 `sys_role` 表。

### 11.1 参数校验

**技术**：jakarta.validation + `@Valid` 注解 + `GlobalExceptionHandler` 统一处理。

**全局异常处理**（`GlobalExceptionHandler`）：

| 异常类型 | HTTP 状态码 | 返回 code | 处理逻辑 |
|---|---|---|---|
| `BusinessException` | 200（普通）/ 401（未授权） | 业务 code | 按 `e.getCode()` 决定 HTTP 状态 |
| `DouyinApiException` | 200 | 500 | 格式化错误信息返回 |
| `ForbiddenException` | 200 | 403 | `ResultCode.FORBIDDEN` |
| `MethodArgumentNotValidException` | 200 | 400 | 取第一个字段错误消息 |
| `BindException` | 200 | 400 | 取第一个字段错误消息 |
| `ConstraintViolationException` | 200 | 400 | 取第一个约束违反消息 |
| `MethodArgumentTypeMismatchException` | 200 | 400 | "参数名 格式不正确" |
| `Exception`（兜底） | 200 | 500 | 写 error 日志，返回通用错误 |

```java
// GlobalExceptionHandler.java:40-56
@ExceptionHandler({ValidateException.class, MethodArgumentNotValidException.class,
        BindException.class, ConstraintViolationException.class})
public ApiResult<Void> handleValidate(Exception e) {
    String msg = "参数校验失败";
    if (e instanceof MethodArgumentNotValidException ex
            && ex.getBindingResult().getFieldError() != null) {
        msg = ex.getBindingResult().getFieldError().getDefaultMessage();
    } else if (e instanceof ConstraintViolationException ex) {
        var violations = ex.getConstraintViolations();
        if (!violations.isEmpty()) {
            msg = violations.iterator().next().getMessage();
        }
    }
    return ApiResult.of(ResultCode.PARAM_ERROR.getCode(), msg, null);
}
```

### 11.3 操作审计日志

**拦截器**：`OperationLogInterceptor`（`HandlerInterceptor`），在 `afterCompletion` 时记录所有 POST/PUT/PATCH/DELETE 请求。

**记录字段**（`OperationLog` 实体）：

| 字段 | 来源 | 说明 |
|---|---|---|
| `userId` / `username` | Request Attribute（JWT 解析后） | 操作人身份 |
| `module` | `@Tag` 注解 或 URL 首段 | 模块名（如"系统用户"） |
| `action` | `@Operation.summary` 或方法名 | 操作描述 |
| `targetType` / `targetId` / `targetName` | 类名 + PathVariable | 操作对象 |
| `content` | `@Operation.description` 或 URL | 详细内容 |
| `requestMethod` / `requestUrl` | HTTP 请求 | 请求方法+路径 |
| `requestParams` | `request.getParameterMap()` | Query 参数（已脱敏） |
| `requestBody` | 请求体 | Request Body（需业务层脱敏） |
| `responseCode` | `response.getStatus()` | HTTP 响应码 |
| `ipAddress` | X-Forwarded-For 或 `getRemoteAddr()` | 客户端 IP |
| `userAgent` | `User-Agent` header | 客户端 UA |
| `durationMs` | `preHandle` 时戳到 `afterCompletion` | 请求耗时 |
| `errorMessage` | 异常 `ex.getMessage()` | 异常信息（失败时） |

**IP 解析优先级**（`OperationLogInterceptor.resolveClientIp()`）：

```
X-Forwarded-For 首节点  →  反向代理后的真实 IP
         ↓（为空时）
request.getRemoteAddr()  →  直连时的客户端 IP
```

```java
// OperationLogInterceptor.java:131-138
private String resolveClientIp(HttpServletRequest request) {
    String forwarded = request.getHeader("X-Forwarded-For");
    if (forwarded != null && !forwarded.isBlank()) {
        int comma = forwarded.indexOf(',');
        return comma > 0 ? forwarded.substring(0, comma).trim() : forwarded.trim();
    }
    return request.getRemoteAddr();
}
```

**记录范围**：仅记录 POST/PUT/PATCH/DELETE 四种写操作，GET 请求不记录。

**容错**：记录失败不影响主业务流程（`try-catch` + `log.warn`）。

### 11.4 敏感字段脱敏（日志层）

项目无通用字段级加密，但对日志输出中的敏感信息做脱敏处理。

**脱敏规则**（`RealDouyinOrderGateway.mask()`）：

```java
// RealDouyinOrderGateway.java:83-92
private String mask(String value) {
    if (value == null || value.isBlank()) return "";
    String normalized = value.trim();
    if (normalized.length() <= 8) return normalized;  // 短字符串不脱敏
    // 保留前4位 + **** + 末4位
    return normalized.substring(0, 4) + "****" + normalized.substring(normalized.length() - 4);
}
```

**适用场景**：日志中打印 `appKey`、`authId` 等抖店凭证时使用 `mask()`，避免日志泄露完整密钥。

```
# 脱敏前
gateway=RealDouyinOrderGateway, appKey=xxx1234567890, shopId=douyin123

# 脱敏后
gateway=RealDouyinOrderGateway, appKey=xxx1****7890, shopId=douyin123
```

### 11.5 统一响应结构（防信息泄露）

所有 API 响应使用 `ApiResult<T>` 统一包装，避免不同 Controller 返回不同结构导致前端解析异常。

```java
// ApiResult.java:36-43

    ApiResult<T> result = new ApiResult<>();
    result.code = code;
    result.msg = msg;
    result.data = data;

```

**响应结构**：

```json

  "code": 200,
  "msg": "操作成功",
  "data": { ... },
  "timestamp": 1746892800000
}
```

错误响应不携带堆栈信息（堆栈仅写入后端日志），防止前端从错误响应推断系统内部结构。

### 12.3 缺失的 HA 能力

| 能力 | 现状 | 风险 |
|---|---|---|
| 多实例部署 | 单 backend 实例 | 进程崩溃 → 服务中断，无冗余 |
| 进程级自愈（OOM） | `restart: on-failure` 只管退出码≠0 | JVM OOM 被 SIGKILL 后容器可能不重启 |
| 数据库主从复制 | 单 PostgreSQL，无流复制 | 磁盘故障 → 全量数据丢失 |
| Redis 持久化 | 默认 RDB 快照 | 进程崩溃 → Token 黑名单丢失（重启后用户需重新登录） |
| 多级健康检查 | 仅检查 `/actuator/health` | 进程假死、线程池耗尽时 healthcheck 仍可能通过 |
| 优雅关闭 | 无 preStop + 确认机制 | 容器 stop 时请求被强制中断 |
| 抖店 API 熔断 | 无 Sentinel/Hystrix | 三方 API 故障时线程池耗尽，引发级联崩溃 |
| 重试 + 幂等保障 | 无通用重试框架 | 网络抖动时数据重复写入 |
| 负载均衡 | 无 Nginx/HAProxy | 无法水平扩容 |
| 日志持久化 | 仅写容器内文件系统 | 容器重建后日志丢失 |
| 监控告警 | 无 | 服务异常无通知，依赖人工发现 |
| 定时任务多实例协调 | 仅 `SampleLifecycleJob` 有 Redis 锁，`OrderSyncJob` 无 | 多实例部署时任务重复执行 |

### 12.4 关键风险场景

**场景 1：JVM OOM 反复重启**

```
OOM kill（退出码 137）→ on-failure 重启 → -Xmx512m 仍不够 → 再次 OOM → 无限循环
服务处于"高频重启"状态，所有请求失败
```

**场景 2：抖店 API 超时，线程池耗尽**

```
抖店 API 响应 > 5s（read-timeout）
→ HikariCP 默认 maximum-pool-size=10 全占满
→ 所有接口（包括 healthcheck）全部阻塞
→ healthcheck 超时 → Docker 标记 unhealthy → 重启
→ 但抖店 API 未恢复 → 再次超时 → CrashLoopBackOff
```

**场景 3：健康检查假阳性**

```
healthcheck 只检查 GET /actuator/health（不检查 DB 写入）
磁盘满 → PostgreSQL 进入 read-only → 所有写入失败
→ healthcheck 仍返回 200 → Docker 认为服务正常
→ 用户看到所有操作报错，但系统"看起来健康"
```

**场景 4：订单同步 Job 被强制中断**

```
OrderSyncJob 执行 insert batch 1000 条，已插入 800 条时 docker-compose stop
→ 进程强制 kill → 事务回滚
→ 但 job offset 未记录 → 下次 job 重新执行同一时间窗口
→ 数据重复或唯一键冲突
```
