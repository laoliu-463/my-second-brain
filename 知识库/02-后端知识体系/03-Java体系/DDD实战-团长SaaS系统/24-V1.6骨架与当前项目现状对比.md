# V1.6骨架与当前项目现状对比

## 概述

V1.6 设计骨架与当前项目"业务方向高度一致"，但 V1.6 是目标态文档，不是现状描述。当前仓库已进入 V2.2 / P3 收口阶段，主链路能力比 V1.6 更靠后；架构实现上还没有达到 V1.6 描述的"7 个领域 + 内部事件驱动 + 分析汇总表"理想骨架。真正差异在于 Test/Real Gateway 契约 + 单活环境 + QA 可复验证据，这套实际架构护城河 V1.6 文档未覆盖。

## 详细对比

| 对比项 | V1.6 骨架 | 当前项目事实 |
|--------|-----------|-------------|
| 阶段 | "领域设计完成，可进入开发" | 已到 real-pre/P3 收口，P3-7 Dashboard 对账已 PASS；仍等真实 `pick_source` 订单样本 |
| 技术栈 | Go / Java / Python 待定 | 已固定：Java 17 + Spring Boot + PostgreSQL + Redis，前端 Vue3/Vite/Naive UI |
| 领域划分 | 7 个独立领域 | 业务实体基本覆盖，但代码按 `controller/service/entity/mapper/gateway` 技术层组织，不是领域 package |
| 事件驱动 | 领域间通过事件通信 | 未看到内部领域事件总线；订单同步后直接调用映射、商家、寄样生命周期服务 |
| Gateway | 文档未细化 | Test/Real Gateway 是强主线，且是项目更实际的架构核心 |
| 状态机 | 商品、寄样、订单简版状态 | 已部分落地：商品 `PENDING_AUDIT/APPROVED/...`，寄样 `SHIPPING/DELIVERED/PENDING_HOMEWORK/COMPLETED` |
| 独家规则 | 独家达人、独家商家独立维度，达人优先 | 服务层已有独家能力，但不是当前主验收闭环；代码中独家商家先于独家达人判断，规则需重新核对 |
| 数据看板 | 分析模块订阅事件，写汇总表 | 主要直接 SQL 聚合订单事实表并做对账，尚不是事件驱动汇总表方案 |
| 部署 | 单体部署、模块化组织 | 单活 `3000/8080` compose 切 `test/real-pre` |

## 最关键差异

1. **事件驱动骨架是目标态**：V1.6 的"7领域+事件通信"是演进方向，当前代码仍为技术分层。订单同步后直接调用下游服务，未引入领域事件总线或 Outbox 模式。

2. **真正的架构护城河未写入 V1.6**：`Test/Real Gateway 契约 + 单活环境切换 + 可复验 QA 证据链`才是当前项目差异点，这套实践在 V1.6 文档里没有对应章节。

3. **独家规则口径需单独定稿**：V1.6 写"独立维度、达人优先"，当前代码实现是"商家优先于达人"，两者判断顺序不一致，需找业务确认。

4. **数据看板是 SQL 聚合不是事件汇总**：DashboardService 直接 SQL 聚合订单事实表做对账，不是"分析模块订阅事件写汇总表"的方案。

## 建议落地

| 动作 | 优先级 | 说明 |
|------|--------|------|
| 不替换现有文档 | — | V1.6 是目标态，直接替换会制造混乱 |
| 将 V1.6 改名为"V2.3 架构目标草案" | 中 | 保留 7 领域划分作为未来代码边界参照 |
| 更新 `docs/02-架构设计.md` 过时环境描述 | 高 | 当前文档可能仍残留旧 Docker 三轨描述 |
| 补一节"当前实现 vs 目标骨架" | 高 | 在架构文档中显式标注现状和目标的差距 |
| 真正动代码时优先从订单同步副作用链引入最小领域事件/Outbox | 低 | 不要一上来大拆包，从增量引入 |
