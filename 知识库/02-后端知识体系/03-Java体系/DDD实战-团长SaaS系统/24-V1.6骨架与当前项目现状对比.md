# V1.6骨架与当前项目现状对比

## 概述

V1.6 设计骨架与当前项目"业务方向高度一致"，但 V1.6 是目标态文档，不是现状描述。当前仓库已进入 V2.2 / P3 收口阶段，主链路能力比 V1.6 更靠后；架构实现上还没有达到 V1.6 描述的"7 个领域 + 内部事件驱动 + 分析汇总表"理想骨架。真正差异在于 Test/Real Gateway 契约 + 单活环境 + QA 可复验证据链，这套实际架构护城河 V1.6 文档未覆盖。

---

## 一、核心设计原则对比

| V1.6 骨架原则 | 当前项目 docs/02 | 对齐结论 |
|--------------|-----------------|---------|
| 1. 领域划分 | 技术分层（controller/service/entity/mapper/gateway） | ⚠️ 路径不同 |
| 2. 事件驱动 | 直接 service 调用，无事件总线 | ⚠️ 路径不同 |
| 3. 状态机 | 商品、寄样、订单有状态机实现 | ✅ 部分对齐 |
| 4. 规则优先级 | 业绩规则在代码中，未独立为优先级表 | ⚠️ 规则在但未结构化 |
| 5. 分析独立 | SQL 聚合，非事件订阅汇总表 | ⚠️ 路径不同但效果等价 |

## 六、⚠️ 独家规则判断顺序——关键冲突（需业务定稿）

### V1.6 设计骨架 §五 业绩归属优先级

```
优先级 1：独家达人（覆盖订单的默认渠道）
优先级 2：独家商家（覆盖订单的默认招商）
优先级 3：默认归因（渠道/招商兜底）
```

### 当前 `AttributionService.resolveAttribution()` 实际代码顺序

```java
// 第 95 行：独家商家先判断
ExclusiveOwner merchantExclusiveOwner = findExclusiveMerchantOwner(merchantId);
if (merchantExclusiveOwner != null) {
    return AttributionResult.attributed(...);  // ← 直接返回，独家达人不会走到
}

// 第 110 行：独家达人后判断
ExclusiveOwner exclusiveOwner = findExclusiveTalentOwner(talentUid);
if (exclusiveOwner != null) {
    return AttributionResult.attributed(...);
}
```

### 结论

**V1.6 设计是"达人优先"，当前代码是"商家优先"——顺序完全相反。**

这是唯一需要改代码的差异。顺序不同意味着：当一个订单同时命中独家达人和独家商家时，业绩归属会不同。

**需要业务方确认**：当订单同时命中独家达人和独家商家时，最终业绩归给谁？
- V1.6 骨架答案：归给独家达人（达人优先）
- 当前代码答案：归给独家商家（商家优先）

## 十、结论与建议落地

### 需要业务定稿的一件事

> **独家达人和独家商家的判断顺序**——V1.6 骨架写"达人优先"，当前代码是"商家优先"。这个顺序会影响业绩归属结果。确认后同步 `AttributionService` 的判断顺序。

### 需要更新文档

- `docs/02-架构设计.md` §六：按 `docs/04` 当前事实重写环境描述
- `docs/03-Test与Real网关契约.md` §五：同上

### 建议演进方向（不改当前代码）

| 动作 | 优先级 | 说明 |
|------|--------|------|
| 将 V1.6 骨架改名为"V2.3 架构目标草案" | 中 | 保留 7 领域划分作为未来代码边界参照 |
| 在 `docs/02-架构设计.md` 补"当前实现 vs 目标骨架"章节 | 高 | 显式标注现状和目标差距 |
| 从订单同步副作用链引入最小 Outbox 模式 | 低 | 不要一上来大拆包，从增量引入 |

### 不需要做的事

- 不替换现有项目文档为 V1.6（V1.6 是目标态，不是现状）
- 不大拆包引入领域事件总线（当前阶段 ROI 不足）
- 不把 SQL 聚合改成事件汇总表（当前够用，已有对账 PASS 证据）
