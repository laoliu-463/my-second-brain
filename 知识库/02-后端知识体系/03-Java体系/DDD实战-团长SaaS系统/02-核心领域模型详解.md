# 02-核心领域模型详解

## 概述

本篇基于真实源码（ProductService.java、AttributionService.java、PickSourceMappingService.java、PickSourceMapping.java、ProductSnapshot.java、ColonelsettlementActivityMapper.java）详解 4 个核心限界上下文的聚合根、值对象、领域服务和关键策略（POLICY），重点标注团长 SaaS 的核心业务不变式。

> 源码中的 `NativeColonelBuyinResolution` 是 Java record 类型，下文完整展示其声明方式。`source` 字段枚举值直接标注来源层，可用于归因溯源日志。

**COALESCE UPSERT 问题（Bug 根因）**：

```java
// ColonelsettlementActivityMapper.xml 或 @Select 注解中的 UPSERT
@Select("""
    INSERT INTO colonel_activity
        (id, activity_id, colonel_buyin_id, extra_data, ...)
    VALUES
        (#{id}, #{activityId}, #{colonelBuyinId}, #{extraData}::jsonb, ...)
    ON CONFLICT (activity_id)
    DO UPDATE SET
        colonel_buyin_id = COALESCE(
            colonel_activity.colonel_buyin_id,   -- 旧值优先，永不覆盖
            EXCLUDED.colonel_buyin_id            -- 只在新值为非null时写入
        ),
        extra_data = COALESCE(
            colonel_activity.extra_data,
            EXCLUDED.extra_data
        )
    """)
void upsertRealActivityMeta(...);
```

时序场景：

| 场景 | 旧主字段 | 新值 | 写入结果 |
|---|---|---|---|
| 首次 UPSERT（API 有值） | null | 719302412345678901 | **719302412345678901** ✅ |
| 更新其他字段（buyinId传null） | 719302412345678901 | null | **719302412345678901** ✅（旧值保留）|
| 首次 UPSERT 时 API 值未传入 | **null** | null | **null 永久** ❌ |

`extra_data` JSONB 在所有场景下均正确写入（含真实 colonel_buyin_id），但旧代码只查主字段，导致第3层解析返回 null。

**sourceType 两种取值语义**：

| sourceType | 含义 | 写入时机 | 归因命中条件 |
|---|---|---|---|
| `PICK_SOURCE` | 外部链接归因 | 用户通过推广链接购买 | pickSource 完全匹配 |
| `NATIVE` | 抖店原生归因 | 用户通过视频/直播间下单（无链接） | (colonelBuyinId + activityId + productId) 精确匹配 |

### 领域服务：ColonelBuyinIdResolver（分层解析策略 POLICY）

```java
// 领域服务：ColonelBuyinIdResolver
// POLICY（策略模式）：分层解析 colonelBuyinId，从本地到远程
public class ColonelBuyinIdResolver {

    public ColonelBuyinId resolve(String activityId, String productId) {
        // 第1层：ProductSnapshot.extraData.colonelBuyinId
        String id = resolveFromSnapshot(activityId, productId);
        if (hasText(id)) return resolved(id, SNAPSHOT);

        // 第2层：OperationState.auditPayload.colonelBuyinId
        id = resolveFromOperationState(activityId, productId);
        if (hasText(id)) return resolved(id, OPERATION_STATE);

        // 第3层：Activity主字段 + Activity.extraData（bug修复点）
        id = resolveFromActivity(activityId);
        if (hasText(id)) return resolved(id, ACTIVITY_TABLE);

        // 第4层：调 ActivityApi.detail() 回填 Activity 表
        id = hydrateFromApi(activityId);
        if (hasText(id)) return resolved(id, API_BACKFILL);

        return unresolved();
    }

    // BUG FIX: resolveFromActivity 增加 extraData fallback
    private String resolveFromActivity(String activityId) {
        // 主字段（原有）
        Activity a = activityMapper.selectByActivityId(activityId);
        if (a != null && a.getColonelBuyinId() != null)
            return String.valueOf(a.getColonelBuyinId());

        // extraData 回源（patch新增）：hydrateColonelActivityMeta
        // 已将真实 colonel_buyin_id 存入 extraData JSONB
        Map extraData = activityMapper.selectExtraDataByActivityId(activityId);
        if (extraData != null && !extraData.isEmpty()) {
            String fromExtra = firstNonBlank(
                readString(extraData, "colonel_buyin_id", "colonelBuyinId"),
                readNestedString(extraData, "extra_data", "colonel_buyin_id")
            );
            if (hasText(fromExtra)) return fromExtra;
        }
        return null;
    }
}

// 值对象：ColonelBuyinId
public record ColonelBuyinId(
    String buyinId,
    IdSource source          // SNAPSHOT | OPERATION_STATE | ACTIVITY_TABLE | API_BACKFILL | UNRESOLVED
) {
    public boolean resolved() { return buyinId != null; }
}

public enum IdSource {
    PRODUCT_SNAPSHOT,
    PRODUCT_OPERATION_STATE,
    COLONEL_ACTIVITY,
    ACTIVITY_API_DETAIL,
    UNRESOLVED
}
```

### 领域服务：NativeMappingUniquenessChecker（不变式执行器）

```java
// 领域服务：NativeMappingUniquenessChecker
// 执行核心业务不变式：同一 NATIVE 键不能对应多个 userId
public class NativeMappingUniquenessChecker {

    public AmbiguityResult check(
            List<PickSourceMapping> existingMappings,
            UUID newUserId) {

        if (existingMappings.isEmpty())
            return AmbiguityResult.allow();

        Set<UUID> uniqueUserIds = existingMappings.stream()
            .map(PickSourceMapping::getUserId)
            .collect(toSet());

        if (uniqueUserIds.size() > 1)
            return AmbiguityResult.ambiguous(uniqueUserIds);

        if (uniqueUserIds.size() == 1 && !uniqueUserIds.contains(newUserId))
            return AmbiguityResult.conflict(uniqueUserIds.iterator().next());

        return AmbiguityResult.allow();
    }
}
```

### 领域服务：AttributionPolicy（归因策略 POLICY）

```java
// 领域服务：AttributionPolicy
// POLICY（策略模式）：订单归因优先级链
public class AttributionPolicy {

    public AttributionResult resolve(Order order) {
        // 第1优先：独家商家归因
        ExclusiveOwner merchant = findExclusiveMerchantOwner(order.getShopId());
        if (merchant != null)
            return attributed(merchant.userId(), REASON_ATTRIBUTED);

        // 第2优先：独家达人归因
        ExclusiveOwner talent = findExclusiveTalentOwner(order.getTalentUid());
        if (talent != null)
            return attributed(talent.userId(), REASON_ATTRIBUTED);

        // 第3优先：抖店原生归因（colonel_order_info 通道）
        if (hasColonelOrderInfo(order)) {
            NativeResolution result = resolveNativeColonel(order);
            if (result.mapping() != null)
                return attributed(result.mapping().getUserId(), REASON_COLONEL_ORDER_INFO);
            return unattributed(result.reason());
        }

        // 第4优先：渠道归因（pick_source 通道）
        if (hasPickSource(order)) {
            PickSourceMapping mapping = findPickSourceMapping(order);
            if (mapping != null)
                return attributed(mapping.getUserId(), REASON_ATTRIBUTED);
            return unattributed(REASON_MAPPING_NOT_FOUND);
        }

        // 兜底：无归因字段
        return unattributed(REASON_NO_PICK_SOURCE);
    }

    // 抖店原生归因精确查询
    private NativeResolution resolveNativeColonel(Order order) {
        String buyinId = order.getColonelOrderInfo().getColonelBuyinId();
        String activityId = order.getColonelOrderInfo().getActivityId();
        String productId = order.getProductId();

        // 精确查询 native mapping
        // WHERE colonel_buyin_id = buyinId
        // AND activity_id = activityId
        // AND product_id = productId
        // AND source_type = 'NATIVE'
        // AND status = 1
        List<PickSourceMapping> mappings = pickSourceMappingMapper.selectList(
            eq(PickSourceMapping::getColonelBuyinId, buyinId),
            eq(PickSourceMapping::getActivityId, activityId),
            eq(PickSourceMapping::getProductId, productId),
            eq(PickSourceMapping::getSourceType, SOURCE_TYPE_NATIVE),
            eq(PickSourceMapping::getStatus, 1)
        );

        if (mappings.size() == 1)
            return resolved(mappings.get(0));
        if (mappings.size() == 0)
            return unresolved(REASON_COLONEL_MAPPING_NOT_FOUND);
        return ambiguous(); // 多个达人操作同一商品
    }
}
```

## 核心业务不变式（Invariant）

### 不变式1：PickSourceMapping NATIVE 唯一性

```
给定的 (colonelBuyinId, activityId, productId, sourceType=NATIVE)
最多只能对应一个 userId

场景：
- 达人A 生成活动3859423商品X的推广链接 → 写入 native mapping(userId=A)
- 达人B 也生成活动3859423商品X的推广链接 → 发现已有 mapping(userId=A)
  → 标记为 AMBIGUOUS，两个人的订单都不归因

实现：PickSourceMappingService.findExistingMapping() 中的
NATIVE 分支按 (colonelBuyinId, activityId, productId, userId) 查找
UPSERT 时发现 existing.userId != newUserId → 拒绝写入
```

### 不变式2：Order Attribution 只读

```
订单归因一旦完成，extraData 不可再修改

场景：
- 订单 3859423 首次归因失败（COLONEL_MAPPING_NOT_FOUND）
- 后续用户转链生成了 native mapping
- 历史订单不会重新归因（除非手动触发 replay）

实现：Order 聚合的 attribution 字段是私有的
只能通过 AttributionService.resolveAttribution() 更新
归因服务本身幂等，重新执行不会破坏数据
```

### 不变式3：ColonelBuyinId 解析一致性

```
同一 activityId + productId 在同一时刻
ColonelBuyinIdResolver 应返回稳定结果

实现：
- 结果带 source 标记，日志可完整追溯来源
- 分层策略按固定顺序执行：Snapshot → OperationState → Activity → API
```
