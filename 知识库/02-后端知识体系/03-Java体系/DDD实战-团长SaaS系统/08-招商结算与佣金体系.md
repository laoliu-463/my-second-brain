# 08-招商结算与佣金体系

## 1. 概述

团长 SaaS 的核心商业模型是"招商分账"：团长（Colonelsettlement）撮合商家商品进入精选联盟，达人（Talent）通过推广链接或视频直播带货，产生订单后各方按比例分账。

本章记录订单结算数据模型和佣金计算逻辑。

## 2. 结算数据模型

### 2.1 订单结算表（colonelsettlement_order）

订单同步后，每条订单在 `colonelsettlement_order` 表中存储完整结算信息：

### 2.2 金额字段说明（单位：分）

所有金额字段（`orderAmount`、`settleColonelCommission` 等）单位为**分**，避免浮点精度问题。

| 字段 | 说明 | 抖店返回字段 |

| `actualAmount` | 实际支付金额 | `actual_amount` |

| `settleSecondColonelCommission` | 二级团长佣金 | `settle_second_colonel_commission` |

## 3. 佣金计算公式

### 3.1 术语定义

| 术语 | 说明 | 代码字段 |
|---|---|---|
| **服务费收入** | 团长从抖店拿到的总佣金 | `serviceFeeIncome` = Σ `settleColonelCommission` |
| **技术服务费** | 平台收取的技术服务费 | `techServiceFee` = Σ `settleColonelTechServiceFee` |
| **达人佣金** | 需要支付给达人的佣金 | `talentCommission` = Σ `settleSecondColonelCommission` |
| **服务费净额** | 扣除给达人的部分后，团长的实际收入 | `serviceFeeNet` = `serviceFeeIncome` - `techServiceFee` - `talentCommission` |
| **商家分账** | 商家按约定比例分得的佣金 | `bizCommission` |
| **渠道分账** | 渠道合作方按约定比例分得的佣金 | `channelCommission` |
| **毛利** | 团长实际留存收益 | `grossProfit` = `serviceFeeNet` - `bizCommission` - `channelCommission` |

### 3.2 计算公式（按活动分桶）

```
serviceFeeIncome   = Σ settleColonelCommission（所有订单）
techServiceFee     = Σ settleColonelTechServiceFee（所有订单）
talentCommission   = Σ settleSecondColonelCommission（所有订单）

serviceFeeNet      = serviceFeeIncome - techServiceFee - talentCommission
                                  （若为负数则取 0）

bizCommission      = Σ ActivityNet × bizRatio(activity)
channelCommission  = Σ ActivityNet × channelRatio(activity)
grossProfit        = serviceFeeNet - bizCommission - channelCommission
                                   （若为负数则取 0）

其中 ActivityNet(activity) = Σ(settleColonelCommission - settleColonelTechServiceFee - settleSecondColonelCommission)
                            （仅限该活动的订单，若为负数则该活动 ActivityNet 取 0）
```

**注意**：bizRatio 和 channelRatio 按**活动维度**配置，不同活动可使用不同的分账比例。

### 3.3 代码实现

```java
// CommissionService.java:46-77
public CommissionSummary calculateByActivityBuckets(List<ActivityCommissionBucket> buckets) {
    // 第一步：全局汇总
    long serviceFeeIncome = sumBuckets(buckets, ActivityCommissionBucket::serviceFeeIncome);
    long techServiceFee = sumBuckets(buckets, ActivityCommissionBucket::techServiceFee);
    long talentCommission = sumBuckets(buckets, ActivityCommissionBucket::talentCommission);

    long serviceFeeNet = Math.max(serviceFeeIncome - techServiceFee - talentCommission, 0L);

    // 第二步：加载分账比例（从 system_config 表查）
    BigDecimal bizRatio = loadRatio(KEY_BIZ_RATIO);      // 全局默认商家分账比例
    BigDecimal channelRatio = loadRatio(KEY_CHANNEL_RATIO); // 全局默认渠道分账比例

    long bizCommission = 0L;
    long channelCommission = 0L;
    for (ActivityCommissionBucket bucket : normalizeBuckets(buckets)) {
        long activityServiceFeeNet = Math.max(
                bucket.serviceFeeIncome() - bucket.techServiceFee() - bucket.talentCommission(), 0L);
        // 按活动维度取分账比例（有 override 用 override，无则用全局默认值）
        BigDecimal activityBizRatio = loadRatio(KEY_BIZ_ACTIVITY_RATIO_PREFIX, bucket.activityId(), bizRatio);
        BigDecimal activityChannelRatio = loadRatio(KEY_CHANNEL_ACTIVITY_RATIO_PREFIX, bucket.activityId(), channelRatio);
        bizCommission += multiplyCent(activityServiceFeeNet, activityBizRatio);
        channelCommission += multiplyCent(activityServiceFeeNet, activityChannelRatio);
    }

    long grossProfit = Math.max(serviceFeeNet - bizCommission - channelCommission, 0L);
    return new CommissionSummary(serviceFeeIncome, techServiceFee, talentCommission,
            serviceFeeNet, bizCommission, channelCommission, grossProfit, bizRatio, channelRatio);
}
```

### 3.4 分桶逻辑（避免 O(n²)）

```java
// CommissionService.java:79-90
// 按 activityId 分桶，每个桶内先 sum，再对桶结果 apply ratio
private Map<String, List<ColonelsettlementOrder>> groupByActivity(List<ColonelsettlementOrder> orders) {
    Map<String, List<ColonelsettlementOrder>> grouped = new LinkedHashMap<>();
    for (ColonelsettlementOrder order : orders) {
        String activityId = normalizeActivityId(order.getActivityId());
        grouped.computeIfAbsent(activityId, key -> new ArrayList<>()).add(order);
    }
    return grouped;
}
```

## 4. 分账比例配置

### 4.1 配置来源

分账比例从 `system_config` 表查询，使用 `BusinessRuleConfigService` 的缓存：

| 配置 Key | 默认值 | 说明 |
|---|---|---|
| `commission.business_default_ratio` | 0.15（15%） | 全局商家分账比例 |
| `commission.channel_default_ratio` | 0.15（15%） | 全局渠道分账比例 |
| `commission.business_activity_ratio.<activityId>` | — | 指定活动的商家分账比例 override |
| `commission.channel_activity_ratio.<activityId>` | — | 指定活动的渠道分账比例 override |

### 4.2 比例加载优先级

```java
// CommissionService.java:99-107
private BigDecimal loadRatio(String overridePrefix, String activityId, BigDecimal defaultRatio) {
    if (activityId != null && !activityId.isBlank()) {
        BigDecimal override = queryRatio(overridePrefix + activityId);
        if (override != null) {
            return override;  // 活动级别 override 优先
        }
    }
    return defaultRatio;  // 无 override 用全局默认值
}
```

**优先级**：活动级别 override > 全局默认

### 4.3 分账比例校验

```java
// CommissionService.java:114-129
private BigDecimal queryRatio(String key) {
    String value = jdbcTemplate.query(
            "SELECT config_value FROM system_config WHERE config_key = ? AND deleted = 0 LIMIT 1", ...);
    BigDecimal ratio = new BigDecimal(value.trim());
    if (ratio.compareTo(BigDecimal.ZERO) < 0) {
        return null;  // 负数比例视为无效，回退到默认值
    }
    return ratio;
}
```

## 6. 金额精度处理

所有金额以**分**为单位存储（Long），计算佣金时转换为元（BigDecimal），结果再转回分：

```java
// CommissionService.java:159-167
private long multiplyCent(long amount, BigDecimal ratio) {
    if (amount <= 0L) return 0L;
    return BigDecimal.valueOf(amount)
            .multiply(ratio)
            .setScale(0, RoundingMode.HALF_UP)  // 四舍五入保留整数分
            .longValue();
}
```

**精度问题**：15% 的分无法整除时（如 100 分 × 15% = 15.0 分没问题，但 101 分 × 15% = 15.15 分），使用 `HALF_UP` 四舍五入，每笔误差最多 ±0.5 分，批量求和后误差可控。
