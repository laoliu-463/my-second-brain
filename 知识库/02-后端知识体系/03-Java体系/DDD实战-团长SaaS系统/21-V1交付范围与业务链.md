# V1交付范围与业务链

## 概述

V1 的核心不是一次性实现完整抖音团长 SaaS，而是优先跑通渠道、招商、管理三条基础业务链：渠道从认领达人到转链、寄样、订单、业绩；招商从同步活动到上架、审核寄样、归因业绩；管理侧提供人员、权限和规则配置。

## 详细内容

### 来源优先级

本页按新设计文档的口径整理，优先级如下：

| 优先级 | 来源 | 用途 |
|--------|------|------|
| 1 | `V1交付范围表.md` | V1 上线验收的范围合同 |
| 2 | `技术落地设计.md` | 工程实现、架构、部署和非功能约束 |
| 3 | `领域设计/*` + `设计骨架.md` | 领域边界、事件、状态机和业务规则 |
| 4 | `saas系统V2.2(定稿).md` | 旧版完整需求，仅作背景参考 |

这批来源与既有 [[DDD实战-团长SaaS系统/index|DDD实战-团长SaaS系统]] 系列存在技术栈差异：现有系列以 Spring Boot 实战实现为主，新来源以 FastAPI V1 设计为主。因此本页只沉淀产品范围与业务链，不直接覆盖既有 Java 实现结论。

### 三条验收主线

| 链路 | V1 路径 | 验收重点 |
|------|---------|----------|
| 渠道链 | 认领达人 -> 选品 -> 转链 -> 寄样 -> 订单 -> 业绩 | `pick_source` 能追溯渠道；寄样能随订单出单完成；渠道能看到自己的业绩 |
| 招商链 | 同步活动 -> 上架商品 -> 审寄样 -> 订单 -> 业绩 | 活动与商品同步后可上架；商品能归属默认招商；订单能形成招商业绩 |
| 管理链 | 配人 -> 配角色 -> 配规则 -> 各域读取配置 | 用户、角色、数据范围和配置项能被业务域读取，并影响寄样、提成等规则 |

### V1 模块范围

`V1交付范围表.md` 将范围拆成 89 个条目，其中 76 个建议进入 V1，13 个延后到 V2 或做简化。

| 模块 | 条目数 | V1 重点 |
|------|--------|---------|
| 商品域 | 14 | 同步活动与商品、招商补充信息、商品库筛选排序、转链与 `pick_source` 映射 |
| 达人域 | 10 | 达人查询与补全、公海/私海、多人认领、保护期、收货地址和标签 |
| 寄样域 | 11 | 寄样申请、7 天重复限制、招商审核、运营发货、订单匹配交作业 |
| 订单域 | 10 | 定时同步团长订单、双轨金额、默认渠道与默认招商归因、订单状态更新 |
| 业绩域 | 10 | 默认归因、双轨提成、退款冲正、业绩列表、订单列表补全业绩 |
| 用户域 | 9 | 登录、预置角色、自定义角色、数据范围、员工与组织管理 |
| 配置域 | 9 | 寄样限制、保护期、全局提成比例、转链模板、配置变更事件 |
| 分析模块 | 8 | 指标卡片、时间口径、汇总表、订单明细下钻 |
| 跨域/集成 | 8 | 单体模块化、PostgreSQL、Redis、Celery、抖音 SDK Token、事件幂等 |

### V1 明确不做或简化

| 能力 | V1 处理 | 影响 |
|------|---------|------|
| 独家达人 / 独家商家 | 不做，整体延后 V2 | 业绩域 V1 使用默认归因：`final_channel = default_channel`，`final_recruiter = default_recruiter` |
| 个别品负责人覆盖 | 不做 | 商品默认招商按活动绑定招商处理 |
| 毛利指标 | 不做 | V1 聚焦服务费收益和提成 |
| 商品负责人变更后重算 | 不做 | 未结算单重算留到后续 |
| 寄样 30 天自动关闭 | 不做自动关闭 | 由运营拒绝或人工处理关单 |
| 高级物流自动化 | 原表存在口径冲突，需二次确认 | `S-09`、`S-10` 标成 V1 确认“是”，但备注又写 V2，需要实现前统一口径 |

### 需要二次确认的口径漂移

| 位置 | 漂移点 | 建议处理 |
|------|--------|----------|
| 寄样域 `S-09` / `S-10` | V1 建议为“否”，V1 确认为“是”，备注写 V2 | 开发排期前确认是否真的要做物流 API 和 Excel 批量导入 |
| 用户域 `U-09` | V1 建议为“否”，V1 确认为“是”，备注同时写 V2 / V1 一人一角色 | 与技术落地设计的“一人多角色：权限并集，数据范围取最宽”对齐 |
| 独家能力 | 领域设计保留完整独家方案，但 V1 范围表明确不做 | 代码可留扩展点，测试和验收不应要求独家计算 |
| 技术栈 | 旧需求写 FastAPI；现有 wiki 系列多为 Spring Boot 实战 | 将“产品/领域设计”和“现有实现”分层维护，避免混成一个技术结论 |

### P0 验收清单

| 验收项 | 最小判断 |
|--------|----------|
| 渠道链端到端 | 渠道认领达人、选商品、复制推广链接、申请寄样，订单同步后能形成渠道业绩 |
| 招商链端到端 | 招商同步活动商品、补充信息上架、审核寄样，订单能追溯默认招商 |
| 管理链端到端 | 管理员配置用户、角色、数据范围和规则后，各域读到配置并生效 |
| 订单同步 | 5-10 分钟增量入库，关键字段变更能触发下游事件 |
| 业绩计算 | V1 使用默认归因和双轨金额计算提成，退款/失效不计入有效业绩 |
| 性能约束 | 列表必须分页并带时间范围，禁止无时间条件扫全量订单 |
