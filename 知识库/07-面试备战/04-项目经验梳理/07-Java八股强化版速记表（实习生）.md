# Java八股强化版速记表（实习生）

## 概述

本页聚焦 Java 面试高频基础题，覆盖集合、JUC、线程池、SQL、Redis，以及 JVM、事务传播、RESTful、Stream/Lambda 等进阶理论，并尽量配上 AI 项目和教务系统的落地例子。适合在 [[06-面试终极速记表（实习生版）]] 基础上做专题强化。

## 详细内容

## 1. 集合与数据结构

| 题目 | 顺口回答 | 项目结合示例 |
|---|---|---|
| ArrayList vs LinkedList | ArrayList 查询快、遍历快，但中间插入删除慢；LinkedList 插入删除快，但随机查询慢。 | AI 项目里 chunk 列表、检索结果列表更适合 ArrayList；教务系统分页结果也更适合 ArrayList。 |
| HashMap vs ConcurrentHashMap | HashMap 多线程不安全，极端情况下会有数据丢失甚至死循环；ConcurrentHashMap 线程安全，底层靠 CAS 和局部加锁。 | AI 项目里异步任务状态统计、教务系统并发场景下的共享计数都更适合线程安全容器。 |

---

## 2. 并发 / JUC

| 题目 | 顺口回答 | 项目结合示例 |
|---|---|---|
| synchronized vs ReentrantLock | synchronized 简单、自动释放锁；ReentrantLock 功能更强，支持公平锁、可中断和 Condition。 | AI 异步任务里简单同步场景用 synchronized 更直接；需要更灵活控制时再考虑 ReentrantLock。 |
| CAS | Compare And Swap，无锁原子操作。很多并发类底层都靠它保证更新时的线程安全。 | ConcurrentHashMap 插入节点、Atomic 计数器更新任务状态都能用到 CAS 思想。 |
| volatile | 保证可见性和禁止指令重排，但不保证复合操作原子性。 | 可用于任务开关、状态标记这类轻量同步场景。 |
| 线程池 | 核心作用是复用线程、控制并发、削峰，避免系统频繁创建销毁线程。 | AI 项目里异步解析简历和生成报告，教务系统里异步统计和通知任务都适合线程池。 |
| 线程池参数 | 最核心的是 corePoolSize、maxPoolSize、queue、threadFactory、handler。 | 通过合理线程数和队列容量配置，保证异步任务不把系统拖垮。 |
| 线程池执行流程 | 核心线程 -> 队列 -> 非核心线程 -> 拒绝策略。 | 面试时可以直接拿这个流程解释异步任务为什么不会无限起线程。 |
| Executors 限制 | 像 FixedThreadPool、CachedThreadPool 这类工厂方法不够安全，极端情况下可能 OOM。 | 实际项目里更推荐自己 new ThreadPoolExecutor，参数更可控。 |

## 3. SQL / 数据库

| 题目 | 顺口回答 | 项目结合示例 |
|---|---|---|
| 索引 | 高频字段加索引能显著提升查询效率；常见失效场景有函数、`like '%xxx'`、类型不一致。 | 教务系统里学生 ID、课程 ID、日期这类字段都适合建索引。 |
| 事务 ACID | ACID 就是原子性、一致性、隔离性、持久性。 | 教务系统单服务写库用 Spring 事务，跨服务流程用 Seata AT 模式。 |
| MVCC | 多版本并发控制，本质上让读写尽量不互相阻塞。 | 教务系统多个用户同时查课程、查消课记录时，并发读性能会更好。 |

## 4. Redis / 缓存

| 题目 | 顺口回答 | 项目结合示例 |
|---|---|---|
| Redis 数据类型 | 常见有 String、Hash、List、Set、ZSet。 | AI 异步任务可以用 Stream；教务系统缓存学生和课程信息更适合 Hash。 |
| 缓存穿透 | 查询不存在数据，缓存和数据库都没有，请求一直打到数据库。常用空值缓存或布隆过滤器。 | 可用于防止热点简历或学生 ID 被恶意打穿。 |
| 缓存击穿 | 热点 key 过期瞬间，大量请求同时查库。可用互斥锁或逻辑过期。 | 教务系统高并发查询课程、学生详情时常考这个点。 |
| 缓存雪崩 | 大量 key 同时过期，导致大量请求回源数据库。常用 TTL 随机化和高可用架构。 | 教务系统接口高可用保障可以从这里展开。 |

## 5. 高级理论 / 进阶

| 题目 | 顺口回答 | 项目结合示例 |
|---|---|---|
| JVM / 内存模型 | JVM 常见关注点是堆、栈、方法区，以及 GC 对性能的影响。 | 线程池和异步任务多时，要注意内存占用和频繁 GC。 |
| 异常处理 | `try/catch/finally` 是基础，项目里更重要的是统一异常和日志记录。 | AI 项目文件解析异常要能记录并兜底，避免任务丢失。 |
| Spring 事务传播 | 最常用的是 REQUIRED 和 REQUIRES_NEW。 | 教务系统本地事务和异步任务事务边界可以拿这个回答。 |
| RESTful 接口设计 | 常见就是 GET/POST/PUT/DELETE，对应资源操作，注意幂等性、分页和状态码。 | 教务系统接口设计可以从这里展开。 |
| Stream / Lambda | 适合做集合过滤、映射、聚合，代码更简洁。 | AI 项目做文本切分、结果映射时可以举例。 |
| 线程池高级配置 | 常见阻塞队列有 ArrayBlockingQueue、LinkedBlockingQueue；拒绝策略有 CallerRunsPolicy 等。 | 面试官往深处问线程池时，这一块很加分。 |
