## 第二部分：项目一 — AI 面试助手

### 问题 1：项目整体介绍

**面试官**："你简历里写的 AI 面试助手是做什么的？为什么做这个项目？"

**你**：

我的第一个项目叫 AI 面试助手，是一个基于 RAG 的简历分析和面试辅助系统。

用户上传 PDF 或 DOCX 格式的简历，后端自动解析文本内容，清洗脏数据和格式噪音，然后按段落或固定长度切成 chunk，通过 Embedding 模型把每个 chunk 转成向量，存入 PostgreSQL 加 pgvector 扩展的向量库。用户提问时，系统先把问题转成向量，在向量库里检索最相似的 N 个 chunk，拼接成 Prompt 发给大模型生成回答。

做这个项目有两个目的：一是让 AI 回答精准，基于真实简历内容而不是胡说八道；二是练习向量数据库、异步任务和大模型集成的工程能力。

这个项目参考了 GitHub 上 Snailclimb 的 interview-guide 项目的 RAG 架构思路，但文本切分策略、向量检索参数、结构化输出格式和 Redis Stream 异步任务都是我独立设计实现的，针对的是简历分析这个具体场景。

**技术亮点**：RAG 检索增强生成 / pgvector 向量数据库 / Redis Stream 异步任务 / 结构化输出控制

**继续追问怎么接**

  - 你可以接：因为它能把 RAG、异步任务和工程化链路串起来，不只是单纯调一个模型。
- 面试官继续问：这个项目最体现你什么能力？
  - 你可以接：我觉得最能体现的是把一个 AI 想法拆成完整后端链路的能力，从检索、存储到异步和输出控制都要自己兜住。

### 问题 2：RAG 原理

**面试官**："RAG 是什么？在项目里怎么用？"

**你**：

RAG 的全称是 Retrieval-Augmented Generation，翻译过来是检索增强生成。核心思想是在大模型生成回答之前，先从一个知识库里检索出和问题最相关的内容，把检索结果拼进 Prompt 里，再让大模型基于这些真实内容生成回答。

这样做有三个好处。第一是回答有依据，不是凭空编造。第二是减少大模型处理长文本的上下文负担，降低 token 消耗。第三是回答可以溯源，用户能看到 AI 引用的是哪段原文。

项目里 RAG 的流程是：用户上传简历 → 文本切分成 chunk → Embedding 模型向量化 → 存入 pgvector。用户提问时：问题向量化 → 在向量库里做余弦相似度检索，取 topK 最相似的 chunk → 拼接成 Prompt → 调用大模型 API 生成回答。

**技术亮点**：余弦相似度检索 / HNSW 近似最近邻算法 / topK 控制上下文长度

**继续追问怎么接**

- 面试官继续问：为什么 RAG 比直接把简历喂给模型更好？
  - 你可以接：因为先检索再生成更可控，能减少噪音上下文，也能降低 token 消耗。
- 面试官继续问：topK 为什么不能太大？
  - 你可以接：太大会把不相关内容也带进去，模型注意力被稀释，反而更容易答偏。

### 问题 3：pgvector 选型

**面试官**："为什么用 pgvector？不用 Milvus 或 Chroma 吗？"

**你**：

pgvector 是 PostgreSQL 的向量扩展，在已有 PostgreSQL 的基础上启用扩展就能用，不需要额外部署任何组件。

Milvus 是独立的向量数据库，功能强大但需要单独部署和维护，增加了运维成本。Chroma 是 Python 库，原生不兼容 Java 环境。我的项目数据量中等，百万级向量以内 pgvector 性能完全够用，延迟在毫秒级别。而且 pgvector 和业务数据存在同一个数据库里，数据管理和备份都更简单。

**技术亮点**：轻量落地 / 无额外组件 / 与业务数据同库

---

### 问题 4：文本切分策略

**面试官**："简历文本为什么要切分？不直接整篇向量化吗？"

**你**：

直接对整篇文档向量化效果很差。简历少则几百字多则几千字，如果转成一个向量，粒度太粗，语义信息被稀释。检索的时候匹配到的结果要么太宽泛、要么太窄，用户问"这个人的项目管理经验怎么样"，系统可能返回的是教育背景而不是项目经历。

切分成 chunk 后，每个向量只代表一段独立语义，检索准确性大幅提升。

我的切分策略是每 500 个字符切一个 chunk，相邻 chunk 之间有 50 个字符的重叠区域。重叠的目的是防止一段完整语义被拦腰切断，导致上下文丢失。

**技术亮点**：重叠 chunk 切分 / 语义完整性保证 / topK 检索精度提升

### 问题 5：向量检索原理

**面试官**："向量检索是怎么工作的？为什么能快速找到相似的向量？"

**你**：

向量检索的核心是计算两个向量的相似度。常用的是余弦相似度——算两个向量夹角的余弦值，结果越接近 1 表示越相似。

pgvector 底层用的是 HNSW 算法，全称是 Hierarchical Navigable Small World。简单说就是给所有向量建一个多层的图结构，搜索时从最高层开始快速定位大致方向，逐层向下精确搜索。因为每层都是近似最近邻，不需要遍历全表，时间复杂度从 O(n) 降到 O(log n)，百万级向量检索能做到毫秒级。

**技术亮点**：HNSW 分层图索引 / 余弦相似度 / ANN近似最近邻

### 问题 6：Redis Stream 异步任务

**面试官**："Redis Stream 在项目里干什么？为什么不用 RabbitMQ 或 Kafka？"

**你**：

上传简历后，文本解析、脏数据清洗、chunk 切分、向量化存储和报告生成，每一步都比较耗时。如果同步执行，用户要等几十秒才能看到结果，体验很差。

我的做法是把整个流程拆成任务扔进 Redis Stream，消费者异步完成整个链路，接口立即返回任务 ID，用户通过轮询查询任务进度。

选 Redis Stream 而不是 RabbitMQ 或 Kafka，是因为项目本身已经用了 Redis，直接复用不需要引入额外组件。而且 Redis Stream 支持消息持久化和消费者组，适合我这种中小规模的任务队列场景。

**技术亮点**：Redis Stream / 消费者组 / 消息持久化 / 异步任务削峰

**继续追问怎么接**

- 面试官继续问：为什么不用 RabbitMQ？
  - 你可以接：因为项目里已经有 Redis，中小规模任务场景下 Stream 足够，而且接入和维护成本更低。
- 面试官继续问：异步之后用户怎么知道任务做完了？
  - 你可以接：上传后先返回任务 ID，前端轮询任务状态，后端返回处理中、成功或者失败。

### 问题 7：消费者挂了怎么办

**面试官**："Redis Stream 消费者挂了，任务丢了怎么办？"

**你**：

Redis Stream 有消费者组的概念，消息被消费后会进入 PENDING 列表，不会自动删除。消费者挂了之后重新启动，可以从 PENDING 列表里读取未确认的消息继续处理，不会丢消息。

另外我还加了一个兜底机制：定时任务每隔 5 分钟扫描 PENDING 列表，如果某条消息处理时间超过 30 分钟还没被 ACK，就重新投递到队列里，防止极端情况下消息彻底丢失。

**技术亮点**：消费者组 / PENDING 列表 / ACK 确认机制 / 定时兜底重投递

### 问题 8：AI 回答不准怎么办

**面试官**："如果 AI 回答跑偏了，你怎么控制它不乱说？"

**你**：

我做了三层控制。

第一层是向量检索层。topK 检索只返回最相关的 chunk，如果检索出来的内容本身就跟问题不相关，再好的模型也答不准。所以 chunk 切分质量和 topK 参数是关键。

第二层是 Prompt 层。我在 Prompt 里明确加了两条指令：一是"只能根据提供的简历内容回答，如果找不到相关信息就如实说你不知道，不要编造"；二是"按结构化格式输出，包括优势分析、风险点、改进建议三个部分"。

第三层是输出校验层。模型返回后校验 JSON 格式是否完整，三个字段是否都有内容，如果有异常就重新生成，最多重试 3 次。

**技术亮点**：检索层过滤 / Prompt 指令约束 / 输出结构化校验

**继续追问怎么接**

- 面试官继续问：如果模型还是乱说怎么办？
  - 你可以接：我会先检查检索结果是不是已经偏了，再看 Prompt 是否把边界写清楚，必要时缩小生成空间。
- 面试官继续问：为什么要结构化输出？
  - 你可以接：因为结构化之后更稳定，也更方便校验、展示和重试。

### 问题 9：项目难点

**面试官**："项目里最难的问题是什么？你怎么解决的？"

**你**：

有两个难点。

第一个是向量检索精度不够。早期版本检索出来的结果经常答非所问。用户问项目经验，返回的是教育背景。我通过调整 chunk 切分策略——从整段切改成重叠切——并且把 topK 从 3 增加到 5，让 Prompt 有更丰富的上下文，检索精度明显提升。

第二个是接口响应慢。向量化和大模型调用都耗时，同步执行要等 20 多秒。用 Redis Stream 异步化之后，接口响应时间降到 200 毫秒以内，用户体验大幅改善。

排查性能问题的工具是 Arthas 的 trace 命令，可以定位到具体哪个方法耗时多少，方便针对性优化。

**技术亮点**：检索策略调优 / Arthas trace 定位瓶颈 / 异步任务优化

### 问题 12：向量检索精度不准怎么优化

**面试官**："在你的 RAG 项目里，向量检索有时候不准，你是怎么优化的？"

**你**：

刚开始确实遇到过检索不够准的问题，主要表现是 chunk 太大或者 topK 设置不合理时，检索结果会混进一些不相关内容。

我的优化方法主要有三点。第一，按段落或者固定长度切分 chunk，让每个向量只聚焦一段相对独立的语义。第二，控制 topK，只保留最相关的片段，避免把太多噪音上下文拼进 Prompt。第三，在 Prompt 拼接时只使用这些检索出来的相关片段，不额外塞无关内容。

通过这几个调整，AI 回答的相关性和稳定性都明显提升了。

**技术亮点**：chunk 切分策略 / topK 控制 / 检索精度优化 / 减少无关上下文

### 问题 13：上传简历后处理慢怎么解决

**面试官**："上传简历后处理速度慢，你怎么解决？"

**你**：

因为文件解析、文本切分、向量化和报告生成都是耗时操作，如果同步执行，接口响应会很慢，用户体验很差。

所以我用了 Redis Stream 做异步任务处理。用户上传后接口先快速返回任务 ID，后面的解析、切分、向量化和报告生成由消费者异步执行。

Redis Stream 还有一个好处是支持消费者组和 pending 消息机制，任务失败后可以做重试，这样既提升了性能，也兼顾了可靠性。

**技术亮点**：Redis Stream 异步任务 / 快速响应 / 消费者组 / 失败重试

### 问题 14：AI 回答不准确怎么控制输出

实际来说，小规模系统用单体确实也可以，但拆微服务能练习服务治理和分布式一致性这些核心能力。

**技术亮点**：模块边界清晰 / 服务治理 / 分布式架构实践

### 问题 3：微服务组件职责

**面试官**："Nacos、Gateway、OpenFeign、Sentinel、Seata 各自负责什么？"

**你**：

**Nacos**：服务的注册中心和配置中心。每个服务启动时向 Nacos 注册，消费者通过 Nacos 发现服务地址，不需要硬编码 IP。配置中心让所有服务的配置文件集中在 Nacos 管理，修改配置不用重启服务。

**Gateway**：统一网关，所有请求的入口。负责路由转发、登录 Token 校验、接口限流。外部请求先到网关，符合路由规则的才转发到具体服务。

**OpenFeign**：声明式 HTTP 客户端。定义一个接口加注解，就能发 HTTP 请求调用其他服务，不需要手动创建 HttpClient，自动处理序列化和反序列化。

**Sentinel**：流量控制和熔断降级。当某个接口 QPS 过高或者响应超时，Sentinel 自动降级返回兜底数据，保护系统不被冲垮。支持按 QPS、线程数、调用关系等多维度限流，比 Hystrix 更丰富。

**Seata**：分布式事务控制器。跨服务操作时保证数据一致性。比如学员报名需要同时创建订单和扣减课时，Seata AT 模式通过记录逆向 SQL 实现自动回滚。

**技术亮点**：注册中心 / 配置中心 / 声明式远程调用 / 限流熔断 / 分布式事务

**继续追问怎么接**

- 面试官继续问：你觉得这里面最难的是哪个？
  - 你可以接：我觉得最难的是 Seata，因为它不只是把组件跑通，还涉及你怎么理解跨服务一致性。
- 面试官继续问：为什么 Gateway 要放在入口？
  - 你可以接：因为鉴权、路由、限流这些公共逻辑放在最前面最集中，也最方便治理。

### 问题 4：Seata 分布式事务原理

**面试官**："Seata 分布式事务是怎么工作的？"

**你**：

Seata AT 模式的工作流程分三步。

第一步是执行业务 SQL 之前，Seata 会拍照当前数据状态。第二步是执行业务 SQL。第三步是在事务提交前，对比拍照的数据和当前数据，如果发现变化就生成逆向 SQL。

如果整个事务链路上所有分支都执行成功，Seata 异步删除逆向 SQL 记录。如果任何一个分支失败，Seata 自动执行逆向 SQL 回滚数据。

对业务代码来说，只需要在跨服务的方法上加一个 @GlobalTransactional 注解，Seata 自动处理分支协调和回滚，对业务透明。

**技术亮点**：AT 模式 / 快照 + 逆向 SQL / 全局锁 / @GlobalTransactional

**继续追问怎么接**

- 面试官继续问：为什么用 AT，不用 TCC？
  - 你可以接：AT 对业务侵入更小，更适合这个项目复杂度；TCC 更灵活，但实现和维护成本更高。
- 面试官继续问：AT 模式最大的代价是什么？
  - 你可以接：它依赖 undo log 和全局锁，事务链路一长或者竞争大时，超时和性能问题会更明显。

### 问题 5：Seata 难点

**面试官**："Seata 配置和使用中遇到过什么坑？"

**你**：

最常遇到的问题是全局锁超时。学员报名的业务流程涉及订单表和课时表两个资源，Seata 用全局锁保证隔离性。但如果网络抖动或者 Seata Server 处理慢，全局锁会超时，事务失败回滚。

排查过程：首先确认 seata.grouplist 配置是否正确指向 Seata Server 地址；其次调整全局事务超时时间，从默认 30 秒调整到 60 秒；最后优化 Seata Server 本身的性能，确保 TM 和 TC 之间的通信稳定。

这个问题花了不少时间，最后通过调整参数和优化网络配置解决了。

**技术亮点**：全局锁超时排查 / 参数调优 / 网络配置优化

### 问题 6：接口响应慢怎么排查

**面试官**："接口响应慢，你怎么排查和解决？"

**你**：

排查用 Arthas 的 trace 命令，一层一层定位到耗时最长的方法。

常见原因有三个。

第一是数据库查询慢——多表关联或者高频查询字段没建索引，导致全表扫描。解决是对高频查询字段建索引，比如 WHERE user_id、WHERE course_id 这些。

第二是远程 Feign 调用串行执行。学员详情页要查课程信息、课时余额、考勤记录，如果三个请求顺序执行，耗时会叠加。解决是用 CompletableFuture 并行调用，三个请求同时发，谁先返回等谁，整体耗时从三个之和降到最慢那个。

第三是热点数据每次都查数据库。学员头像、课程信息这些变化不频繁的数据，缓存到 Redis 里，多次访问只查一次库。

实测一套优化下来，接口平均响应时间从 800 毫秒降到了 300 毫秒左右。

**技术亮点**：Arthas trace 定位 / 索引优化 / CompletableFuture 并行调用 / Redis 热点缓存

**继续追问怎么接**

- 面试官继续问：为什么先用 Arthas？
  - 你可以接：因为它能直接在线看到哪个方法慢，比一上来猜数据库还是远程调用更有效。
- 面试官继续问：并行调用有没有风险？
  - 你可以接：有，前提是这些调用彼此独立，而且线程池不能乱开，不然只是把串行瓶颈换成并发压力。

### 问题 7：Redis 在项目里的用法

**面试官**："Redis 在教务系统里怎么用？"

**你**：

两个主要用法。

第一是缓存学生信息和课程信息。学生详情页、课程列表页这些高频访问但变化不频繁的数据，存 Redis 里，设置较短 TTL 比如 5 分钟。第二次访问直接走缓存，不查库，数据库压力明显下降。

第二是作为异步任务队列。生成课表、发送通知这些耗时操作，扔进 Redis Stream，接口立即返回。

**技术亮点**：String/Hash 缓存 / TTL 过期策略 / Redis Stream 异步队列

### 问题 8：微服务项目怎么搭建

**面试官**："你教务管理系统是微服务，能讲一下你是怎么搭建的吗？"

**你**：

我们的教务管理系统采用 Spring Cloud Alibaba 微服务架构，把学生管理、课程管理和消课管理拆成独立服务，每个服务都可以单独开发、部署和扩展。

同时我用 Gateway 做统一网关，处理请求路由、鉴权和限流；服务间通信用 OpenFeign，开发体验上会更像本地方法调用。

稳定性方面，我用 Sentinel 做流量控制和熔断；事务方面，单服务用 Spring 事务，跨服务流程用 Seata 分布式事务，保证多表、多模块操作的一致性。

**技术亮点**：微服务拆分原则 / Nacos + Gateway + Feign + Sentinel + Seata / 服务治理 + 数据一致性

### 问题 9：服务之间怎么调用，失败了怎么办

**面试官**："服务之间如何调用？如果调用失败或者出现异常，你怎么处理？"

**你**：

服务调用主要通过 OpenFeign 实现，声明式调用像本地方法一样，代码可读性和维护性都比较好。

为了保证调用链稳定，我会用 Sentinel 做限流和熔断，避免某个服务异常时把整个系统拖垮。

对于一些异步任务或者耗时操作，比如消课统计、报表生成、通知类处理，我会用 Redis Stream 做异步处理，让主接口尽快返回。

如果服务调用失败，一般会配合重试、降级或者兜底返回来处理，优先保证系统可用性和核心业务稳定。

**技术亮点**：OpenFeign 声明式调用 / Sentinel 限流熔断 / Redis Stream 异步化 / 重试与降级

### 问题 11：Sentinel 限流规则怎么配

**面试官**："你项目里如何使用 Sentinel 做限流？具体规则怎么配置？"

**你**：

我主要会给关键接口做限流保护，比如查询课程、消课这类高频接口。

在 Sentinel 控制台里，我优先配的是 QPS 限流规则。比如某个核心查询接口设置最大 100 QPS，超过阈值之后就触发限流或者降级，避免流量一下子把下游服务和数据库压垮。

除了普通 QPS 限流，我还会配置热点参数限流。比如某些课程 ID 是热点数据，瞬时请求特别集中，就可以针对这个参数单独做保护，避免热点请求把缓存和数据库打穿。

这样做的效果是，接口在高并发下依然能稳定返回，哪怕不能完全成功，也能优雅降级，而不是全链路雪崩。

**技术亮点**：QPS 限流 / 热点参数限流 / 熔断降级 / 高并发稳定性

**继续追问怎么接**

- 面试官继续问：100 QPS 这个值怎么定？
  - 你可以接：一般结合接口耗时、机器资源和压测结果来定，先保守设置，再根据实际流量往上调。
- 面试官继续问：为什么热点参数限流很关键？
  - 你可以接：因为很多时候不是整个接口热，而是某个课程 ID 特别热，按参数限流更精准。

### 问题 12：接口稳定性怎么保障

**面试官**："微服务调用多了，接口有时候不稳定，你如何保证稳定性？"

**你**：

我主要从三个层面来保证接口稳定性。

第一是熔断降级。通过 Sentinel 给服务间调用设置熔断策略，如果下游服务异常或者响应超时，就自动走降级逻辑，避免一个服务出问题把整条调用链都拖挂。

第二是限流保护。对高频接口设 QPS 限流，防止瞬时高并发直接击穿数据库或者把应用线程打满。

第三是异步化处理。像消课统计、报表生成这类耗时操作，我会放到 Redis Stream 里异步处理，让主请求线程尽快释放，保证用户接口先有响应。

这样就算局部服务出现问题，系统整体还是可用的，至少能返回兜底结果或者提示用户稍后再试。

**技术亮点**：Sentinel 熔断降级 / 限流保护 / Redis Stream 异步削峰 / 系统可用性

### 问题 13：跨服务事务一致性怎么做

**面试官**："报名课程要创建订单又要扣课时，两个服务操作如何保证一致性？"

**你**：

这个场景我会用 Seata 的 AT 模式来处理分布式事务。

业务流程上，报名课程可能会同时涉及订单服务和课时服务，这两个操作都会放在一个全局事务里面执行。只要其中任意一个子服务执行失败，Seata 就会让整个事务链路一起回滚。

这样可以保证订单创建和课时扣减要么同时成功，要么同时失败，不会出现一个成功一个失败的脏数据情况。

**技术亮点**：Seata AT 模式 / 全局事务 / 跨服务一致性 / 原子性保障

### 问题 14：SQL 性能怎么优化

**面试官**："消课统计或查询接口慢，如何优化？"

**你**：

我一般会分三步来做。

第一步是索引优化。对学生 ID、课程 ID、日期这些高频查询条件建立索引，先解决最基础的全表扫描问题。

第二步是缓存优化。把一些高频查询结果或者统计结果放到 Redis 里，减少数据库压力。

第三步是 SQL 本身优化。对于复杂的多表关联查询，我会先看执行计划，再调整查询条件、关联顺序或者拆分查询，尽量避免低效 JOIN 和无意义扫描。

优化之后，接口响应通常可以从秒级降到毫秒级，系统可用性和用户体验都会明显好很多。

**技术亮点**：索引优化 / Redis 缓存 / EXPLAIN 执行计划 / 多表查询调优

### 问题 15：微服务调用失败怎么处理

**面试官**："服务间调用失败或超时，如何保证业务继续？"

**你**：

服务调用本身是通过 OpenFeign 做的，但真正保证稳定性的关键在于失败后的处理策略。

我会配合 Sentinel 做熔断和限流，如果某个服务连续超时或者异常率过高，就先把它熔断掉，避免持续请求把调用链拖死。

对于非核心数据，可以走降级逻辑，比如返回缓存数据、默认值，或者提示用户稍后重试。

如果是一些耗时但不强依赖实时返回的操作，就改成异步处理，让主流程先完成，后续再补偿。

整体目标不是“所有调用都必须立刻成功”，而是“即使局部失败，系统整体也还能继续提供服务”。

**技术亮点**：OpenFeign 调用 / Sentinel 熔断限流 / 降级兜底 / 异步补偿 / 可用性优先

**继续追问怎么接**

- 面试官继续问：降级一般返回什么？
  - 你可以接：看业务重要性，非核心数据可以返回缓存、默认值或者稍后重试提示，核心链路优先保证一致性。
- 面试官继续问：为什么不把所有失败都重试？
  - 你可以接：因为盲目重试可能放大雪崩，只有可重试且幂等的场景才适合重试。

### 问题 1：ArrayList vs LinkedList

**面试官**："ArrayList 和 LinkedList 的区别？什么时候用哪个？"

**你**：

ArrayList 底层是动态数组。查询快，因为可以通过下标直接计算地址 O(1) 时间找到元素。但是插入和删除慢，因为在中间插入或删除需要把后面所有元素往后移或往前移，最坏情况 O(n)。

LinkedList 底层是双向链表。插入和删除快，只要改前后节点的指针就行，O(1) 时间。但是查询慢，需要从头或从尾遍历，O(n)。

实际使用中，如果主要是查询操作，比如存向量检索结果列表，用 ArrayList。如果是频繁在中间插入删除，比如维护任务队列，用 LinkedList。

在 AI 面试助手项目里，chunk 列表和检索结果列表都是只查询不频繁修改，所以用的是 ArrayList。

**技术亮点**：数组连续内存 / 链表节点指针 / O(1)插入 vs O(n)查询

---

### 问题 2：HashMap 原理

**面试官**："HashMap 的底层结构是什么？JDK 1.8 有什么区别？"

**你**：

JDK 1.7 及之前，HashMap 底层是数组加链表。Put 时先用 key 的 hashCode 算桶位置，如果哈希冲突就链表头插。

JDK 1.8 做了优化：链表长度超过 8 就转成红黑树，查询效率从 O(n) 降到 O(log n)。红黑树节点比链表节点大，所以只有冲突足够多才转树。

HashMap 默认负载因子 0.75，当已占用桶数量超过 75% 就扩容为原来两倍。扩容操作很重，建议提前预估数据量避免频繁扩容。

**技术亮点**：数组+链表+红黑树 / 哈希冲突解决 / 红黑树转链条件 / 负载因子

---

### 问题 3：ConcurrentHashMap

**面试官**："ConcurrentHashMap 是怎么保证线程安全的？"

**你**：

JDK 1.7 用分段锁，每个段独立加锁，不同段的操作互不影响。

JDK 1.8 取消了分段锁，改用 CAS 加 synchronized。一个桶里只有一个线程能写，用 synchronized 锁住头节点；读操作大部分用 CAS 无锁实现，只有写才加锁。synchronized 锁住了头节点而不是整个表，并发度比分段锁更高。

**技术亮点**：JDK 1.7 分段锁 / JDK 1.8 CAS+synchronized / 写读分离

---

### 问题 4：synchronized vs ReentrantLock

**面试官**："synchronized 和 ReentrantLock 有什么区别？"

**你**：

synchronized 是 JVM 内置关键字，自动获取锁和释放锁，简单但功能有限。ReentrantLock 是 JDK 提供的类，功能更丰富。

synchronized 不支持公平锁和非公平锁切换，不支持打断。ReentrantLock 可以设为公平锁或非公平锁，支持 tryLock 尝试获取，获取不到可以立即返回，还支持 interrupt 打断等待中的线程。

性能方面，JDK 1.6 对 synchronized 做了很多优化，包括偏向锁、轻量级锁、自旋锁，高并发下性能差距已经很小了。

在教务系统里，简单的对象锁用 synchronized，复杂的场景比如 Redis 分布式锁需要可重入特性和超时控制，用 ReentrantLock 更灵活。

**技术亮点**：JVM 内置 / 可重入 / 公平/非公平 / tryLock / 打断机制 / JDK 1.6 优化

---

### 问题 5：线程池

**面试官**："线程池的核心参数有哪些？工作流程是什么？"

**你**：

线程池有 7 个核心参数。

corePoolSize 是核心线程数，池里一直保留的线程数量。maximumPoolSize 是最大线程数，核心线程满了之后最多能扩展到的线程数量。keepAliveTime 是非核心线程的空闲存活时间。unit 是存活时间的单位。workQueue 是存放待执行任务的队列。threadFactory 是创建线程的工厂，可以自定义线程名称和优先级。handler 是拒绝策略，队列满了且线程数达到最大时触发。

工作流程是：任务进来后先看核心线程是否满了，没满就创建核心线程执行。满了就进队列排队。队列满了且线程数没达到最大，就创建临时线程执行。队列满了且线程数也达到最大，就触发拒绝策略。

拒绝策略有四种：AbortPolicy 直接抛异常，CallerRunsPolicy 让调用线程自己执行，DiscardPolicy 默默丢弃任务不抛异常，DiscardOldestPolicy 丢弃队列里最老的任务。

**技术亮点**：7 参数 / 3 阶段流程 / 4 种拒绝策略 / 队列选择

### 问题 6：JUC 并发工具类

**面试官**："JUC 里常用的并发工具类有哪些？"

**你**：

**CountDownLatch**：倒数计数器，用于等待多个线程执行完成再继续。比如主线程等所有子任务执行完再汇总结果。

**CyclicBarrier**：循环栅栏，让一组线程相互等待，都到了栅栏位置再一起继续执行。适合多线程分阶段执行任务的场景。

**Semaphore**：信号量，控制同时访问某个资源的线程数量。比如限流，只允许 N 个并发请求同时访问。

**CompletableFuture**：异步编程利器，支持链式调用、并行执行、异常处理。在教务系统里用 CompletableFuture 并行调用三个 Feign 接口，耗时从三者之和降到最慢那个。

**技术亮点**：CountDownLatch 倒计时 / CyclicBarrier 循环同步 / Semaphore 限流 / CompletableFuture 并行调用

### 问题 7：ThreadLocal

**面试官**："ThreadLocal 是什么？有什么坑？"

**你**：

ThreadLocal 是线程本地变量。每个线程访问 ThreadLocal 拿到的都是该线程独立的副本，不同线程之间互不干扰。常用来传递上下文信息，比如用户 ID、TraceID。

主要坑是内存泄漏。ThreadLocalMap 的 Entry 引用了 ThreadLocal 对象，ThreadLocal 对象被回收后，Entry 还存在，但强引用链断了之后就成了内存泄漏。JDK 建议用完调用 remove 方法清除。实际项目中，在拦截器里设置 TraceID 后，一定要在 finally 里 remove 掉。

另一个坑是线程池场景。线程池里的线程是复用的，如果上一个任务遗留了 ThreadLocal 数据，下一个任务可能读到脏数据。所以在线程池环境下，用完 ThreadLocal 必须 remove。

**技术亮点**：线程副本隔离 / 内存泄漏场景 / remove 必要性 / 线程池坑

### 问题 8：JVM 和内存模型

**面试官**："你怎么理解 JVM 内存模型？项目里会关注什么？"

**你**：

JVM 运行时内存我会先按几个核心区域来理解：堆、栈、方法区，再加上程序计数器和本地方法栈。

栈主要存方法调用和局部变量，线程私有；堆主要存对象实例，是 GC 重点回收区域；方法区主要放类信息、常量和静态变量。

项目里我最关注的是两点。第一是对象创建太多会不会让堆压力变大，比如 AI 项目里文本切分、向量结果拼装、JSON 结构化输出，如果一次性堆太多对象，就容易触发频繁 GC。第二是线程池和异步任务多了以后，线程数和任务堆积会不会把内存顶高。

所以如果被问到项目里的 JVM 关注点，我一般会回答：不是去背垃圾回收器名字，而是会关注对象量、线程数、缓存规模和 GC 对接口延迟的影响。

**技术亮点**：堆 / 栈 / 方法区 / GC 性能影响 / 线程池内存压力

### 问题 9：异常处理怎么设计

**面试官**："项目里的异常处理你是怎么做的？"

**你**：

我会把异常处理分三层。

第一层是业务方法内部，对可预期的异常做兜底，比如 AI 项目里文件解析失败、格式不支持、向量化调用异常，这类错误我会捕获并记录清楚原因。

第二层是统一异常处理层，用全局异常处理器把异常转换成统一的响应结构，避免接口直接把堆栈暴露给前端。

第三层是日志和任务状态层。比如异步任务失败了，不能只打印一行日志就结束，还要把任务状态标成失败，必要时保留重试入口。

这样做的目的就是三件事：用户能看懂、开发能排查、任务不丢失。

**技术亮点**：局部捕获 + 全局异常处理 + 日志记录 + 任务状态兜底

### 问题 10：Stream 和 Lambda 在项目里怎么用

**面试官**："Java 8 的 Stream 和 Lambda 你怎么理解？项目里怎么用？"

**你**：

Lambda 本质上是把函数式写法带进 Java，可以把原来很多匿名内部类和循环逻辑写得更简洁。

Stream 则更适合做集合处理，比如 filter、map、collect、groupingBy 这些操作。它的优势是表达力强，尤其适合“先筛选、再转换、再聚合”这类流水线操作。

项目里 AI 面试助手做文本块处理、结果映射、结构化结果收集时，Stream 会很好用。教务系统里做一些统计列表转换、按条件筛选或者按字段分组时，也可以用 Stream 让代码更清楚。

不过我也会注意一个点：不要为了用 Stream 而用 Stream。特别是在很复杂的业务逻辑里，如果链式操作太长，反而不如普通 `for` 循环更好读。

**技术亮点**：Lambda 表达式 / Stream 流式处理 / filter-map-collect / 可读性权衡

### 问题 11：线程池高级配置怎么答

**面试官**："线程池除了核心参数，你还了解哪些高级配置？"

**你**：

如果面试官继续往深里问线程池，我一般会从阻塞队列、拒绝策略和线程工厂这三块展开。

阻塞队列常见有 ArrayBlockingQueue 和 LinkedBlockingQueue。前者是有界数组队列，容量固定，更适合明确控流；后者默认容量很大，如果不手动限制，极端情况下容易堆积太多任务。

拒绝策略常见有四种。AbortPolicy 直接抛异常，CallerRunsPolicy 让提交任务的线程自己执行，DiscardPolicy 静默丢弃，DiscardOldestPolicy 丢弃最旧任务。实际项目里如果想让系统降速但不直接报错，CallerRunsPolicy 会比较常见。

线程工厂主要是为了统一线程命名，这样日志排查时很方便，一看名字就知道是 AI 异步线程还是教务统计线程。

所以如果要把线程池答得像项目里真用过，我会强调：线程数、队列容量、拒绝策略和线程命名，四个都要一起考虑。

**技术亮点**：阻塞队列类型 / 拒绝策略 / 自定义线程工厂 / 线程命名与排障

**继续追问怎么接**

- 面试官继续问：你会优先选哪种拒绝策略？
  - 你可以接：很多业务里我会优先考虑 CallerRunsPolicy，让系统自然降速，而不是直接把任务丢掉。
- 面试官继续问：为什么线程命名重要？
  - 你可以接：因为线上排障时最先看到的往往是日志和线程名，命名清楚能大幅降低定位成本。

## 第五部分：数据库八股

### 问题 1：MySQL 索引

**面试官**："MySQL 索引的底层结构是什么？B+ 树有什么优势？"

**你**：

MySQL InnoDB 引擎的索引底层结构是 B+ 树。

B+ 树是一种多路平衡查找树，所有数据存在叶子节点，且叶子节点之间用链表相连。非叶子节点只存索引键和子节点指针，不存实际数据。

相比二叉树，B+ 树更矮更宽，层级更少，磁盘 IO 次数少。相比哈希索引，B+ 树支持范围查询和排序。

B+ 树还有一个特点是非叶子节点不存数据，每个节点能容纳更多索引键，同样数据量下比红黑树层级更少。

InnoDB 表会自动给主键建主键索引，也叫聚簇索引，数据直接存在叶子节点。普通索引也叫二级索引，叶子节点存主键值，查到主键后再查主键索引叫回表。

**技术亮点**：B+ 树多路平衡 / 链表范围查询 / 聚簇索引 / 二级索引+回表

---

### 问题 2：索引失效场景

**面试官**："什么情况下索引会失效？"

**你**：

常见失效场景有这几种。

第一种，索引列上用了函数或计算。比如 WHERE YEAR(create_time) = 2026，索引列被包裹了函数，索引失效。

第二种，类型不匹配。比如字段是字符串类型，查询时写了 WHERE age = 18，age 是数字，MySQL 会做隐式类型转换导致索引失效。

第三种，LIKE 以通配符开头。比如 WHERE name LIKE '%王'，前缀通配符导致无法利用索引有序性，只能全表扫描。

第四种，OR 连接条件。OR 两边有一边没用索引，整体可能失效。可以用 UNION 改写。

第五种，WHERE NOT IN、<>、IS NOT NULL 等条件，容易导致全表扫描。

**技术亮点**：函数索引 / 隐式类型转换 / 前缀通配符 / OR 改写

---

### 问题 3：事务隔离级别

**面试官**："MySQL 的事务隔离级别有哪些？分别解决什么问题？"

**你**：

MySQL 有四种隔离级别。

读未提交，一个事务能读到另一个事务未提交的数据，会有脏读问题。读已提交，事务只能读到其他事务已提交的数据，解决了脏读，但会有不可重复读问题——同一数据两次读结果不一样。可重复读，事务期间相同查询返回相同结果，解决了不可重复读，但会有幻读问题——同一条件两次查，第二次多出来一行。串行化，所有事务串行执行，完全避免幻读，但性能最差。

MySQL 默认用的是可重复读。InnoDB 在可重复读级别下，通过 MVCC 快照读解决了大部分幻读问题，只有当前读（SELECT FOR UPDATE）才需要 Gap 锁。

**技术亮点**：4 隔离级别 / 脏读/不可重复读/幻读 / MVCC 快照读 / Gap 锁

---

### 问题 4：MVCC 原理

**面试官**："MVCC 是怎么实现的？"

**你**：

MVCC 全称是多版本并发控制，核心思想是每个事务看到的都是数据的一个快照，写操作不会阻塞读操作。

InnoDB 里有两列隐藏字段：DB_TRX_ID 记录最后修改的事务 ID，DB_ROLL_PTR 指向 undo log 的指针。

每行数据有多个版本，通过 undo log 串联起来。查询时，事务只能看到 trx_id 小于等于自己且不在活跃事务列表里的数据版本。

读已提交每次查询都生成新快照，所以每次可能读到不同的已提交数据。可重复读在事务开始时生成快照，整个事务期间都用同一个快照。

更新时，数据行复制一份新版本，原版本的 undo log 指针指向新版本，通过版本链实现多版本。

**技术亮点**：undo log 版本链 / trx_id / 快照读 vs 当前读 / 读已提交 vs 可重复读

---

### 问题 5：SQL 优化思路

**面试官**："SQL 慢怎么排查和优化？"

**你**：

排查用 EXPLAIN 分析执行计划，看是否走了索引、是否有全表扫描、是否回表次数过多。

常见优化手段。

第一，对高频查询字段建索引，包括 WHERE 条件列、JOIN 的连接列、ORDER BY 的排序列。

第二，避免 SELECT *，只查需要的字段，减少网络传输和内存占用。

第三，用分页 LIMIT 限制返回条数，不要一次性返回全表数据。

第四，避免在索引列上用函数，让索引列保持原值。

第五，批量操作替代循环单条操作，减少数据库交互次数。

在教务系统里，学员列表查询原来用了 SELECT * 加多表 JOIN，导致全表扫描和大量回表。加了分页和只查必要字段后，查询时间从 2 秒降到了 200 毫秒。

**技术亮点**：EXPLAIN 分析 / 索引优化 / 分页 / 覆盖索引 / 批量操作

### 问题 6：ACID 在项目里怎么落地

**面试官**："ACID 你在项目里是怎么落地理解的？"

**你**：

ACID 我不会只背定义，我会直接结合项目去讲。

原子性就是一个操作要么全成功，要么全失败。比如教务系统里单服务写库时，用 Spring 事务保证报名记录写入和本地扣减逻辑一起提交。

一致性就是事务前后数据状态要符合业务规则。比如报名成功之后，订单状态、课时余额这些字段不能彼此矛盾。

隔离性就是多个事务并发执行时互相尽量不干扰，这块要靠隔离级别、MVCC 和必要的锁来保证。

持久性就是事务一旦提交，数据就不会因为服务进程结束而丢失，这本质上依赖数据库的日志和落盘机制。

如果是跨服务场景，我会继续补一句：单服务靠 Spring 事务，跨服务靠 Seata 做全局事务，把 ACID 里的原子性和一致性尽量往分布式场景延伸。

**技术亮点**：ACID 概念项目化表达 / Spring 本地事务 / Seata 全局事务 / 业务一致性

**继续追问怎么接**

- 面试官继续问：一致性和原子性有什么区别？
  - 你可以接：原子性强调“要么一起成功要么一起失败”，一致性强调“结果必须符合业务规则”，两者相关但不完全一样。
- 面试官继续问：为什么跨服务时 ACID 更难？
  - 你可以接：因为资源不在同一个数据库里，本地事务边界失效了，需要额外协调者去保证整体一致。

### 问题 7：MVCC 在项目里怎么讲得更像实战

**面试官**："你提到 MVCC，那在项目里它真正帮你解决了什么问题？"

**你**：

MVCC 的核心价值是让读操作和写操作尽量不要互相阻塞。

如果没有 MVCC，很多并发查询场景下读写容易互相卡住，接口响应会明显变慢。比如教务系统里很多用户同时查课程、查消课记录、查学生信息，如果每次读都要等写结束，体验会很差。

有了 MVCC 之后，普通查询更多走的是快照读，读请求看到的是某个时刻的一致性视图，所以很多读操作不需要跟写操作硬碰硬。

当然，如果是 `select ... for update` 这种当前读，那又是另一套逻辑，会涉及行锁、间隙锁这些机制。

所以如果面试官追问 MVCC，我会总结成一句：它不是“让数据更安全”，而是“在保证一致性前提下，让并发读更快”。

**技术亮点**：快照读 / 当前读 / 并发读性能 / MVCC 的项目价值

**继续追问怎么接**

- 面试官继续问：MVCC 能完全替代锁吗？
  - 你可以接：不能。普通读很多时候能靠 MVCC，但更新、当前读、强一致场景还是要靠锁。
- 面试官继续问：为什么 MVCC 总和隔离级别一起考？
  - 你可以接：因为它本来就是隔离性的一部分，特别是读已提交和可重复读的区别，很大程度就体现在快照生成时机上。

## 第六部分：Redis 八股

### 问题 1：Redis 数据类型

**面试官**："Redis 支持哪些数据类型？分别在项目里怎么用？"

**你**：

String 是最基本的类型，存单个值，项目里缓存 Token、缓存单个学员信息。

Hash 存对象结构，比如缓存完整学员档案，字段对应对象的属性，HMSET 一次存整个对象。

List 是链表，支持 LPUSH/RPOP，常用作简单的任务队列，实现先进先出。

Set 是无序不重复集合，用于去重，比如存已处理的任务 ID，重复任务直接跳过。

ZSet 是有序集合，每个元素带分数，用于排行榜场景，比如按课时消耗排序。

在教务系统里，学员课时排行榜用 ZSet，key 是 student:rank，分数是已消耗课时。

**技术亮点**：String 值 / Hash 对象 / List 队列 / Set 去重 / ZSet 排行

### 问题 2：缓存三大问题

**面试官**："Redis 缓存穿透、击穿、雪崩是什么？怎么解决？"

**你**：

**缓存穿透**：查询一个不存在的数据，Redis 没有，数据库也没有，每次都打到数据库。比如恶意请求用不存在的学员 ID 查询。

解决：把空结果也缓存起来，过期时间短一些比如 5 分钟。或者用布隆过滤器，把所有存在的学员 ID 提前放入过滤器，请求进来先过过滤器，不存在的直接返回。

**缓存击穿**：一个热点 key 过期瞬间，大量请求同时涌入直接查数据库。比如学员详情页缓存过期，同时有 1000 个请求进来。

解决：用分布式锁，只允许一个线程查库，其他线程等待。或者把热点数据设置为永不过期，异步更新。

**缓存雪崩**：大量 key 同时过期，或者 Redis 本身挂了，导致大量请求同时打到数据库。

解决：过期时间加随机偏移量，避免同时过期。Redis 部署用主从+哨兵保证高可用。接口层加限流和降级兜底。

**技术亮点**：空值缓存 / 布隆过滤器 / 分布式锁 / 热点永不过期 / 限流降级

---

### 问题 3：Redis 分布式锁

**面试官**："Redis 分布式锁怎么实现？有什么坑？"

**你**：

分布式锁用 SET key value NX PX ttl 命令加锁，NX 保证只有锁不存在时才设置成功，PX 设置过期时间防止死锁。

释放锁用 Lua 脚本，先判断锁值是否是自己存的 UUID，是才删除，保证原子性且只删除自己的锁。

主要坑有三个。

第一，锁值不能用固定字符串，必须用 UUID，否则释放锁时可能删掉别人刚获取的锁。

第二，锁必须设置过期时间，否则加锁后进程崩溃锁就永远无法释放。

第三，业务执行时间可能超过锁过期时间，需要续期也叫看门狗机制。Redisson 实现了这个功能。

在教务系统里，定时关闭超时课时记录用的就是 Redis 分布式锁，多实例只允许一个节点执行。

**技术亮点**：SET NX PX / Lua 脚本原子性 / UUID 锁值 / 过期时间防死锁 / 看门狗续期

---

### 问题 4：Redis 和 Memcached 区别

**面试官**："Redis 和 Memcached 的区别？"

**你**：

数据类型方面，Redis 支持 String、Hash、List、Set、ZSet 五种类型，Memcached 只支持 String。

持久化方面，Redis 支持 RDB 和 AOF 两种持久化方式，Memcached 不支持持久化，重启数据丢失。

内存管理方面，Memcached 用 slab 分配，内存预分配不浪费但也不够灵活。Redis 用自己的内存管理，可以动态分配和回收。

集群方面，Redis 支持主从、哨兵、Cluster 模式，Memcached 用客户端分片。

选择的话，如果只需要缓存简单字符串用 Memcached，性能稍高。如果需要丰富数据结构和持久化，用 Redis。

**技术亮点**：5 数据类型 vs 纯 String / RDB+AOF 持久化 / slab 分配 / 集群模式

## 第七部分：Spring 八股

### 问题 1：Spring IoC 原理

**面试官**："Spring IoC 是什么？是怎么工作的？"

**你**：

IoC 是控制反转，把对象的创建和依赖管理从应用代码移到框架。应用程序不再自己 new 对象，而是等框架创建好之后注入进来。

Spring IoC 的工作流程分三步。第一步是启动时扫描含 @Component、@Service、@Repository 等注解的类，解析并注册成 BeanDefinition。第二步是根据 BeanDefinition 创建 Bean 实例，如果是单例就先缓存起来。第三步是依赖注入，把 Bean 的依赖属性填充进去，包括 @Autowired 注入和 @Value 注值。

Spring 还支持延迟初始化（@Lazy），第一次使用时才创建。还支持自定义 BeanFactoryPostProcessor 在 Bean 创建前修改 BeanDefinition。

**技术亮点**：BeanDefinition / 依赖注入 / 单例缓存 / @Lazy / BeanFactoryPostProcessor

**继续追问怎么接**

- 面试官继续问：你项目里 IoC 最直观的体现是什么？
  - 你可以接：最直观的就是 Controller、Service、Mapper 这些对象都不是我自己 new 的，而是 Spring 托管和注入的。
- 面试官继续问：为什么 IoC 比手动 new 好？
  - 你可以接：因为依赖关系更清晰、可测试性更高，也更方便统一管理生命周期。

---

### 问题 2：Spring AOP 原理

**面试官**："Spring AOP 的原理是什么？JDK 动态代理和 CGLIB 有什么区别？"

**你**：

AOP 是面向切面编程，核心是把和业务无关的公共逻辑抽出来统一处理，比如日志、事务、权限校验。

Spring AOP 基于代理模式实现。目标对象实现了接口，默认用 JDK 动态代理，生成一个实现同样接口的代理类，通过反射调用目标方法，在前后加增强逻辑。

如果目标对象没有实现接口，用 CGLIB 动态代理，通过继承目标类生成子类，在子类方法前后加增强逻辑。CGLIB 不能代理 final 类和 final 方法。

Spring 默认策略是如果目标类有接口用 JDK 动态代理，没有接口用 CGLIB。可以通过配置强制使用 CGLIB。

项目中自定义的 @DataScope 数据权限注解和 @RequireRoles 角色校验注解，就是用 AOP 切面统一拦截，在 Controller 方法执行前判断权限，不通过抛异常。

**技术亮点**：代理模式 / JDK 动态代理接口 / CGLIB 继承生成子类 / Spring AOP 应用

**继续追问怎么接**

- 面试官继续问：你项目里哪里最能体现 AOP？
  - 你可以接：统一异常、日志、权限校验、事务，这几个地方都是典型 AOP 场景。
- 面试官继续问：为什么 final 方法不能被 CGLIB 代理？
  - 你可以接：因为 CGLIB 本质是生成子类，final 方法不能被重写。

---

### 问题 3：Spring 事务传播行为

**面试官**："Spring 事务的传播行为有哪几种？"

**你**：

REQUIRED 是默认行为，如果当前有事务就加入，没有就新建事务。最常用。REQUIRES_NEW 每次都新建事务，挂起当前事务，两个事务相互独立。SUPPORTS 如果当前有事务就加入，没有就不开事务。NOT_SUPPORTED 如果当前有事务就挂起，以非事务执行。MANDATORY 必须在事务里执行，没有就抛异常。NEVER 必须不在事务里执行，有事务就抛异常。NESTED 如果当前有事务就嵌套执行，没有就新建事务。

教务系统里，生成课表的异步任务用 REQUIRES_NEW，单独事务不受主事务影响。发送通知用 REQUIRED，加入主事务，主事务失败通知也不发。

**技术亮点**：7 种传播行为 / REQUIRED 默认 / REQUIRES_NEW 独立事务 / NESTED 嵌套

**继续追问怎么接**

- 面试官继续问：什么时候会想到 REQUIRES_NEW？
  - 你可以接：像异步任务、日志记录、补偿动作这类不希望跟主事务一起回滚的场景，我会优先想到 REQUIRES_NEW。
- 面试官继续问：为什么 REQUIRED 最常用？
  - 你可以接：因为大多数业务就是当前有事务就加入，没有就自己开一个，默认最贴近常规需求。

---

### 问题 4：RESTful 接口怎么设计

**面试官**："你平时接口是怎么按 RESTful 风格设计的？"

**你**：

我一般会先按资源来设计接口，而不是按动作名堆接口。

比如学生、课程、报名记录这些都是资源。查列表用 GET，新增用 POST，修改用 PUT 或 PATCH，删除用 DELETE。这样接口语义会比较统一，后期维护也更清晰。

另外我会注意三个细节：第一是分页参数要统一，比如 `pageNum`、`pageSize`；第二是状态码和响应结构要统一；第三是幂等性要有意识，比如重复提交时，接口应该尽量避免把同一业务执行两次。

教务系统里如果设计报名接口、课程管理接口，用 RESTful 风格会让前后端协作成本更低。

**技术亮点**：资源导向设计 / GET-POST-PUT-DELETE / 分页统一 / 幂等性意识

**继续追问怎么接**

- 面试官继续问：为什么幂等性重要？
  - 你可以接：因为网络重试、前端重复点击都很常见，不考虑幂等就可能把同一业务执行两次。
- 面试官继续问：PUT 和 POST 你怎么区分？
  - 你可以接：简单理解就是 POST 更偏创建，PUT 更偏对已有资源做整体更新。

## 第八部分：分布式基础

### 问题 1：CAP 定理

**面试官**："CAP 定理是什么？"

**你**：

CAP 定理说的是分布式系统不可能同时满足一致性、可用性和分区容错性，只能三选二。

C 是一致性，所有节点在同一时刻看到的数据相同。A 是可用性，每次请求都能在有限时间内得到响应。P 是分区容错，网络分区时系统仍能运行。

分布式系统必须保证 P，所以实际上是在 C 和 A 之间权衡。

追求强一致性的系统比如 ZooKeeper、Etcd，选择 CP，可用性稍弱。网络分区时 ZooKeeper 会停止响应，保证数据一致。

追求高可用的系统比如 Redis、Eureka，选择 AP，允许短暂数据不一致。

MySQL 主从同步是 AP 模型，异步复制允许从库数据略微滞后。

**技术亮点**：C 一致性 / A 可用性 / P 分区容错 / CP vs AP 取舍

---

### 问题 2：BASE 理论

**面试官**："BASE 理论是什么？"

**你**：

BASE 是对 CAP 中一致性和可用性权衡的实践总结。

Basically Available 基本可用，指系统出现故障时允许损失部分可用性，保证核心功能可用。比如双十一时下单功能可用，但查询功能降级。

Soft state 软状态，指数据状态可以在一段时间内不一致，允许存在中间状态。比如订单状态显示"处理中"，过几秒才变成"已支付"。

Eventually consistent 最终一致，指系统在没有新写入的情况下，最终所有节点数据达到一致。比如 MySQL 主从同步，延迟几秒后从库数据和主库一致。

Seata 的 AT 模式就是 BASE 理论的实现，正常情况下保证最终一致性，分支事务执行后立即释放锁，性能优先。

**技术亮点**：基本可用 / 软状态 / 最终一致 / Seata AT 模式应用

## 第九部分：刁钻追问

### 追问 1：Seata 和Saga 的区别

**面试官**："Seata 和 Saga 模式有什么区别？"

**你**：

Seata AT 模式靠记录逆向 SQL 自动回滚，适合短事务，对业务代码零侵入。缺点是需要解析 SQL，有一定性能开销，且不支持跨库跨表。

Saga 模式没有回滚机制，靠正向补偿操作逐步回退。适合长事务和事务链较长的场景，比如跨多个外部系统调用。缺点是需要为每个步骤写补偿逻辑。

Seata 适合项目里这种本地事务跨服务场景，Saga 适合外部系统多、流程长的场景。

---

### 追问 2：Redis 主从同步原理

**面试官**："Redis 主从同步是怎么工作的？"

**你**：

Redis 主从同步分两步。第一步是全量复制，主库收到 SLAVEOF 命令后用 BGSAVE 生成 RDB 文件，发送给从库，从库加载到内存。第二步是增量复制，主库在执行写命令时把命令发给从库，从库执行同样的命令。

从库断开重连后，主库判断从库的 repl_backlog_buffer 里的偏移量，在 buffer 里找差异部分发给从库，而不是全量同步。

**技术亮点**：全量复制 BGSAVE / 增量复制 repl_backlog / 断线重连增量同步
