# 面试终极速记表（实习生版）

## 概述

本页把 AI 面试助手项目、教务管理系统、Java 八股/JUC、SQL/Redis 高频题，以及常见刁钻追问整理成一份总速记表。适合面试前最后一轮通读，也适合作为 [[04-完整逐字稿版]] 和 [[05-面试当天速翻版]] 之间的中间层材料。

## 详细内容

## 1. 自我介绍

**面试官问**

> 请做一个自我介绍。

**顺口回答**

面试官您好，我叫曹家宁，大连交通大学软件工程本科二年级，求职方向是 Java 后端实习。

在校期间我系统学习了 Java 基础、集合、异常、泛型、反射、Stream，以及线程池、乐观锁、悲观锁等并发基础。

技术栈方面，我主要使用 Spring Boot、MyBatis/MyBatis-Plus、Spring Cloud Alibaba，包括 Nacos、Gateway、OpenFeign、Sentinel、Seata；数据库用 MySQL 和 PostgreSQL，缓存用 Redis。

项目方面，我做过两个项目：一个是 AI 面试助手，核心是 RAG、Redis Stream 和 Prompt 控制；另一个是教务管理系统，核心是微服务、分布式事务、Redis 缓存和 Sentinel 限流。

我希望通过实习参与真实业务开发，在工程能力和业务理解上继续提升。

## 2. AI 面试助手项目（5个深度场景）

| 场景 | 面试官问法 | 顺口回答 | 技术亮点 / 项目落地 | 追问策略 |
|---|---|---|---|---|
| 向量检索精度 | “向量检索有时候不准，怎么优化？” | chunk 太大或者 topK 不合理时容易混进无关内容，所以我会调整 chunk 粒度、控制 topK，并且 Prompt 里只用这些检索片段。 | chunk 切分 + topK 控制 + 精确向量检索 | topK 怎么选：通过实验调试选最优值 |
| 异步任务处理 | “上传简历处理慢怎么解决？” | 文件解析、向量化、报告生成都比较耗时，所以我用 Redis Stream 做异步任务，接口先快速返回，消费者异步执行。 | Redis Stream 异步任务 + 快速响应 + 可靠性 | 任务失败：重试、pending 机制、兜底扫描 |
| AI 回答不准 | “AI 回答不准怎么办？” | 我从三层控制：向量检索只给相关 chunk，Prompt 约束“只能根据提供内容回答”，输出再做结构化。 | RAG + Prompt 约束 + 输出结构化 | Prompt 怎么写：举“无法判断就直说”的例子 |
| 开源改造 | “和 Snailclimb 项目有什么区别？” | 我参考了它的架构思路，但 RAG 检索、异步任务、结构化输出都是我自己按简历场景做的改造实现。 | 独立实现 + 场景化优化 | 具体改了什么：chunk、topK、Prompt、异步链路 |
| 工程化/可扩展性 | “项目上线了吗？系统可用性？” | 目前是个人作品集，没有大规模用户，但接口分层、异常处理、异步任务、Docker 部署都按真实工程方式做了。 | 工程化 + 可扩展性 + 可落地 | 怎么扩展：Docker 部署、后续接权限和监控 |

## 3. 教务管理系统（5个深度场景）

| 场景 | 面试官问法 | 顺口回答 | 技术亮点 / 项目落地 | 追问策略 |
|---|---|---|---|---|
| Sentinel 限流 | “如何用 Sentinel 做接口限流？规则怎么配置？” | 核心接口配 QPS 限流，比如课程查询和消课接口；热点课程 ID 再配热点参数限流，超阈值就降级。 | QPS 限流 + 热点参数保护 + 高并发稳定性 | 降级策略：缓存/默认值；规则热加载 |
| 接口稳定性 | “微服务调用多，接口不稳定，怎么保证？” | 我会从熔断降级、限流保护和 Redis Stream 异步任务三个方向来保证整体稳定性。 | Sentinel + 异步任务 + 接口可用性 | 追问：限流和熔断规则怎么配 |
| 分布式事务 | “报名课程时创建订单和扣课时，如何保证一致性？” | 用 Seata AT 模式把多个服务操作放到同一个全局事务里，任何一个子服务失败就整体回滚。 | 分布式事务 + 原子性保证 | 追问：事务异常时如何回滚 |
| SQL 性能优化 | “消课统计慢怎么优化？” | 高频查询字段加索引，统计结果缓存到 Redis，再配合 EXPLAIN 优化复杂多表查询。 | SQL 优化 + Redis 缓存 + 接口性能提升 | 追问：缓存失效怎么处理 |
| 微服务异常处理 | “服务调用失败或超时怎么办？” | OpenFeign 做调用，Sentinel 做熔断和限流，非核心数据降级，耗时任务异步处理。 | 熔断/限流 + 异步任务 + 系统可用性 | 追问：降级返回什么 |

## 4. Java八股 / JUC / 线程池

| 关键词 | 顺口回答 | 技术亮点 / 项目落地 |
|---|---|---|
| ArrayList vs LinkedList | ArrayList 查询快，LinkedList 插入删除快，核心看业务操作偏读还是偏写。 | 集合选择 + 性能权衡 |
| HashMap vs ConcurrentHashMap | HashMap 线程不安全；ConcurrentHashMap 通过 CAS 和局部加锁提升并发安全。 | 并发集合 |
| synchronized vs ReentrantLock | synchronized 更简单；ReentrantLock 支持公平锁、可中断、Condition。 | 并发控制 |
| CAS | Compare And Swap，无锁原子操作，常见于 Atomic 类和并发容器。 | 无锁并发 |
| volatile | 保证可见性并禁止指令重排，常用于状态标记。 | 轻量同步 |
| 线程池核心参数 | 核心线程数、最大线程数、队列、拒绝策略是最关键的四块。 | 并发控制 |
| 线程池执行流程 | 核心线程 -> 队列 -> 非核心线程 -> 拒绝策略。 | 异步任务管理 |
| Executors 限制 | 默认工厂方法不够安全，容易 OOM，实际更推荐手动建 ThreadPoolExecutor。 | 工程化意识 |
| 项目落地 | AI 项目做异步解析和报告生成，教务系统做异步统计和削峰。 | 异步 + 高并发 + 稳定性 |

## 5. SQL / Redis 高频题

| 关键词 | 顺口回答 | 技术点 |
|---|---|---|
| SQL 索引 | 高频字段加索引，优先避免全表扫描。 | 查询优化 |
| 事务 ACID / MVCC | 单服务事务靠 Spring，跨服务事务靠 Seata，多版本并发控制提升读写性能。 | 一致性 + 并发优化 |
| Redis 数据类型 | String、Hash、List、Set、ZSet，各自适合缓存、对象、队列、去重、排行榜。 | 缓存和数据结构应用 |
| 缓存穿透 | 对不存在数据的查询打到数据库，靠空值缓存或布隆过滤器处理。 | 数据保护 |
| 缓存击穿 | 热点 key 失效瞬间被并发打穿，靠互斥锁或逻辑过期处理。 | 并发防护 |
| 缓存雪崩 | 大量 key 同时过期，靠 TTL 随机化和高可用部署兜底。 | 系统可用性 |

## 7. ACID / MVCC / 分布式事务

| 场景 | 面试官问法 | 顺口回答（项目结合） | 技术亮点 / 项目落地 | 追问扩展策略 |
|---|---|---|---|---|
| 单服务事务 | “数据库事务 ACID 各个属性是什么？” | ACID 就是原子性、一致性、隔离性、持久性。项目里单服务内部的报名或写库操作，我会用 Spring 事务保证原子性和一致性。 | Spring 本地事务 / ACID 基础 | 隔离级别怎么选、如何避免脏读和不可重复读 |
| 跨服务事务 | “报名课程跨多个服务，如何保证 ACID？” | 跨服务时我会用 Seata AT 模式，把学生、订单、消课这些服务放到同一个全局事务里，只要有一个子服务失败，整体就回滚。 | Seata 全局事务 / 跨服务原子性与一致性 | 事务失败怎么处理：回滚、日志、重试 |
| MVCC | “MVCC 在项目中怎么用？” | MVCC 的作用是让并发读写尽量不互相阻塞。比如教务系统里多个用户同时查课程或消课记录，读操作不会因为写操作就完全卡住，并发性能会更好。 | 并发读写优化 / 数据隔离 | 读写冲突怎么办：隔离级别 + 锁策略 |
| 高并发接口 | “接口高并发时，事务如何保证性能？” | 我会把核心一致性仍然交给事务保证，但外围配合 Redis 缓存、异步任务和 Sentinel 限流，把真正重的请求挡在事务之外。 | 高并发下的事务性能平衡 | 缓存击穿和过期怎么处理 |
| 异常处理 | “如果分布式事务中一部分服务失败，你如何排查和恢复？” | 我会先看日志和调用链，判断是 SQL、缓存还是服务调用问题；恢复上依赖 Seata 的回滚机制，再结合异常日志和必要重试保证一致性。 | 日志排查 / 事务恢复 / 可观测性 | 生产环境怎么监控：Seata 控制台、告警、日志 |

### 口语化总结

> 在项目里，ACID 保证操作原子、一致、隔离和持久。单服务用 Spring 事务，跨服务用 Seata AT 模式做全局事务。MVCC 保证并发读写不完全互相阻塞，高并发接口再配合 Redis 缓存、异步任务和 Sentinel 限流，兼顾性能、稳定性和一致性。
