## 二、项目一：AI 面试助手（RAG 简历分析）

### 2.1 项目整体介绍

**面试官问：这个项目是做什么的？为什么做？**

这个项目叫 AI 面试助手，是一个基于 RAG 的简历分析和面试辅助系统。

用户上传 PDF 或 DOCX 格式的简历后端自动解析文本内容，清洗脏数据，然后按段落或固定长度切成 chunk，通过 Embedding 模型把文本转成向量，存入 PostgreSQL 加 pgvector 扩展的向量库。用户提问时，先把问题转成向量，在向量库里检索最相似的 N 个 chunk，拼接成 Prompt 发给大模型生成回答。

做这个项目有两个原因：一是让 AI 回答精准，基于真实简历内容而不是胡编；二是练习微服务架构、向量数据库和异步任务处理这些工程能力。

### 2.2 RAG 相关

**面试官问：RAG 是什么？项目里怎么用？**

RAG 是 Retrieval-Augmented Generation，检索增强生成。核心思想是先从知识库检索相关内容，再把检索结果拼进 Prompt 里让大模型生成回答。这样有两个好处：一是回答有依据，不是胡编；二是可以溯源到原文，用户能核验。

项目里的流程是：用户上传简历 → 切 chunk → 向量化 → 存入 pgvector。用户提问时：问题向量化 → 检索 top-K 最相似的 chunk → 拼接 Prompt → 调用大模型生成回答。

**面试官问：为什么用 pgvector，不用 Milvus 或 Chroma？**

pgvector 是 PostgreSQL 的向量扩展，优势是轻量、无需额外部署组件，项目已有 PostgreSQL 直接启用扩展就行。Milvus 需要单独部署运维，Chroma 是 Python 库不兼容 Java 项目。数据量中等的情况下 pgvector 完全够用，百万级向量检索延迟也在可接受范围。

**面试官问：文本为什么要切分？直接整篇向量化不行吗？**

不行。简历通常很长，常见的一页到三页，直接转成一个向量粒度太粗，检索命中率很低。比如用户问"这个人的项目管理经验怎么样"，如果整篇文档是一个向量，那检索结果要么太宽要么太窄。切成 chunk 后，每个 chunk 只包含一段独立内容，语义更集中，检索相关性更高。

切分策略我用的是重叠切分——每 500 个字符切一个 chunk，相邻 chunk 之间有 50 个字符的重叠，保证上下文不被打断。

**面试官问：向量检索的原理是什么？**

向量检索的核心是计算相似度。常用的是余弦相似度或者欧氏距离。我项目里用的是余弦相似度，算的是两个向量夹角的余弦值，越接近 1 越相似。pgvector 底层支持近似最近邻搜索（ANN），用的是 HNSW 算法，不用遍历全表就能找到最相似的向量，时间复杂度从 O(n) 降到 O(log n)，百万级数据也能做到毫秒级检索。

### 2.3 异步任务处理

**面试官问：上传简历后解析、切分、向量化、生成报告，这些操作同步执行吗？**

不是，用的 Redis Stream 做异步处理。用户上传简历后接口立即返回，后端把任务扔进 Stream，消费者异步完成整个流程：解析 → 清洗 → 切分 → 向量化 → 存储 → 生成报告。用户可以通过轮询或 WebSocket 查询任务进度。

选 Redis Stream 的原因是项目本身已有 Redis，不需要额外引入 RabbitMQ 等组件，而且 Stream 支持消息持久化和消费者组，方便多消费者负载均衡。

### 2.4 项目难点

**面试官问：项目里遇到过什么难点？怎么解决的？**

有两个主要难点。

第一个是检索精度不够。早期用整篇文档向量化，检索出来的结果相关性很差。解决思路是改用重叠 chunk 切分，每个 chunk 控制在 500 字符，保留语义完整性；同时调整 topK 参数，从 3 增加到 5，让回答有更丰富的上下文。

第二个是接口响应慢。向量化和大模型调用都是耗时操作，同步执行用户要等很久。解决是用 Redis Stream 异步化，接口立即返回任务 ID，用户轮询查进度。

### 2.5 项目真实性追问

**面试官问：这个项目是不是跟着教程做的？**

参考过 GitHub 上 Snailclimb 的 interview-guide 项目，学习了 RAG 的架构思路和向量数据库的使用方法。但我没有直接复制，而是针对简历分析这个具体场景做了改造：调整了文本切分策略、适配了简历解析的格式清洗、设计了结构化输出格式（优势/风险/改进建议），并独立实现了 Redis Stream 异步任务和 pgvector 集成。

### 2.6 项目是否上线

**面试官问：项目上线了吗？有真实用户吗？**

这是我的个人项目，没有大规模上线。但工程层面是完整的：接口分层、统一的异常处理、异步任务管理、Docker 容器化部署。项目架构可以扩展到生产级别，核心链路都跑通了。

## 三、项目二：教务管理系统

### 3.1 项目整体介绍

**面试官问：这个项目是做什么的？**

教务管理系统是一个微服务架构的管理平台，主要功能包括学生管理、课程管理、课时管理和考勤记录。我负责后端全部接口、数据库表设计、微服务集成和 Redis 缓存优化。

做这个项目的目的是练习微服务架构的实战能力，包括服务拆分、注册发现、远程调用、分布式事务和缓存设计。

### 3.2 微服务架构

**面试官问：为什么要拆成微服务？**

两个原因。一是业务边界清晰，学生服务、课程服务、课时服务各自独立，职责不重叠，拆开后降低耦合，方便独立开发部署。二是想通过这个项目练习 Spring Cloud Alibaba 全家桶，了解微服务组件在真实项目中的配合方式。

小规模项目其实可以单体，但拆微服务的主要价值是练习服务治理和分布式能力。

**面试官问：微服务组件各自负责什么？**

**Nacos**：服务的注册中心和配置中心。每个服务启动时向 Nacos 注册，消费者通过 Nacos 发现服务地址；同时所有配置集中在 Nacos 管理，修改配置不用重启服务。

**Gateway**：统一网关，所有请求先到网关。负责路由转发、登录鉴权、限流熔断。

**OpenFeign**：声明式 HTTP 客户端。定义一个接口打注解就能发 HTTP 请求，不用手写 HttpClient，自动完成序列化反序列化。

**Sentinel**：流量控制和熔断降级。当某个接口 QPS 过高或者响应超时，Sentinel 自动降级返回兜底数据，保护系统不被冲垮。

**Seata**：分布式事务。跨服务操作时保证数据一致性，比如学员报名课程需要同时创建订单和扣减课时，Seata 保证这两个操作要么同时成功要么同时回滚。

### 3.3 分布式事务

**面试官问：跨服务的事务怎么保证一致性？**

用的是 Seata 的 AT 模式。学员报名课程的业务流程涉及两个服务：订单服务创建订单，课时服务扣减课时。如果不用分布式事务，网络抖动或者某个服务挂了，就会出现订单创建成功但课时没扣掉，数据就不一致了。

Seata AT 模式的工作原理是：业务 SQL 执行前拍照，执行后比对数据快照，如果有问题就逆向回滚。整个过程对业务代码透明，只需要在 @GlobalTransactional 注解标注跨服务方法即可。

### 3.4 接口性能

**面试官问：接口响应慢怎么排查和解决？**

排查思路是用 Arthas trace 定位慢在哪一步。常见原因有三个：一是多表关联查询没建索引，导致全表扫描；二是远程 Feign 调用串行执行；三是热点数据每次都查数据库。

我的解决思路：高频查询字段加索引；多个远程调用用 CompletableFuture 并行执行；热点数据用 Redis 缓存。经过优化接口响应从平均 800 毫秒降到 300 毫秒左右。

### 3.5 Redis 使用

**面试官问：Redis 在项目里怎么用？**

两个用法。一是缓存学生信息和课程信息，减少数据库查询压力。比如学生详情页，用户频繁访问但数据变化少，存 Redis 里几秒内重复访问不用查库。二是作为异步任务队列，用 Redis Stream 削峰。

### 3.6 项目难点

**面试官问：项目里最难的部分是什么？**

最复杂的是 Seata 分布式事务的配置和调试。AT 模式依赖 UNDO_LOG 表，每次业务 SQL 都要记录逆向 SQL，网络延迟或者 Seata Server 抖动会导致全局锁超时，需要合理配置 seata.grouplist 和事务超时时间。排查这个问题花了挺长时间，最后通过调整事务超时和 Seata 服务端性能参数解决了。

## 五、刁钻追问

**面试官问：如果 AI 回答跑偏了怎么发现和解决？**

三方面控制：一是向量检索层保证 top-K 结果都是相关 chunk，如果返回的 chunk 本身就不相关，再好的模型也答不准；二是 Prompt 里明确限制"只能根据提供的简历内容回答，不知道的说不知道"；三是结构化输出后可以校验格式完整性，比如优势/风险/改进建议三个字段必须有内容，否则重试。

**面试官问：Redis Stream 消费者挂了怎么办？**

Redis Stream 支持消费者组，消费者挂掉后消息会 pending，不会丢失。消费者恢复后可以重新读取 pending 消息。另外我还加了定时任务扫描 pending 消息，如果超过一定时间没处理就重新投递，防止消息彻底丢失。

**面试官问：Sentinel 熔断和 Hystrix 有什么区别？**

Sentinel 是阿里巴巴开源的，核心区别在于限流维度更丰富——Hystrix 主要靠线程池隔离，Sentinel 支持按 QPS、线程数、调用关系等多维度限流和熔断。另外 Sentinel 有控制台可以实时查看请求分布和限流效果，开发调试更方便。
