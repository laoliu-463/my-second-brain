## 二、项目一：AI 面试助手

### 2.1 项目介绍（30 秒）

> **提示**：RAG 流程 → 简历上传 → 切 chunk → 向量化 → pgvector 存储 → 提问检索 → 大模型生成

**关键词**：RAG / 检索增强生成 / 向量数据库 / 异步任务

**顺口**：用户上传简历 → 系统自动切 chunk → Embedding 转向量 → 存 pgvector → 用户提问 → 检索最相关 chunk → 拼 Prompt → 大模型回答。保证回答有依据，不是胡编。

### 2.2 RAG 原理

> **提示**：先检索再生成，有据可查

**关键词**：Retrieval-Augmented Generation / 余弦相似度 / ANN(HNSW) / top-K 检索

**顺口**：先从向量库检索相关内容，再拼进 Prompt 给大模型。pgvector 用 HNSW 算法做近似最近邻搜索，百万数据毫秒级。

### 2.3 pgvector 选型

> **提示**：轻量 + 项目已有 PG + 无需额外运维

**顺口**：项目已有 PostgreSQL，启用扩展就行。Milvus 要单独部署，Chroma 不兼容 Java。数据量中等，pgvector 够用。

### 2.4 文本切分

> **提示**：粒度细 → 检索准 + 上下文不断

**顺口**：太长粒度粗，检索不准。500 字符切一个 chunk，邻 chunk 重叠 50 字符，保证上下文不被打断。

### 2.5 异步任务

> **提示**：上传慢 → Redis Stream → 接口立即返回

**顺口**：解析 + 向量化 + 生成报告都耗时。扔 Redis Stream 里异步跑，接口秒级返回任务 ID，用户轮询查进度。

### 2.6 项目难点

> **提示**：检索不准 → 调 chunk 和 topK；接口慢 → 异步化

**顺口**：难点一检索不准 → 改重叠切分 + topK 从 3 扩到 5。难点二接口慢 → Redis Stream 异步化。

## 三、项目二：教务管理系统

### 3.1 项目介绍（30 秒）

> **提示**：微服务架构 / 学生 + 课程 + 课时 / 我负责后端 + 缓存 + 分布式事务

**顺口**：微服务教务系统，学生管理、课程管理、课时管理。我负责后端接口、数据库设计、Redis 缓存、Seata 分布式事务。

### 3.2 为什么微服务

> **提示**：业务边界清晰 + 练习微服务全家桶

**顺口**：模块边界清晰，独立开发部署，降低耦合。主要还是想练习 Spring Cloud Alibaba 全家桶。

### 3.3 组件作用（背住）

| 组件 | 一句话职责 |
|---|---|
| Nacos | 注册中心 + 配置中心 |
| Gateway | 统一网关（路由 + 鉴权 + 限流） |
| OpenFeign | 声明式 HTTP 客户端 |
| Sentinel | 流量控制 + 熔断降级 |
| Seata | 分布式事务（跨服务一致性） |

### 3.4 分布式事务

> **提示**：Seata AT 模式 / 拍照 + 比对 + 逆向回滚 / @GlobalTransactional

**顺口**：学员报名 = 创建订单 + 扣课时，跨两个服务。Seata AT 模式执行 SQL 前拍照，执行后比对，有问题自动逆向回滚。方法上加 @GlobalTransactional 注解即可。

### 3.5 接口慢怎么办

> **提示**：Arthas 定位 → 索引 + 并行 Feign + 缓存热点

**顺口**：三板斧：高频字段加索引；多个 Feign 调用 CompletableFuture 并行；热点数据进 Redis 缓存。实测 800ms → 300ms。

### 3.6 项目难点

> **提示**：Seata 配置调优 / 全局锁超时

**顺口**：Seata 全局锁超时最难调。查 UNDO_LOG、调整 seata.grouplist 和事务超时时间才解决。

## 五、必背口诀

### 八股速记

```
ArrayList：查询快（数组下标），插入慢（扩容拷贝）
LinkedList：插入快（节点改指针），查询慢（遍历）
HashMap：数组+链表，查询O(1)，红黑树防退化
ConcurrentHashMap：CAS+sychronized分段锁
synchronized：悲观锁，适合低并发
ReentrantLock：可重入，可公平，可打断
线程池：核心线程满了→队列→最大线程→拒绝策略
Redis：String存值，Hash存对象，List做队列，Set去重，ZSet做排行榜
```

### MySQL 索引

```
主键自动建唯一索引
高频查询字段建索引（WHERE/LIKE/ORDER BY）
覆盖索引减少回表
联合索引最左前缀匹配
避免在索引列上做函数/运算
```

### 事务隔离级别

```
读未提交：脏读
读已提交：解决脏读（Oracle默认）
可重复读：解决不可重复读（MySQL默认）
串行化：解决幻读（性能差）
MVCC实现快照读，读写不阻塞
```

### Redis 三大问题

```
雪崩：大量key同时过期 → 过期时间+随机偏移
击穿：热点key过期 → 互斥锁/逻辑过期
穿透：查不存在数据 → 空值缓存/布隆过滤器
```

### CAP 定理

```
C：一致性，所有节点同一时刻数据相同
A：可用性，每次请求都有响应
P：分区容错，网络分区时系统仍能运行
CA：传统数据库（MySQL/PG）
CP：ZooKeeper/Etcd/HBase
AP：Redis/Eureka/Cassandra
```
