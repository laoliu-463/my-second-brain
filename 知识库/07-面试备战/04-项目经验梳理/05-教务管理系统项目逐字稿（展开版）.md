# 教务管理系统项目逐字稿（展开版）

## 概述

本页聚焦第二个项目“教务管理系统”的面试展开版答法。每个问题都按“面试官问法 -> 顺口回答 -> 技术亮点与追问扩展”组织，适合单独强化教务系统这条项目线，也能和 [[02-两项目完整问答模板]]、[[03-脱稿速记卡片]] 组成一套从长答到速记的备考链路。

## 详细内容

### 1. 项目整体

**面试官问法（展开版）**

> 请介绍一下你的教务管理系统，做了哪些功能，你负责什么？

**顺口回答（口语化，可直接背）**

我的教务管理系统是一个微服务架构项目，主要面向教培场景，功能包括学生管理、课程管理和消课管理等模块。

我主要负责后端接口对接和调试，同时参与数据库表设计和微服务组件集成，还做了 Redis 缓存和接口性能优化。

技术栈包括 Spring Boot、MyBatis/MyBatis-Plus，以及 Spring Cloud Alibaba 生态里的 Nacos、Gateway、OpenFeign、Sentinel 和 Seata。

这个项目的设计目标，一方面是提高教务管理效率，另一方面也是想通过真实业务流程练习微服务、缓存和分布式事务的实战能力。

**技术亮点 / 项目场景说明**

- 微服务架构落地
- Redis 缓存优化接口性能
- 分布式事务保证多模块操作一致性

**追问扩展**

- 为什么用微服务，不直接做单体？
  - 回答重点：模块边界清晰，可独立部署、扩展，也能练习微服务治理能力。

### 2. 微服务拆分逻辑

**面试官问法（展开版）**

> 微服务是如何拆分的？每个模块负责什么？

**顺口回答（口语化，可直接背）**

这个系统的模块边界比较清晰，所以我按业务职责做了拆分：

- 学生服务负责学生信息管理
- 课程服务负责课程信息管理
- 消课服务负责课时扣减和消课记录

这样拆分之后，每个服务都可以独立开发、部署和扩展，模块之间的耦合度会更低，后期维护也更清晰。

如果只是一个很小的项目，单体当然也能做；但我这里选择微服务，主要是为了练习 Spring Cloud Alibaba 这套组件的真实配合方式。

**技术亮点 / 项目场景说明**

- 微服务拆分逻辑
- 模块独立性和边界清晰
- 可扩展、易维护

**追问扩展**

- 每个服务怎么通信？
  - 用 OpenFeign 做声明式远程调用。
- 多服务一致性怎么保证？
  - 用 Seata 做分布式事务管理。

### 3. 微服务核心组件

**面试官问法（展开版）**

> 你用的微服务组件作用分别是什么？

**顺口回答（口语化，可直接背）**

这套项目里我主要用了五个核心组件：

- **Nacos**：服务注册与发现，加上配置中心。这样服务之间调用不依赖写死的 IP，配置也能集中管理。
- **Gateway**：统一网关，负责请求入口、路由转发、鉴权和限流。
- **OpenFeign**：声明式 HTTP 客户端，服务间调用时像调本地方法一样，开发效率更高。
- **Sentinel**：做流量控制和熔断降级，防止某个接口流量过高或者下游异常把整个系统拖垮。
- **Seata**：负责分布式事务，保证跨服务操作的一致性，比如报名课程时创建记录和扣减课时要么一起成功，要么一起回滚。

**技术亮点 / 项目场景说明**

- 微服务核心组件的职责分层
- 不只会背概念，也能结合场景落地解释
- Seata 体现分布式事务实践能力

**追问扩展**

- 为什么不直接用单体？
  - 回答重点：不是单体不行，而是这个项目更适合练习服务治理、远程调用和一致性问题。
- Sentinel 具体怎么限流？
  - 可答：按 QPS、线程数或者热点参数做规则控制，保护高频接口。

### 4. 接口性能优化

**面试官问法（展开版）**

> 系统接口慢怎么办？你做了什么优化？

**顺口回答（口语化，可直接背）**

项目里一开始有些接口比较慢，主要原因是多表查询和远程调用叠加。

我主要做了三个方向的优化：

1. SQL 优化，对高频查询字段补充必要索引
2. Redis 缓存高频数据，比如学生信息和课程信息
3. 对耗时操作做异步化，避免阻塞主线程

优化之后，关键接口的响应时间大概能控制在 300ms 左右，用户体验会稳定很多。

**技术亮点 / 项目场景说明**

- SQL 索引优化
- Redis 缓存热点数据
- 异步任务削峰和缩短主链路耗时

**追问扩展**

- 异步任务怎么保证安全性？
  - 可答：可配合 Redis Stream、任务状态记录和失败重试机制。
- SQL 优化具体怎么做？
  - 可答：先看执行计划，避免全表扫描，再针对查询条件和关联字段加索引。

### 5. 数据库事务与索引

**面试官问法（展开版）**

> 如何保证数据一致性？索引怎么用？

**顺口回答（口语化，可直接背）**

单服务内部的数据库操作，我用 Spring 声明式事务来保证一致性。

如果是跨服务的业务流程，比如报名课程同时要创建记录和扣减课时，我会用 Seata 分布式事务来保证要么都成功、要么都回滚，避免出现部分成功、部分失败的情况。

索引方面，我会对高频查询字段，比如学生 ID、课程 ID 这些条件列建立索引，减少全表扫描，提高查询效率。

**技术亮点 / 项目场景说明**

- Spring 本地事务 + Seata 分布式事务的分层使用
- 高并发查询场景下的索引意识
- 一致性和性能一起考虑

**追问扩展**

- 两个服务同时修改同一数据怎么办？
  - 可答：Seata AT 模式会配合全局锁和回滚日志保证一致性。
- 索引什么时候会失效？
  - 可答：函数处理、隐式类型转换、`like '%xxx'` 这类写法都可能导致索引失效。

### 6. 项目难点

**面试官问法（展开版）**

> 项目遇到的技术难点是什么？你怎么解决？

**顺口回答（口语化，可直接背）**

我觉得项目里有两个比较典型的难点。

第一个是多表查询加远程调用导致接口慢，这块我通过 SQL 优化、Redis 缓存和异步任务做了性能提升。

第二个是微服务组件集成，尤其是对 Nacos 注册发现和 Seata 分布式事务这类组件，刚开始理解和调试都比较花时间。后面通过自己搭环境、排调用链、查事务回滚过程，逐步把核心用法跑通了。

最后的结果是，性能、稳定性和一致性都比初始版本更好。

**技术亮点 / 项目场景说明**

- 性能优化能力
- 微服务组件落地与调试能力
- 分布式事务实战经验

### 7. 上线情况

**面试官问法（展开版）**

> 项目上线了吗？

**顺口回答（口语化，可直接背）**

这个项目主要是校内或者个人作品集项目，没有大规模真实用户上线。

但在开发过程中，像接口分层、异常处理、异步任务、微服务组件集成这些工程化内容都是完整落地的，所以从结构上看，它是可以继续往生产环境扩展的。

**技术亮点 / 项目场景说明**

- 工程化思维
- 可扩展性意识
- 能把“作品集项目”讲成“可演进系统”

### 8. 项目亮点总结

**面试官问法（展开版）**

> 这个项目有什么亮点或者独特之处？

**顺口回答（口语化，可直接背）**

我觉得这个项目的亮点主要有四个：

1. 用微服务架构把学生、课程、消课等模块拆开，结构更清晰，也更方便后续扩展
2. 用 Redis 缓存和异步任务优化接口性能，提升用户访问体验
3. 用 SQL 索引和 Seata 分布式事务同时兼顾性能和数据一致性
4. 整体可扩展性比较高，后面要继续加排课、考勤或者财务模块也比较自然
