## 零、入门导学：操作系统到底是什么

这一节的目标不是先背定义，而是先建立底层心智模型：

```text
操作系统到底是什么？
为什么需要操作系统？
Java 程序和操作系统是什么关系？
面试怎么回答？
```

### 1. 一句话理解

```text
操作系统是管理计算机硬件和软件资源的“总管”。
```

计算机里有 CPU、内存、硬盘、网卡、键盘、鼠标、显示器，也有进程、线程、文件等软件资源。这些资源不能让微信、浏览器、IDEA、Java 服务、MySQL、Redis 等程序随便抢。

所以操作系统的第一层作用就是：

```text
统一管理资源，统一分配资源，防止不同程序互相干扰。
```

### 2. 饭店类比

可以把计算机想象成一个大型饭店：

| 计算机资源 | 饭店类比 |
|---|---|
| CPU | 厨师 |
| 内存 | 操作台 |
| 硬盘 | 仓库 |
| 网卡 | 外卖窗口 |
| 键盘鼠标 | 点餐入口 |
| 显示器 | 出餐展示屏 |
| 应用程序 | 不同订单 |

如果没有饭店经理统一安排，所有订单都会抢厨师、抢操作台、抢仓库和外卖窗口。操作系统就像饭店经理，负责分配 CPU 时间、安排内存空间、管理磁盘文件、控制网络设备，并防止不同程序互相破坏。

### 3. 为什么必须有操作系统

应用程序不能直接随便操作硬件。比如 Java 读文件：

```java
FileInputStream fis = new FileInputStream("a.txt");
```

真实链路不是 Java 程序直接伸手去硬盘拿文件，而是：

```text
Java 程序
↓
JVM
↓
操作系统
↓
硬盘
```

再比如 Java 创建 Socket：

```java
Socket socket = new Socket("127.0.0.1", 8080);
```

真实链路是：

```text
Java 程序
↓
JVM
↓
操作系统网络模块
↓
网卡
```

所以操作系统是应用程序和硬件之间的中间层。它既保护硬件不被应用程序乱用，也给应用程序提供统一、稳定的访问入口。

### 4. 操作系统五个核心职责

面试先记这五个：

```text
1. 进程管理
2. 内存管理
3. 文件管理
4. 设备管理
5. 提供系统调用
```

| 职责 | 负责什么 | Java 关联点 |
|---|---|---|
| 进程管理 | 创建、销毁、调度进程和线程，分配 CPU 时间 | Spring Boot 服务运行后，本质是一个 JVM 进程 |
| 内存管理 | 分配、回收、隔离内存，管理虚拟内存 | JVM 堆、栈、方法区都来自 OS 分配给 JVM 的内存 |
| 文件管理 | 创建、删除、读取、写入文件，管理目录和权限 | `FileInputStream`、`FileOutputStream` 底层依赖 OS 文件系统 |
| 设备管理 | 管理硬盘、网卡、键盘、鼠标、显示器等设备 | Java 网络通信不直接控制网卡，而是走 OS 网络接口 |
| 系统调用 | 给应用程序提供访问底层资源的接口 | 读写文件、创建线程、收发网络数据都离不开系统调用 |

### 5. 操作系统和 Java 的关系

Java 程序不是直接跑在硬件上的：

```text
Java 代码
↓
字节码 .class
↓
JVM
↓
操作系统
↓
硬件
```

运行一个后端服务：

```bash
java -jar hospital-system.jar
```

真实含义是：

```text
操作系统启动了一个 java 进程
这个进程里运行 JVM
JVM 加载 class 字节码
JVM 创建 main 线程、GC 线程、业务线程
操作系统调度这些线程使用 CPU
```

因此 Java 后端面试问操作系统，本质是在问你是否理解：

```text
Java 线程为什么需要操作系统调度？
Java I/O 为什么和系统调用有关？
JVM 内存为什么离不开操作系统内存？
高并发为什么和线程、I/O 多路复用、内核通知机制有关？
```

### 6. 用户态和内核态先建立直觉

操作系统为了安全，把运行状态分成两层：

```text
用户态：普通应用程序运行的地方，权限低
内核态：操作系统内核运行的地方，权限高
```

Java 程序、浏览器、微信、IDEA 等大多数时候运行在用户态。操作系统内核运行在内核态，可以操作 CPU、内存、磁盘、网卡等底层资源。

类比饭店：

```text
普通服务员 = 用户态
饭店经理/后厨管理者 = 内核态
```

服务员不能随便改电路、开仓库、调厨师班表，需要向经理申请。应用程序也是一样，Java 程序想读文件、发网络包、创建线程，就要通过系统调用进入内核态，让操作系统帮它做。

### 7. 面试标准回答

> 什么是操作系统？

```text
操作系统是计算机系统中最基础的系统软件，主要负责管理计算机的硬件和软件资源，比如 CPU、内存、磁盘、文件、I/O 设备、进程和线程等。

它一方面向下管理硬件资源，另一方面向上为应用程序提供系统调用接口，让应用程序可以安全、统一地使用底层资源。

比如 Java 程序运行时，本质上是操作系统启动了一个 JVM 进程，Java 线程最终也需要由操作系统进行调度，Java 的文件读写和网络通信底层也都依赖操作系统提供的能力。
```

> 为什么需要操作系统？

```text
因为计算机中的 CPU、内存、磁盘、网卡等硬件资源需要被多个程序共同使用，如果没有操作系统统一管理，程序之间会互相抢占资源，导致系统混乱甚至崩溃。

操作系统负责资源分配、进程调度、内存保护、文件管理和设备管理，同时提供系统调用接口，让应用程序不用直接操作硬件，从而提高系统的安全性、稳定性和易用性。
```

> 操作系统和 Java 有什么关系？

```text
Java 程序运行时并不是直接运行在硬件上，而是运行在 JVM 上，JVM 又运行在操作系统之上。

一个 Java 后端服务启动后，本质上是一个 JVM 进程。JVM 中的 main 线程、GC 线程、业务线程池线程最终都要交给操作系统调度。Java 的文件 I/O、网络 I/O、线程创建、内存申请等操作，底层也都离不开操作系统提供的系统调用。
```

### 8. 小测试答案模板

> 为什么应用程序不能直接操作硬件？

因为硬件资源是所有程序共享的。如果每个程序都能直接操作 CPU、内存、硬盘、网卡，就会出现资源抢占、数据破坏和安全问题。所以操作系统要把硬件管起来，应用程序只能通过操作系统提供的接口申请资源。

> Java 后端服务启动后，为什么说它本质上是一个 JVM 进程？

因为执行 `java -jar app.jar` 时，操作系统实际启动的是一个 `java` 进程。这个进程内部运行 JVM，JVM 再加载 `.class` 字节码，并创建 main 线程、业务线程和 GC 线程等。因此从操作系统视角看，Spring Boot 服务本质就是一个 JVM 进程。

> Java 读文件、发网络请求，为什么最终和操作系统有关？

因为 `FileInputStream`、`Socket` 这类 Java API 只是上层封装。真正读磁盘、发网络包的是操作系统。Java 程序会通过 JVM 调用底层系统调用，请操作系统访问文件系统、网络协议栈和硬件设备。

### 9. 一句话记忆

```text
操作系统负责管硬件，JVM 负责跑 Java，Java 程序想用硬件资源，最终都要通过操作系统。
```

## 二、用户态和内核态

## 1. 为什么要有用户态和内核态？

上一节已经建立了一个底层前提：

```text
应用程序不能直接操作硬件。
```

原因是硬件和内核资源太关键，比如：

```text
内存
磁盘
网卡
CPU
进程
线程
```

如果普通程序可以随便操作这些资源，就可能出现：

```text
一个程序乱改别人的内存
一个程序随便删除磁盘文件
一个程序霸占 CPU 不放
一个程序乱发网络数据
```

系统会变得非常不安全，也很不稳定。所以操作系统把程序运行状态分成两层：

```text
用户态
内核态
```

---

## 2. 用户态是什么？

用户态就是普通应用程序运行的状态。

比如：

```text
Java 程序
浏览器
微信
IDEA
游戏
普通后端服务
```

这些程序大多数时间都运行在用户态。

用户态特点：

```text
权限低
不能直接访问硬件
不能直接操作内核数据
不能随意管理内存和设备
```

类比：

```text
用户态 = 饭店普通服务员
```

服务员可以接单、传话、服务顾客，但不能直接进总控室改电闸、调后厨设备、改账本。

Java 程序、浏览器、IDEA、MySQL 客户端，大多数时候也都运行在用户态。

---

## 3. 内核态是什么？

内核态是操作系统内核运行的状态。

内核态权限很高，可以操作：

```text
CPU
内存
磁盘
网卡
文件系统
进程线程
设备驱动
```

类比：

```text
内核态 = 饭店经理 / 后厨总控室
```

它有最高权限，可以调配厨师、开仓库、控制设备、处理重要资源。

---

## 4. 用户态程序想操作硬件怎么办？

通过：

```text
系统调用
```

系统调用就是：

```text
应用程序向操作系统申请使用底层资源的接口。
```

比如 Java 读文件：

```java
FileInputStream fis = new FileInputStream("a.txt");
```

表面看是 Java 代码，底层大致是：

```text
Java 代码
↓
JVM
↓
发起系统调用
↓
操作系统内核
↓
磁盘读取文件
↓
返回数据给 Java 程序
```

---

## 5. 系统调用是什么？

系统调用可以理解为：

```text
用户态程序进入内核态办事的“申请窗口”。
```

用户态程序不能直接操作硬件，但可以向操作系统发起请求：

```text
我要读文件
我要写文件
我要创建线程
我要申请内存
我要发送网络数据
```

操作系统检查权限和参数后，再帮它完成。

所以系统调用解决的是一个核心矛盾：

```text
应用程序权限不能太高，但应用程序又必须使用底层资源。
```

---

## 6. 常见系统调用场景

Java 后端里很多操作最终都会用到系统调用：

```text
读写文件
网络通信
创建线程
进程切换
申请内存
Socket 连接
数据库连接
获取系统时间
```

### 6.1 Java 读文件

```java
new FileInputStream("a.txt");
```

底层需要操作系统读取磁盘。

### 6.2 Java 发网络请求

```java
Socket socket = new Socket("127.0.0.1", 8080);
```

底层需要操作系统操作网卡和 TCP/IP 协议栈。

### 6.3 Java 创建线程

```java
new Thread(() -> {
    System.out.println("hello");
}).start();
```

Java 线程最终要映射到操作系统线程，由操作系统调度。

## 7. 用户态切换到内核态的成本

用户态和内核态切换不是免费的。

每次切换通常都需要：

```text
保存当前程序现场
切换 CPU 特权级/执行状态
执行内核代码
从内核态返回用户态
恢复用户程序现场
```

这个过程比普通函数调用更重，所以频繁系统调用会带来性能开销。

这和 Java 后端高并发直接相关。比如传统 BIO：

```text
一个连接一个线程
每个线程阻塞等待 I/O
频繁进行系统调用和线程切换
高并发时性能差
```

而 NIO / I/O 多路复用可以减少线程数量和不必要等待，提高高并发处理能力。

## 8. 用户态和内核态的区别

| 对比点 | 用户态 | 内核态 |
|---|---|---|
| 运行主体 | 普通应用程序 | 操作系统内核 |
| 权限 | 低 | 高 |
| 是否能直接访问硬件 | 不能 | 能 |
| 是否能访问内核数据 | 不能 | 能 |
| 出错影响 | 通常只影响当前程序 | 可能影响整个系统 |
| 典型场景 | Java 程序、浏览器 | 文件系统、网络协议栈、设备驱动 |

## 9. 面试标准回答：用户态和内核态区别

> 用户态和内核态有什么区别？

```text
用户态是普通应用程序运行的状态，权限较低，不能直接访问硬件资源和操作系统内核数据。

内核态是操作系统内核运行的状态，权限较高，可以直接管理 CPU、内存、磁盘、网卡、进程线程等底层资源。

区分用户态和内核态主要是为了系统安全和稳定。应用程序如果需要读写文件、访问网络、创建线程等，需要通过系统调用从用户态切换到内核态，由操作系统完成相关操作。
```

## 10. 面试标准回答：什么是系统调用？

> 什么是系统调用？

```text
系统调用是操作系统提供给应用程序使用底层资源的接口。普通应用程序运行在用户态，不能直接访问硬件资源，所以当程序需要进行文件读写、网络通信、创建线程、申请内存等操作时，需要通过系统调用进入内核态，由操作系统内核代为完成。

比如 Java 程序中的文件 I/O、Socket 网络通信、线程创建等，底层都依赖操作系统提供的系统调用。
```

## 11. 和 Java 八股怎么关联？

可以这样串：

```text
Java 程序运行在 JVM 上，JVM 本身是操作系统中的一个进程。

当 Java 程序只是做普通计算时，主要在用户态执行。

当 Java 程序需要读文件、写文件、发网络包、创建线程时，就需要通过 JVM 调用操作系统提供的系统调用，从用户态切换到内核态。

所以 Java 的 I/O、线程、网络通信，本质都离不开操作系统。
```

## 12. 最终记忆版

```text
用户态：普通程序运行的地方，权限低，不能直接操作硬件。
内核态：操作系统内核运行的地方，权限高，可以操作硬件和核心资源。
系统调用：用户态程序请求操作系统帮忙使用底层资源的接口。

Java 文件 I/O、网络 I/O、线程创建，最终都需要系统调用。
用户态和内核态切换有成本，所以高并发场景要尽量减少不必要的阻塞和频繁切换。
```

## 13. 小测试答案模板

> 为什么普通 Java 程序不能直接操作磁盘和网卡？

因为磁盘、网卡都是底层硬件资源，权限太高。如果普通 Java 程序可以直接操作，就可能乱读写文件、破坏数据、乱发网络包，甚至影响整个系统安全。所以 Java 程序不能直接碰硬件，只能通过 JVM 调用操作系统提供的接口，让操作系统去操作磁盘和网卡。

> 什么是系统调用？它解决了什么问题？

系统调用就是应用程序向操作系统申请使用底层资源的接口。它解决的问题是：应用程序权限低，不能直接操作硬件；但程序又确实需要读文件、写文件、发网络请求、创建线程。所以操作系统提供系统调用，让用户态程序可以安全地请求内核态帮忙完成这些操作。

> 为什么说用户态和内核态切换有性能成本？

因为从用户态切到内核态时，CPU 需要保存当前程序的执行现场，切换到内核权限，执行内核代码，处理完后再恢复现场回到用户态。这个过程比普通函数调用更重。如果频繁系统调用、频繁线程切换，高并发场景下开销就会变明显。因此 NIO、I/O 多路复用这类模型，本质上是在减少不必要的阻塞、线程数量和切换成本。

## 三、进程和线程

## 1. 一句话理解

```text
进程：正在运行的程序，是操作系统分配资源的基本单位。
线程：进程中的执行单元，是 CPU 调度的基本单位。
```

## 2. 什么是进程？

电脑里同时运行的程序，每一个都是独立的进程：

```text
微信       → 进程 A
浏览器     → 进程 B
IDEA       → 进程 C
MySQL      → 进程 D
Redis      → 进程 E
Java 后端   → 进程 F
```

启动一个 Spring Boot 项目：

```bash
java -jar hospital-system.jar
```

操作系统会创建一个：

```text
java 进程 / JVM 进程
```

这个进程拥有自己的：

```text
内存空间        ← 虚拟地址空间
线程            ← 里面真正执行代码的单元
文件句柄        ← 打开的文件
Socket 连接     ← 网络连接
运行状态        ← 就绪/运行/阻塞等
程序代码        ← 磁盘上的 jar 包被加载进来
```

所以：**进程不是代码文件本身**。jar 包只是磁盘上的文件，被运行起来后才变成进程。

## 3. 饭店类比

把进程想象成一家正在营业的饭店：

```text
店面空间  = 内存空间
厨师和服务员 = 线程（真正干活的人）
仓库记录  = 文件资源
外卖电话  = Socket 连接
营业规则  = 程序代码
```

两家饭店 = 两个进程。默认互相隔离：

```text
A 饭店不能随便拿 B 饭店仓库里的食材
A 饭店不能随便改 B 饭店的账本
```

这就像操作系统中**进程之间默认内存隔离**。

## 4. 什么是线程？

一家饭店不能只有店面，还需要有人干活：

```text
厨师做菜
服务员接待
收银员结账
外卖员打包
```

这些真正干活的人，就像线程。

```text
进程提供资源环境
线程负责真正执行任务
```

Java 里也一样。Spring Boot 服务启动后是一个 JVM 进程，里面有很多线程：

```text
JVM 进程
├── main 线程         ← 程序入口
├── GC 线程           ← 垃圾回收
├── JIT 编译线程      ← 热点代码编译
├── Tomcat 工作线程   ← 处理 HTTP 请求
├── 线程池线程        ← 异步任务执行
└── 日志线程          ← 异步写日志
```

真正执行代码的是**线程**，不是进程本身。

## 5. 进程和线程的关系

```text
进程 = 一个运行中的程序容器
线程 = 容器里真正干活的执行线

进程像饭店
线程像饭店里的员工

一家饭店至少有一个员工（线程）
一个进程至少有一个线程（main 线程）
```

## 6. 核心区别

| 对比点   | 进程              | 线程              |
| ------ | --------------- | --------------- |
| 本质    | 运行中的程序          | 进程内的执行单元        |
| 资源    | 拥有独立资源          | 共享进程资源          |
| 内存    | 进程间内存隔离         | 同进程线程共享内存       |
| 调度    | 资源分配基本单位        | CPU 调度基本单位      |
| 开销    | 创建、切换开销大        | 创建、切换开销小        |
| 通信    | 进程间通信复杂         | 线程间通信较方便        |
| 安全性   | 一个进程崩溃通常不影响其他进程 | 一个线程出问题可能影响整个进程 |

## 7. 为什么进程是资源分配的基本单位？

操作系统给资源时，以进程为单位分配：


操作系统给这个 JVM 进程分配：

```text
一块虚拟内存空间
文件描述符表

线程资源
运行权限
```

→ **进程是资源分配的基本单位**

类比：饭店拥有自己的店面、仓库、电话、账本。

## 8. 为什么线程是 CPU 调度的基本单位？

CPU 真正执行的是一条条指令，这些指令属于线程的执行流。

操作系统调度 CPU 时，是说：

```text
让线程 A 执行一会儿
再切换到线程 B 执行一会儿
```

不是说"让整个进程执行一下"。

→ **线程是 CPU 调度的基本单位**

类比：经理安排工作时安排哪个员工去干活，而不是让整家饭店去干活。

## 9. Java 程序和进程线程的关系

```text
运行一个 Java 程序 = 操作系统启动一个 JVM 进程
Java 里的 Thread = JVM 中的线程，最终映射到操作系统线程
Java 线程最终由操作系统调度执行
```

比如：

```java
public class Main {
    public static void main(String[] args) {
```

运行：


真实执行过程：

```text
操作系统启动 java 进程
    ↓
JVM 在这个进程中运行
    ↓
JVM 创建 main 线程
    ↓
main 线程执行 main 方法
    ↓
CPU 调度 main 线程执行字节码对应的机器指令
```

## 11. 面试标准回答

> 进程和线程有什么区别？

```text
进程是程序的一次运行实例，是操作系统分配资源的基本单位；线程是进程中的执行单元，是 CPU 调度的基本单位。

一个进程可以包含多个线程，同一进程内的线程共享进程的堆内存、方法区、文件句柄等资源，但每个线程有自己的程序计数器、虚拟机栈和本地方法栈。

进程之间默认相互隔离，通信成本较高；线程之间共享同一进程资源，通信相对方便，但也更容易出现线程安全问题。

在 Java 中，一个运行中的 Java 程序通常对应一个 JVM 进程，而 Java 中创建的线程最终会映射到操作系统线程，由操作系统调度执行。
```

> Java 程序是进程还是线程？

```text
一个 Java 程序运行起来后，本质上是操作系统中的一个 JVM 进程。JVM 进程内部会包含多个线程，比如 main 线程、GC 线程、JIT 编译线程以及业务线程池线程。

真正执行 Java 代码的是线程，比如 main 方法由 main 线程执行，Web 请求通常由 Tomcat 工作线程执行。
```

> 为什么线程切换比进程切换开销小？

```text
进程之间拥有独立的地址空间和资源，进程切换时需要切换更多上下文信息，比如地址空间、页表、文件资源等，所以开销较大。

线程属于同一个进程时，共享进程的地址空间和资源，线程切换主要切换线程自己的上下文，比如程序计数器、栈等，因此开销相对较小。
```

## 12. 追问：线程共享什么？独占什么？（Java 面试高频）

同一 JVM 进程内，线程**共享**：

```text
堆（Heap）           ← 所有线程共享，new 的对象放这里
方法区（Method Area） ← 类信息、常量、静态变量
运行时常量池
类信息
静态变量
文件句柄
Socket 连接等进程资源
```

每个线程**独有**：

```text
程序计数器（PC）     ← 当前执行到哪一行字节码
虚拟机栈            ← 方法调用栈帧
本地方法栈           ← native 方法调用栈帧
```

记忆口诀：

```text
对象和类信息通常共享；
方法调用栈和执行位置每个线程自己一份。
```

---

## 13. 为什么线程共享资源会有安全问题？

多个线程同时操作同一份数据：

```java
count++;
```

实际是三步：

```text
读取 count（线程 A 读到 0，线程 B 也读到 0）
加 1（线程 A 算 1，线程 B 算 1）
写回 count（线程 A 写 1，线程 B 也写 1）
```

结果：count 应该是 2，实际是 1 → **更新丢失**

这就是线程安全问题。Java 解决方案：

```text
synchronized       ← 互斥锁
ReentrantLock     ← 可重入锁
volatile           ← 可见性
AtomicInteger     ← 原子操作
ConcurrentHashMap ← 并发容器
```

## 四、线程上下文切换

## 1. 一句话理解

```text
上下文切换：CPU 从执行一个线程，切换去执行另一个线程时，
需要保存当前线程状态，并恢复另一个线程状态的过程。
```

简单说：

```text
CPU 换人干活，需要先保存上一个人的工作现场，
再恢复下一个人的工作现场。
```

---

## 2. 饭店类比

厨师正在做菜 A：

```text
锅里炒着菜
旁边放着调料
菜单看到第 3 步
火候调到中火
```

突然经理说"先别做 A 了，去做菜 B"。

厨师不能直接跑，要先记录：

```text
菜 A 做到哪一步了
用了哪些调料
锅里状态怎么样
下一步该做什么
```

然后再去做菜 B。

过一会儿切回来做 A 时，还要恢复现场。

这就是上下文切换。

---

## 3. 上下文包括什么？

线程的上下文就是"执行现场"，主要包括：

```text
程序计数器       ← 执行到哪一行字节码了
寄存器状态       ← CPU 寄存器里的临时数据
栈信息           ← 方法调用栈帧
线程状态         ← 就绪 / 运行 / 阻塞
CPU 现场         ← 各种硬件寄存器值
```

先记住：

```text
上下文 = 线程的执行现场
```

---

## 4. 为什么会发生上下文切换？

常见原因：

```text
1. CPU 时间片用完
2. 线程主动让出 CPU（yield）
3. 线程阻塞（I/O 等待、sleep）
4. 更高优先级线程需要执行
5. 线程等待 I/O 完成
6. 线程等待锁（synchronized 拿不到锁）
```

### 时间片用完

操作系统给每个线程分配一小段时间片，比如：

```text
线程 A 执行 10ms → 时间片用完 → 切换
线程 B 执行 10ms → 时间片用完 → 切换
线程 C 执行 10ms → 时间片用完 → 切换
```

### 线程阻塞

Java 线程访问数据库：

```text
线程 A 发起 SQL 查询
→ 等待数据库返回（阻塞）
→ 操作系统切换去执行其他线程
```

### 等待锁

```java
synchronized (lock) {
    // 临界区
}
```

线程 A 拿着锁，线程 B 想进来但拿不到 → B 阻塞 → 触发线程调度切换。

---

## 5. 为什么上下文切换有成本？

CPU 切换线程时不是"瞬间免费切换"，要做很多事：

```text
保存当前线程现场
加载新线程现场
切换内核数据结构
可能导致 CPU 缓存失效（TLB、L1/L2/L3 缓存被冲掉）
```

这段时间 CPU **没有真正处理业务逻辑**。

所以线程太多时可能出现：

```text
CPU 很忙（一直在切换）
但业务处理并不快
大量时间浪费在线程切换上
```

这就是著名的：
```text
Amdahl 定律的延伸：
并行化收益被上下文切换成本抵消
```

---

## 6. 为什么线程不是越多越好？

以 8 核 CPU 服务器为例：

```text
开 1000 个线程
```

真正同时运行的线程最多也就 8 个（每核一个）。

其他 992 个都在等。

CPU 不停轮换：

```text
线程 1 → 线程 2 → 线程 3 → ... → 线程 1000 → 线程 1 → ...
```

结果：

```text
上下文切换成本变高
内存占用增加
锁竞争变严重
系统吞吐量下降
```

**所以：线程不是越多越好。**

---

## 7. 和 Java 线程池的关系

线程池的核心目的之一就是：

```text
控制线程数量，避免无限创建线程。
```

```java
ThreadPoolExecutor executor = new ThreadPoolExecutor(
    10,          // 核心线程数
    20,          // 最大线程数
    60, TimeUnit.SECONDS,
    new LinkedBlockingQueue<>(100)
);
```

含义：

```text
不要来一个任务就创建一个新线程
用有限数量的线程复用处理任务
```

线程池减少的问题：

```text
线程频繁创建销毁开销
线程数量失控
上下文切换过多
系统资源耗尽
```

---

## 8. 线程数怎么设置？

### CPU 密集型任务

```text
大量计算：加密解密、压缩解压、图像处理、复杂算法
```

主要吃 CPU，线程太多没用。

```text
线程数 = CPU 核心数 + 1
```

### I/O 密集型任务

```text
数据库查询、Redis 查询、网络请求、文件读写、第三方接口调用
```

经常等待 I/O，一个线程等 I/O 时 CPU 可以去执行其他线程。

```text
线程数 = CPU 核心数 * 2
```

更精确的公式：

```text
线程数 = CPU核心数 * (1 + 等待时间 / 计算时间)
```

### 记忆口诀

```text
CPU 密集型：线程数接近 CPU 核心数
I/O 密集型：线程数可以适当多一些
```

## 9. 面试标准回答

> 什么是上下文切换？

```text
上下文切换是指 CPU 从执行一个线程切换到执行另一个线程时，
需要保存当前线程的执行现场（程序计数器、寄存器、栈等），
并恢复另一个线程的执行现场的过程。

上下文切换本身会消耗 CPU 资源，如果线程数量过多，
CPU 可能频繁在线程之间切换，导致真正用于业务执行的时间减少，
从而影响系统性能。
```

> 为什么线程不是越多越好？

```text
线程并不是越多越好。因为 CPU 核心数量有限，
真正能同时执行的线程数量有限。

如果创建过多线程，会带来线程栈内存占用增加、
上下文切换成本上升、锁竞争加剧等问题。
大量线程频繁切换时，CPU 时间可能被浪费在线程调度上，
反而降低系统吞吐量。

所以 Java 后端通常使用线程池控制线程数量，
CPU 密集型任务线程数接近 CPU 核心数，
I/O 密集型任务可以适当增加线程数。
```

## 10. 和项目怎么结合

结合后端项目可以这样说：

```text
在项目中，简历解析、AI 评分、报告生成这类任务，
如果每来一个请求就创建一个新线程，
会导致线程数量不可控，容易造成上下文切换过多和内存占用升高。

所以更合理的做法是使用线程池或消息队列异步处理，
通过核心线程数、最大线程数和队列长度限制并发量，
避免瞬时高并发把服务资源打满。
```

## 11. 小测试答案模板

> 什么是线程上下文切换？

线程上下文切换就是 CPU 从执行一个线程切换到执行另一个线程时，先保存当前线程的执行现场，再恢复另一个线程的执行现场。

这个现场包括线程执行到哪里了、寄存器状态、程序计数器、栈信息等。简单说就是：CPU 换线程干活前，要先记住上一个线程干到哪了，再接着干下一个线程。

> 为什么线程太多反而会让程序变慢？

因为 CPU 核心数有限，真正能同时执行的线程也有限。如果线程特别多，大量线程就会争抢 CPU，操作系统需要频繁在线程之间切换。

每次切换都要保存和恢复现场，还可能带来缓存失效、锁竞争加剧、线程栈内存占用增加等问题。结果就是 CPU 看起来很忙，但很多时间花在线程调度和切换上，真正执行 Java 业务代码的时间反而减少。

> 为什么 Java 后端常用线程池，而不是每来一个任务就 new Thread？

因为每来一个任务就 `new Thread`，线程数量会失控。线程创建和销毁本身有开销，线程太多还会增加内存占用、上下文切换和锁竞争，严重时可能把服务资源打满。

线程池的作用就是控制线程数量、复用已有线程、用队列缓冲任务，让并发量可控。核心线程数、最大线程数、队列长度和拒绝策略都可以限制系统压力，所以 Java 后端通常用线程池处理异步任务和高并发请求。

## 五、进程 / 线程的状态与调度

这一节解决几个非常自然的问题：

```text
线程为什么会从运行变成等待？
什么是阻塞？
什么是就绪？
操作系统是怎么安排线程执行的？
Java 里的线程状态和操作系统线程状态有什么关系？
```

---

## 1. CPU 不可能同时执行所有线程

一个 Java 后端服务里可能有很多线程：

```text
Tomcat 请求线程
业务线程池线程
GC 线程
JIT 编译线程
日志线程
定时任务线程
```

但是 CPU 核心数量有限。比如 8 核 CPU，同一时刻真正能并行执行的线程数量也是有限的。其他线程要么等 CPU，要么等锁，要么等 I/O。

所以操作系统必须做一件事：

```text
决定哪个线程先执行，哪个线程后执行，哪个线程暂时等待。
```

这个过程叫：

```text
线程调度
```

---

## 2. 先用饭店类比理解线程状态

把线程想象成饭店员工，一个员工可能处于这些状态：

```text
刚入职，还没开始干活
已经准备好，等经理安排
正在干活
因为等食材，暂时干不了
活干完了，下班
```

对应到线程：

```text
新建
就绪
运行
阻塞 / 等待
终止
```

类比里，经理就是操作系统调度器，厨师就是 CPU，员工就是线程。

---

## 3. 操作系统层面的常见线程状态

简化理解，操作系统里的线程大致可以分成五类：

| 状态 | 含义 | 饭店类比 |
|---|---|---|
| 新建状态 | 线程刚被创建，还没真正开始执行 | 新员工刚登记 |
| 就绪状态 | 已经准备好，可以被 CPU 执行 | 穿好工服，等经理安排 |
| 运行状态 | 拿到 CPU 时间片，正在执行代码 | 正在后厨干活 |
| 阻塞状态 | 暂时不能继续执行，等锁、等 I/O、等事件 | 等食材、等工具、等别人交接 |
| 终止状态 | 执行完毕或异常退出 | 活干完下班 |

核心区别：

```text
就绪：线程可以运行，只是还没轮到 CPU。
运行：线程正在 CPU 上执行。
阻塞：线程现在没条件继续执行，给它 CPU 也没用。
```

---

## 4. 状态流转怎么发生

可以先记这条简化链路：

```text
新建
 ↓ start / 创建内核线程
就绪
 ↓ 被调度器选中
运行
 ↓ 时间片用完 / yield / 被抢占
就绪

运行
 ↓ 等锁 / 等 I/O / sleep / wait / join
阻塞或等待
 ↓ 条件满足 / 被唤醒 / 时间到
就绪

运行
 ↓ run 方法执行完或异常退出
终止
```

重点是：

```text
阻塞状态恢复后通常不是立刻运行，而是先回到就绪队列，再等待 CPU 调度。
```

线程睡醒、I/O 返回、锁拿到了，只表示它重新具备运行条件，不代表马上拿到 CPU。

---

## 5. Java 线程状态

Java 中 `Thread.State` 有 6 种：

```text
NEW
RUNNABLE
BLOCKED
WAITING
TIMED_WAITING
TERMINATED
```

| Java 状态 | 含义 | 常见触发 |
|---|---|---|
| NEW | 创建了线程对象，但还没调用 `start()` | `new Thread()` |
| RUNNABLE | 可运行，可能正在运行，也可能在等 CPU | `start()` 后未进入 Java 等待态 |
| BLOCKED | 等待进入 `synchronized` 保护的临界区 | 等 monitor 锁 |
| WAITING | 无限期等待，需要其他线程唤醒 | `wait()`、`join()`、`LockSupport.park()` |
| TIMED_WAITING | 带超时时间的等待 | `sleep()`、`wait(timeout)`、`join(timeout)` |
| TERMINATED | `run()` 执行结束或异常退出 | 线程生命周期结束 |

---

## 6. Java 的 RUNNABLE 容易误解

Java 的 `RUNNABLE` 不严格区分操作系统里的“就绪”和“运行”。

也就是说：

```text
线程已经启动，具备运行条件，正在等 CPU
线程已经拿到 CPU，正在执行代码
```

这两种在 Java 层面通常都可能显示为：

```text
RUNNABLE
```

所以面试里要特别注意：

```text
Java RUNNABLE = OS 就绪 + OS 运行 的合并视角
```

另外，Java 线程转储里看到 `RUNNABLE`，也不一定代表线程正在疯狂消耗 CPU。某些阻塞式 native I/O 场景可能也会显示为 `RUNNABLE`，需要结合线程栈、CPU 占用和 I/O 状态一起判断。

---

## 7. BLOCKED、WAITING、TIMED_WAITING 的区别

### BLOCKED

`BLOCKED` 通常表示线程在等 `synchronized` 的 monitor 锁。

```java
synchronized (lock) {
    // 临界区
}
```

如果线程 A 已经拿到 `lock`，线程 B 也想进入同一个 `synchronized` 代码块，线程 B 就会进入 `BLOCKED`。

注意这个细节：

```text
Java Thread.State.BLOCKED 专指等待 synchronized 锁。
它不等于中文里所有“线程被阻塞”的泛称。
```

比如等待数据库、等待网络返回，中文可以说“请求线程阻塞了”，但在 Java 线程状态里不一定显示为 `BLOCKED`。

### WAITING

`WAITING` 表示无限期等待，需要其他线程唤醒。

常见方法：

```java
obj.wait();
thread.join();
LockSupport.park();
```

它表达的是：

```text
我先不争 CPU 了，等别人通知我。
```

### TIMED_WAITING

`TIMED_WAITING` 表示带时间限制的等待。

常见方法：

```java
Thread.sleep(1000);
obj.wait(1000);
thread.join(1000);
```

它表达的是：

```text
我等一段时间，时间到了再回到可运行状态，继续竞争 CPU。
```

## 9. start() 和 run() 的区别

这是 Java 面试高频。

### 调用 start()

```java
Thread t = new Thread(() -> {
    System.out.println(Thread.currentThread().getName());
});

t.start();
```

`start()` 会真正启动一个新线程。这个线程进入可运行状态，等待操作系统调度，最终执行 `run()` 方法。

### 直接调用 run()

```java
t.run();
```

这只是普通方法调用，不会创建新线程，也不会交给操作系统调度一个新的执行流。

一句话：

```text
start() 是启动新线程。
run() 是在当前线程里普通调用一个方法。
```

饭店类比：

```text
start 是叫新员工开始干活。
run 是老板自己照着员工工作手册干一遍。
```

## 10. 线程调度和时间片

线程调度就是：

```text
操作系统决定哪个线程获得 CPU 执行权。
```

常见影响因素：

```text
线程状态
CPU 时间片
线程优先级
是否阻塞
系统调度策略
负载情况
```

时间片就是：

```text
操作系统分给某个线程连续执行的一小段 CPU 时间。
```

时间片到了，操作系统可能切换到另一个线程。这也是为什么单核 CPU 也能“同时”运行多个程序：

```text
不是真的同一时刻都在执行，
而是切换得足够快，让人感觉它们在同时运行。
```

Java 可以设置线程优先级：

```java
thread.setPriority(Thread.MAX_PRIORITY);
```

但线程优先级只是建议：

```text
最终是否优先执行，还是由操作系统调度器决定。
```

## 11. 阻塞为什么会影响高并发

假设一个请求线程正在查数据库：

```text
线程 A 处理请求
↓
发 SQL 到数据库
↓
等待数据库返回
```

等待期间，线程 A 通常不再消耗大量 CPU 计算，但它仍然占着线程资源：

```text
线程栈内存
Tomcat 工作线程名额
线程池 worker 名额
数据库连接或网络连接
请求上下文
```

如果很多请求都在等待数据库：

```text
大量 Tomcat 线程阻塞
线程池被占满
数据库连接池被占满
新请求排队或被拒绝
接口变慢甚至超时
```

所以高并发系统要关注：

```text
I/O 耗时
数据库响应时间
连接池大小
线程池大小
异步处理
超时时间
降级和限流
```

阻塞本身不一定消耗 CPU，但它会占用并发处理名额。这就是“线程被卡住”对吞吐量的伤害。

| 场景 | 可能状态 | 说明 |
|---|---|---|
| 正在执行 Controller / Service / Mapper 普通代码 | RUNNABLE | 可能正在运行，也可能在等 CPU |
| 多线程竞争 `synchronized` | BLOCKED | 等 monitor 锁 |

| `run()` 方法执行结束 | TERMINATED | 线程生命周期结束 |

```text
RUNNABLE 表示线程已经可以运行，可能正在运行，也可能在等待 CPU 调度；

需要注意的是，Java 的 RUNNABLE 合并了操作系统层面的就绪和运行状态，Java 的 BLOCKED 也专指等待 synchronized 锁，不等于所有 I/O 等待场景。
```

## 14. 面试标准回答：阻塞和等待有什么区别

可以这样答：

```text
在 Java Thread.State 里，BLOCKED 通常表示线程在等待获取 synchronized 锁，是锁竞争导致的阻塞。

WAITING 表示线程主动进入无限期等待状态，比如调用 wait、join、park，需要其他线程唤醒。

TIMED_WAITING 是带超时时间的等待，比如 sleep、wait(timeout)、join(timeout)，时间到了之后会重新回到可运行状态。

从更泛化的操作系统和后端语境看，阻塞还可以指等待 I/O、等待网络、等待数据库返回等场景，但这些不一定对应 Java Thread.State.BLOCKED。
```

## 16. 小测试答案模板

> `start()` 和 `run()` 的区别是什么？

如果大量 Tomcat 线程或业务线程池线程都被卡住，新请求就没有线程处理，只能排队、超时或被拒绝，最终表现为吞吐下降、响应变慢、接口超时。

## 六、并发和并行

## 1. 并发

```text
并发是多个任务在同一时间段内交替执行。
```

比如单核 CPU：

```text
一会儿执行任务 A
一会儿执行任务 B
切换得很快
看起来像同时执行
```

---

## 2. 并行

```text
并行是多个任务在同一时刻真正同时执行。
```

比如多核 CPU：

```text
核心1执行任务A
核心2执行任务B
```

---

## 面试标准回答

> 并发和并行有什么区别？

```text
并发指多个任务在同一时间段内交替执行，强调任务调度能力；并行指多个任务在同一时刻真正同时执行，通常需要多核 CPU 支持。

简单说，并发是看起来同时执行，并行是真正同时执行。
```

可以用于不同机器之间的进程通信。


## 八、线程同步和锁

多个线程同时执行，可能导致数据丢失。


## 面试标准回答

> 为什么多线程需要同步？

```text
因为同一进程内的多个线程会共享堆内存、方法区、文件句柄等资源。如果多个线程同时修改同一份共享数据，就可能产生数据不一致问题。

```

## 九、死锁

## 1. 什么是死锁？

死锁就是多个线程或进程互相等待对方释放资源，导致都无法继续执行。

类比：

```text
你拿着钥匙 A，等别人给你钥匙 B；
别人拿着钥匙 B，等你给他钥匙 A；
两个人都不放手，谁也走不了。
```

---

## 2. 死锁四个必要条件

必须背：

```text
互斥条件
请求并保持条件
不可剥夺条件
循环等待条件
```

解释：

| 条件    | 含义             |
| ----- | -------------- |
| 互斥    | 资源一次只能被一个线程占用  |
| 请求并保持 | 已经拿着资源，还请求其他资源 |
| 不可剥夺  | 别人不能强行抢走资源     |
| 循环等待  | 多个线程形成首尾等待环    |

---

## 3. 如何避免死锁？

```text
破坏四个必要条件之一
```

常见做法：

```text
固定加锁顺序
设置锁超时时间
避免嵌套锁
一次性申请所有资源
使用 tryLock
```

---

## 面试标准回答

> 什么是死锁？如何避免？

```text
死锁是指多个线程或进程在执行过程中，因为争夺资源而相互等待，导致都无法继续执行。

死锁产生需要满足四个必要条件：互斥、请求并保持、不可剥夺、循环等待。

避免死锁可以从破坏这些条件入手，比如固定加锁顺序，避免循环等待；使用 tryLock 设置超时时间，避免无限等待；或者尽量避免嵌套锁，一次性申请所需资源。
```

## 十、内存管理

操作系统需要管理内存：

```text

实现虚拟内存
```

---

## 1. 为什么每个进程有独立地址空间？

为了安全和稳定。

一个进程不能随便访问另一个进程的内存。

比如浏览器崩了，不应该直接把 MySQL 进程搞崩。

---

## 2. 物理地址和虚拟地址

```text
物理地址：真实内存条上的地址
虚拟地址：进程看到的地址
```

程序看到的是虚拟地址，操作系统和硬件 MMU 会把虚拟地址转换为物理地址。

---

## 面试标准回答

> 为什么操作系统要引入虚拟地址？

```text
虚拟地址让每个进程都拥有独立的地址空间，进程之间互不干扰，提高了系统安全性和稳定性。同时，虚拟地址也简化了程序的内存使用，让程序感觉自己拥有连续完整的内存空间，底层由操作系统负责映射到实际物理内存。
```

## 十一、虚拟内存

## 1. 什么是虚拟内存？

```text
虚拟内存是操作系统提供的一种内存管理机制，让程序感觉自己拥有比实际物理内存更大的连续内存空间。
```

它通过：

```text
内存 + 磁盘
```

一起配合。

当物理内存不够时，可以把暂时不用的数据换到磁盘。

---

## 2. 为什么需要虚拟内存？

```text
扩大可用内存
隔离进程地址空间
提高内存利用率
让程序使用连续地址空间
```

---

## 3. 缺页中断

如果程序访问的页不在物理内存中，会触发缺页中断。

操作系统会：

```text
暂停当前程序
从磁盘加载对应页面到内存
更新页表
继续执行程序
```

---

## 面试标准回答

> 什么是虚拟内存？

```text
虚拟内存是操作系统提供的一种内存管理机制，它为每个进程提供独立、连续的虚拟地址空间。程序访问的是虚拟地址，操作系统通过页表和 MMU 将虚拟地址映射到物理地址。

当物理内存不足时，操作系统可以把暂时不用的页面换出到磁盘，需要时再换入内存，从而让程序感觉可用内存比实际物理内存更大。
```

优点：


```text

```

优点：

```text

```

```text

```

---

## 面试标准回答

> 分页和分段有什么区别？

```text
分页是按照固定大小把内存划分为页，主要是为了方便内存管理和虚拟内存映射；分段是按照程序的逻辑结构划分，比如代码段、数据段、堆段和栈段。

分页对程序员透明，页大小固定，能减少外部碎片；分段更符合程序逻辑，但段大小不固定，容易产生外部碎片。
```

## 十三、页面置换算法

## 1. FIFO

最近最少使用。

## 面试标准回答

> LRU 是什么？

```text

```

## 十五、I/O 模型

这个非常重要，Java 后端高频。

常见 I/O 模型：

```text
阻塞 I/O
非阻塞 I/O
I/O 多路复用
信号驱动 I/O
异步 I/O
```

重点掌握前三个。

---

## 1. 阻塞 I/O BIO

线程发起 I/O 后，如果数据没准备好，就一直等。

类比：

```text
你去窗口取餐，餐没好，你一直站着等。
```

缺点：

```text
一个连接一个线程，连接多时线程资源压力大。
```

Java 传统 Socket 就是 BIO 思想。

---

## 2. 非阻塞 I/O NIO

线程发起请求后，如果数据没准备好，不会一直等，而是立即返回。

类比：

```text
你去窗口问餐好了没，没好就先走，过会儿再来问。
```

缺点：

```text
需要不断轮询，浪费 CPU。
```

---

## 3. I/O 多路复用

一个线程可以同时监听多个连接。

类比：

```text
餐厅给你一个取餐叫号器。
你不用一直问，叫号器响了你再去拿。
```

Linux 中常见：

```text
select
poll
epoll
```

Java NIO、Netty、Redis、Nginx 都和 I/O 多路复用有关。

---

## 面试标准回答

> 什么是 I/O 多路复用？

```text
I/O 多路复用是指一个线程可以同时监听多个 I/O 事件，当某个连接上的数据准备好后，操作系统通知应用程序进行处理。这样可以避免一个连接对应一个线程，减少线程资源消耗，提高高并发场景下的处理能力。

Linux 下常见的 I/O 多路复用机制有 select、poll 和 epoll。Java NIO、Netty、Redis、Nginx 等都大量使用了这种思想。
```

---

## 十六、select、poll、epoll

## 1. select

缺点：

```text
文件描述符数量有限（默认 1024）
每次调用都要遍历所有 fd
需要频繁拷贝 fd 集合到内核
```

---

## 2. poll

相比 select：

```text
没有固定 fd 数量限制（用链表）
```

但还是要遍历所有 fd。

---

## 3. epoll

优势：

```text
不用每次遍历全部 fd
通过事件通知机制，只返回就绪的 fd
高效处理高并发连接
```

三个核心系统调用：

```text
epoll_create()  创建 epoll 实例
epoll_ctl()    注册/注销要监听的文件描述符
epoll_wait()   等待事件就绪
```

---

## 面试标准回答

> select、poll、epoll 区别？

```text
select、poll、epoll 都是 Linux 下的 I/O 多路复用机制。

select 使用固定大小的 fd 集合，文件描述符数量有限（默认 1024），并且每次调用都需要把 fd 集合从用户态拷贝到内核态，再遍历全部 fd 判断就绪状态。

poll 用链表结构解决了 fd 数量限制问题，但仍然需要遍历全部 fd，拷贝开销依然存在。

epoll 通过红黑树管理要监听的 fd，每次注册新的 fd 只需调用 epoll_ctl 追加到红黑树，不需要重复拷贝；调用 epoll_wait 时只返回就绪的 fd，避免遍历全部连接，因此更适合高并发场景。
```

---

## 十七、epoll 的 LT 和 ET 模式（重点补充）

## 1. 两种触发模式

### Level Triggered 水平触发（默认）

```text
只要条件满足，就一直通知。
```

类比：

```text
你问服务员餐好了没，餐好了你就一直站在窗口等，
只要你不去拿，服务员就一直喊你。
```

只要 socket 接收缓冲区有数据，就会一直返回就绪。

---

### Edge Triggered 边沿触发

```text
状态变化时才通知一次。
```

类比：

```text
服务员只喊你一次，餐好了的那一瞬间叫你一声，
之后你不去拿，他不会再喊你。
```

只有 socket 缓冲区从"空"变为"非空"时，才通知一次。

---

## 2. 代码表现

epoll 默认是 LT 模式：

```c
// LT（水平触发）
epoll_ctl(epfd, EPOLL_CTL_ADD, fd, &ev);

// ET（边沿触发）
ev.events = EPOLLIN | EPOLLET;
epoll_ctl(epfd, EPOLL_CTL_ADD, fd, &ev);
```

---

## 3. 为什么 Netty 用 LT，RocketMQ 用 ET？

**Netty（LT）**：

```text
LT 模式编程更简单，不需要一次性把数据读完。
只要缓冲区有数据，epoll_wait 就会一直返回。
Netty 的 ByteBuf 有完整的读索引管理，配合 LT 很自然。
```

**RocketMQ（ET）**：

```text
ET 性能更高，减少无效通知。
但编程更复杂，必须一次性读完所有数据，否则会漏掉。
RocketMQ 用 ET 配合非阻塞 socket，手动循环读取直到真正无数据。
```

---

## 面试标准回答

> epoll 的 LT 和 ET 模式有什么区别？

```text
epoll 支持两种触发模式：LT 水平触发和 ET 边沿触发。

LT 模式下，只要文件描述符就绪，epoll_wait 就会一直返回通知，直到应用程序处理完。比如 socket 缓冲区有数据，就会持续通知，必须读走所有数据才会停。

ET 模式下，只在状态变化时通知一次。比如 socket 缓冲区从空变为非空时通知一次，缓冲区一直有数据也不会再通知，必须应用程序主动循环读取直到再次变空才会在下一次变化时收到通知。

LT 编程更简单，Netty 默认用 LT；ET 性能更高但编程更复杂，RocketMQ 等追求高性能的中间件会选择 ET。ET 模式下如果没有一次性读完数据，很容易漏掉后续到达的数据。
```

---

## 十八、同步、异步、阻塞、非阻塞

这几个词非常容易混。

## 1. 阻塞 / 非阻塞

关注的是：

```text
调用方等待结果时，是否会被卡住
```

阻塞：

```text
没结果就一直等
```

非阻塞：

```text
没结果马上返回
```

---

## 2. 同步 / 异步

关注的是：

```text
结果由谁来通知
```

同步：

```text
调用方自己主动等结果、查结果
```

异步：

```text
结果好了之后，别人通知调用方
```

---

## 组合关系

```text
同步阻塞：BIO，调用方一直等
同步非阻塞：NIO，调用方不断轮询
异步阻塞：一般不这样用
异步非阻塞：AIO，结果由内核通知回调
```

---

## 面试标准回答

> 同步、异步、阻塞、非阻塞区别？

```text
阻塞和非阻塞关注的是调用线程是否会被挂起等待结果；同步和异步关注的是结果通知机制。

阻塞表示调用后如果结果没准备好，线程会一直等待；非阻塞表示调用后如果结果没准备好，会立即返回。

同步表示调用方需要自己等待或轮询结果；异步表示调用方发起请求后可以继续做其他事，结果完成后由回调、事件或通知机制告知调用方。
```

## 十九、零拷贝（重点补充）

## 1. 传统 I/O 的数据拷贝次数

假设 Java 程序要从磁盘读取文件并通过网卡发送出去：

```text
磁盘 → 内核缓冲区(read buffer) → 用户缓冲区 → Socket缓冲区 → 网卡
```

共 **4 次拷贝**，其中 **2 次 CPU 拷贝**。

---

## 2. mmap（内存映射）

```text
磁盘文件映射到内核缓冲区（用户空间也能访问）
```

数据路径：

```text
磁盘 → 内核缓冲区（映射到用户空间）→ Socket缓冲区 → 网卡
```

变成 **3 次拷贝**，CPU 拷贝减少 **1 次**。

Java 中对应：

```java
MappedByteBuffer buffer = new RandomAccessFile(file, "r")
    .getChannel()
    .map(FileChannel.MapMode.READ_ONLY, 0, len);
```

RocketMQ 的索引文件（ConsumeQueue）用 mmap 加速读取。

---

## 3. sendfile（Linux 内核 2.1+）

完全绕过用户空间：

```text
磁盘 → 内核缓冲区 → 网卡
```

变成 **2 次拷贝**，**0 次 CPU 拷贝**。

Java 中对应：

```java
fileChannel.transferTo(position, len, socketChannel);
```

Kafka 读取消息并发送给 Consumer 时底层就用了 sendfile：

```text
磁盘文件 → 内核页缓存 → 网卡
     sendfile 系统调用
```

这是 Kafka 高吞吐的关键之一。

---

## 4. 对比

| 方案     | 拷贝次数 | CPU 拷贝 | 用户态参与 |
| ------ | ---- | ----- | ------ |
| 传统 I/O  | 4 次  | 2 次   | 需要      |
| mmap    | 3 次  | 1 次   | 需要（映射） |
| sendfile | 2 次  | 0 次   | 不需要     |

---

## 面试标准回答

> 什么是零拷贝？为什么 Kafka 和 RocketMQ 用它？

```text
零拷贝是指在 I/O 操作中，避免数据在内核缓冲区和用户缓冲区之间重复拷贝的技术。

传统 I/O 中，磁盘文件经过内核缓冲区、用户缓冲区、Socket 缓冲区才能到网卡，需要 4 次拷贝和 2 次 CPU 拷贝。

mmap 把磁盘文件直接映射到用户可访问的内核缓冲区，减少一次 CPU 拷贝，RocketMQ 的索引文件读取就用了 mmap。

sendfile 进一步让数据完全在内核空间流转，从磁盘直接到网卡，零次 CPU 拷贝，Kafka 的高吞吐消息发送就依赖 sendfile。

零拷贝能显著降低 CPU 开销和延迟，对高吞吐场景（Kafka、RocketMQ、Nginx）非常关键。
```

## 二十一、Java 后端关联点（重点补充）

## 1. JVM 和操作系统

Java 程序运行时，本质是一个 JVM 进程。

```text
一个 Java 服务 = 一个 JVM 进程
```

JVM 里面有很多线程：

```text
main 线程
GC 线程
JIT 编译线程
业务线程池线程
```

---

## 2. Java 线程和操作系统线程

现代 JVM 中，Java 线程映射为操作系统原生线程：

```text
Java Thread (new Thread())
    → JVM Thread (Java 线程对象)
    → OS Thread (Linux 内核线程，pthread)
```

Java 线程的阻塞/唤醒最终依赖 OS 的 `pthread_cond_wait / pthread_cond_signal`。

所以：

```text
Java 线程的阻塞 = 内核线程阻塞
Java 线程的上下文切换 = OS 线程上下文切换
```

这也是为什么 Java 并发编程里的问题（死锁、线程数过多）和 OS 层面的问题高度相关。

---

## 3. Java 协程（Virtual Threads / Loom）

Java 21 引入的虚拟线程（协程）：

```text
传统：Java 线程 → OS 线程（1:1）
协程：Java 协程 → 调度线程 → OS 线程（M:N）
```

特点：

```text
协程挂起不阻塞 OS 线程
单线程可以挂起成百上千个协程
减少线程数，降低上下文切换开销
```

适用于：

```text
IO 密集型任务（高并发网络请求）
爬虫
微服务高并发调用
```

**不适用于**：

```text
CPU 密集型任务（计算密集，协程切换也需要调度开销）
```

## 4. Java 内存模型和 OS 内存的关系

JVM 启动时向 OS 申请内存：

```text
java -Xms512m -Xmx1024m
     ↑初始堆    ↑最大堆
```

OS 视角看到的：

```text
JVM 进程占用 ~1GB 虚拟内存
OS 通过 cgroup 限制实际物理内存
```

---

## 5. JVM GC 和 OS 内存压力

GC 时会发生：

```text
标记阶段：遍历对象图，不影响 OS
清除/整理阶段：移动对象，可能触发 OS 换页
Full GC 整理：大量内存移动，可能产生缺页中断
```

不同 GC 对 OS 内存压力不同：

```text
CMS/G1：并发标记阶段对 OS 影响小，但碎片整理会 stop-the-world
ZGC：染色指针，GC 大部分并发，几乎不停顿
Shenandoah：类似 ZGC，Oracle OpenJDK 支持
```

## 6. 堆外内存 DirectMemory

Java NIO 的 ByteBuffer 默认堆内，但 Netty 等高性能框架用堆外内存：

```java
// 堆外内存，不受 GC 管理
ByteBuffer buffer = ByteBuffer.allocateDirect(1024);
```

为什么用堆外：

```text
减少 GC 频率（Netty 的 ByteBuf 池化，对象多，GC 压力大）
可以直接操作 OS 内存，和 sendfile 配合实现零拷贝
减少 JVM 和 OS 之间的数据拷贝
```

Kafka 和 Netty 大量使用堆外内存。

## 7. Java I/O 和操作系统 I/O 的对应

| Java I/O  | 底层 OS I/O        | 备注              |
| --------- | ---------------- | ---------------- |
| `InputStream.read()` | 阻塞 I/O (BIO)       | 同步阻塞           |
| `SocketChannel.read()` | 非阻塞 I/O         | 可设置 non-blocking |
| `Selector.select()` | I/O 多路复用（epoll）  | Java NIO 底层     |
| `AsynchronousServerSocketChannel` | 异步 I/O (Linux AIO) | Java AIO，Linux 7+ 支持不完善，Netty 不用 |

## 8. Netty 和 epoll

Netty 在 Linux 上底层用 epoll（通过 JNI 调用）：

```text
Java 代码：channel.read() → Netty NioEventLoop → epoll_wait → 返回就绪的 fd → 读取数据
```

Netty 的线程模型：

```text
Boss Group：接收连接（1 个或多个线程）
Worker Group：处理 IO 事件（多个线程，默认 CPU 核数 * 2）
```

每个 Worker 线程内部跑 epoll loop：

```java
// Netty 内部简化逻辑
while (true) {
    int ready = epollWait(epfd, events, MAX_EVENTS, timeout);
    for (每个就绪的 fd) {
        处理读写事件
    }
}
```

## 9. 数据库连接和线程

MySQL 连接默认 max_connections = 151。

每个 MySQL 连接底层对应一个 OS 线程（进程内）。

如果 Java 服务用 BIO 模式连接 MySQL：

```text
每个连接 = 1 个线程 = 1 个 OS 线程
连接数多 → 线程数多 → 上下文切换开销大
```

这就是为什么 Druid/HikariCP 连接池很重要：

```text
限制活跃连接数
复用连接
减少线程创建销毁开销
```

## 10. Redis 和 Linux 内存

Redis 纯内存数据库，数据全在 RAM。

如果 Redis 内存快满了，Linux OS 可能触发 OOM Killer：

```text
Redis 进程占用内存超过了 OS 阈值
Linux 选择杀掉 Redis 进程
```

Redis 的 maxmemory 配置：

```text
maxmemory 3gb
maxmemory-policy allkeys-lru
```

控制 Redis 自身不超过阈值，不触发 OS OOM。

## 11. 总结：Java 后端排查命令对照

| 问题  | 看什么     | 命令              |
| --- | -------- | ---------------- |
| Java 进程是否存活 | 进程       | `ps -ef | grep java` |
| 进程占多少内存   | RES 内存    | `top -p {pid}`    |
| 线程数是否过多   | 线程数      | `ps -eLf | grep java | wc -l` |
| 文件描述符是否耗尽 | fd 数量     | `ls /proc/{pid}/fd | wc -l` |
| 端口被占用      | 端口占用     | `netstat -tunlp | grep 8080` |
| 大量 TIME_WAIT  | 网络状态     | `ss -s` 或 `netstat -an | grep TIME_WAIT` |
| 磁盘 I/O 压力大   | 磁盘吞吐     | `iostat -x 1`      |
| 机器负载高      | load average | `uptime` 或 `top`   |

## 二十二、最终背诵版

```text
1. 操作系统负责管理 CPU、内存、磁盘、文件、I/O 设备等资源，并为应用程序提供系统调用。

2. 用户态权限低，内核态权限高。应用程序访问文件、网络等资源时，需要通过系统调用进入内核态。

3. 进程是资源分配的基本单位，线程是 CPU 调度的基本单位。

4. 同一进程内线程共享堆、方法区、文件句柄等资源，但每个线程有自己的栈和程序计数器。

5. 线程上下文切换是 CPU 换线程执行时保存旧现场、恢复新现场的过程；线程过多会增加切换成本、内存占用和锁竞争。

6. Java 后端通常用线程池控制线程数量、复用线程，避免每个任务都 new Thread 导致资源失控。

7. 并发是同一时间段内交替执行，并行是同一时刻真正同时执行。

8. 进程间通信方式包括管道、消息队列、共享内存、信号量、Socket 等。

9. 死锁产生需要四个条件：互斥、请求并保持、不可剥夺、循环等待。

10. 避免死锁可以固定加锁顺序、设置超时时间、避免嵌套锁、一次性申请资源。

11. 虚拟内存让每个进程拥有独立连续的虚拟地址空间，并通过页表映射到物理内存。

12. 分页按固定大小划分内存，分段按程序逻辑划分内存。

13. LRU 是最近最少使用，缓存满时淘汰最长时间未被访问的数据。

14. 文件描述符是操作系统为打开文件或 Socket 分配的整数标识。

15. BIO 是阻塞 I/O，一个连接通常对应一个线程。

16. NIO 是非阻塞 I/O，不会一直等待数据就绪。

17. I/O 多路复用可以用一个线程监听多个连接，常见机制有 select、poll、epoll。

18. select 有 fd 数量限制且需要遍历全部 fd；poll 解决数量限制但仍需遍历；epoll 基于事件通知，更适合高并发。

19. epoll LT 模式下只要就绪就持续通知，ET 模式下只在状态变化时通知一次；Netty 用 LT，RocketMQ 用 ET。

20. 阻塞/非阻塞关注线程是否等待；同步/异步关注结果通知方式。

21. 零拷贝（mmap/sendfile）减少内核缓冲区和用户缓冲区之间的数据拷贝，Kafka 用 sendfile，RocketMQ 用 mmap。

22. Java 程序本质是 JVM 进程，Java 线程最终由操作系统调度；虚拟线程（Loom）实现了 M:N 的协程模型，减少线程上下文切换开销。

23. Netty、Redis、Nginx 等高性能组件大量使用 I/O 多路复用；堆外内存（DirectMemory）配合零拷贝实现高吞吐数据传输。

24. 常用排查命令：ps、top、netstat、lsof、tail、grep、df、free、ss、iostat。
```
