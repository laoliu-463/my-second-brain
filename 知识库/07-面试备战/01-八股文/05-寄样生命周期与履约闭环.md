# 寄样生命周期与履约闭环

## 一、寄样状态机

### 1.1 六态流转

```
APPLIED（已申请）
    ↓ 审核通过
APPROVED（已审核）
    ↓ 发货
SHIPPED（已发货）
    ↓ 签收
SIGNED（已签收）
    ↓ 待交作业
PENDING_HOMEWORK（待交作业）
    ↓ 命中订单
COMPLETED（已完成）
```

| 状态 | 说明 | 可执行动作 |
|---|---|---|
| `APPLIED` | 申请人提交寄样申请 | 审核通过/拒绝 |
| `APPROVED` | 审核通过，等待发货 | 发货 |
| `SHIPPED` | 已发出样品 | 签收 |
| `SIGNED` | 达人已签收样品 | 填写物流信息/待交作业 |
| `PENDING_HOMEWORK` | 达人需提交推广证据 | 命中订单自动完成 |
| `COMPLETED` | 履约完成 | - |
| `REJECTED` | 申请被拒 | - |
| `CLOSED` | 超时关闭 | - |

### 1.2 状态流转代码

```java
public class SampleLifecycleService {

    public void processAction(Long sampleId, SampleAction action, SampleActionRequest request) {
        Sample sample = sampleMapper.selectById(sampleId);

        switch (action) {
            case APPROVE -> {
                validateTransition(sample.getStatus(), APPROVED);
                sample.setStatus(APPROVED);
                // 发货操作人从 ops_staff 改为 biz_leader
            }
            case SHIP -> {
                validateTransition(sample.getStatus(), APPROVED, SHIPPED);
                sample.setStatus(SHIPPED);
                sample.setTrackingNo(request.getTrackingNo());
                sample.setShipTime(LocalDateTime.now());
                sample.setShipOperator(getCurrentUserId());
            }
            case SIGN -> {
                validateTransition(sample.getStatus(), SHIPPED, SIGNED);
                sample.setStatus(SIGNED);
                sample.setSignTime(LocalDateTime.now());
            }
            case SUBMIT_HOMEWORK -> {
                validateTransition(sample.getStatus(), SIGNED, PENDING_HOMEWORK);
                sample.setStatus(PENDING_HOMEWORK);
            }
            case AUTO_COMPLETE -> {
                // 订单命中时自动完成
                validateTransition(sample.getStatus(), PENDING_HOMEWORK, COMPLETED);
                sample.setStatus(COMPLETED);
                sample.setCompleteTime(LocalDateTime.now());
            }
            case REJECT -> {
                sample.setStatus(REJECTED);
                sample.setRejectReason(request.getReason());
            }
            case CLOSE -> {
                // 超时自动关闭
                sample.setStatus(CLOSED);
                sample.setCloseReason("超时未完成，系统自动关闭");
            }
        }

        sampleMapper.updateById(sample);
        // 写状态流转日志
        sampleStatusLogService.log(sampleId, action, getCurrentUserId());
    }
}
```

## 二、寄样自动完成机制

### 2.1 订单命中触发

```java
// OrderSyncService 订单同步时检查
public void syncOrder(OrderRawData raw) {
    Order order = orderMapper.selectByPickSource(raw.getPickSource());

    if (order != null) {
        // 找到关联的寄样单
        Sample sample = sampleMapper.selectByTalentAndProduct(
            order.getTalentId(), order.getProductId());

        if (sample != null && sample.getStatus() == PENDING_HOMEWORK) {
            // 命中订单，自动完成寄样
            sampleLifecycleService.processAction(
                sample.getId(),
                SampleAction.AUTO_COMPLETE,
                null
            );
            log.info("Sample {} auto-completed by order {}", sample.getId(), order.getId());
        }
    }
}
```

### 2.2 定时任务兜底

```java
// SampleLifecycleJob 每天检查超时寄样
@Scheduled(cron = "0 0 2 * * ?")  // 每天凌晨2点
public void closeExpiredSamples() {
    // 使用 Redis 分布式锁，多实例只执行一次
    String lockKey = "job:sample-lifecycle:close-expired";
    if (!redisLockService.tryLock(lockKey, Duration.ofMinutes(5))) {
        return;
    }

    try {
        // 查询超时未完成的寄样单
        List<Sample> expired = sampleMapper.selectExpired(
            LocalDateTime.now().minusDays(sampleDefaultStandard.getAutoCloseDays())
        );

        for (Sample sample : expired) {
            processAction(sample.getId(), SampleAction.CLOSE, null);
        }
    } finally {
        redisLockService.unlock(lockKey);
    }
}
```

## 三、寄样资格校验

### 3.1 申请前预检

```java
public class SampleEligibilityService {

    public EligibilityCheckResult checkEligibility(Talent talent, Product product) {
        List<String> violations = new ArrayList<>();

        // 1. 达人状态校验
        if (talent.getStatus() == TalentStatus.BLACKLISTED) {
            violations.add("达人在黑名单中，禁止寄样");
        }
        if (talent.isInProtectionPeriod() && !talent.hasRecentOrders()) {
            violations.add("达人在保护期内且近期无订单产出");
        }

        // 2. 独家达人阈值校验
        if (talent.isExclusive()) {
            long monthlySamples = sampleMapper.countThisMonth(talent.getId());
            if (monthlySamples >= exclusiveTalentThreshold) {
                violations.add("独家达人本月寄样次数已达上限");
            }
        }

        // 3. 重复申请校验
        if (sampleMapper.hasActiveSample(talent.getId(), product.getId())) {
            violations.add("该达人对此商品已有进行中的寄样单");
        }

        // 4. 商品状态校验
        if (product.getBizStatus() != APPROVED && product.getBizStatus() != ASSIGNED) {
            violations.add("商品状态不是 APPROVED 或 ASSIGNED，无法寄样");
        }

        if (violations.isEmpty()) {
            return EligibilityCheckResult.pass();
        }
        return EligibilityCheckResult.fail(violations);
    }
}
```

## 四、分布式锁防护

### 4.1 为什么需要分布式锁？

多实例部署时，定时任务每个实例都会执行。如果不用锁：

```
实例A: 扫描超时寄样单 → 查到 ID=1,2,3 → 开始关单
实例B: 扫描超时寄样单 → 查到 ID=1,2,3 → 也开始关单
结果: 寄样单被重复关闭，状态日志重复写入
```

### 4.2 Redis 分布式锁实现

```java
public class RedisLockService {
    public boolean tryLock(String lockKey, Duration ttl) {
        String value = UUID.randomUUID().toString();
        Boolean acquired = redisTemplate.opsForValue()
            .setIfAbsent(lockKey, value, ttl);
        return Boolean.TRUE.equals(acquired);
    }

    public void unlock(String lockKey) {
        // 使用 Lua 脚本保证原子性：只有持有锁的人才删除
        String script = """
            if redis.call('get', KEYS[1]) == ARGV[1] then
                return redis.call('del', KEYS[1])
            else
                return 0
            end
            """;
        redisTemplate.execute(
            new DefaultRedisScript<>(script, Long.class),
            Collections.singletonList(lockKey),
            lockValue.get()
        );
    }
}
```

## 五、面试高频追问

**Q: 寄样状态机为什么不直接用 Spring State Machine？**
A: Spring State Machine 适合状态多、流转规则复杂、需要可视化状态图的场景。本项目寄样只有 7 个状态，流转规则固定，用枚举 + switch 校验完全够用。引入 State Machine 会增加依赖复杂度和学习成本。

**Q: 自动完成寄样是同步还是异步？**
A: 当前是同步的——订单同步时立即检查并更新寄样状态。如果未来订单量大，可以在订单同步后发消息到 MQ，异步处理寄样完成逻辑。

**Q: 分布式锁用 Redis 够吗？Redis 挂了怎么办？**
A: 对于大多数场景够用，Redis 挂了你说的"多实例重复执行"问题本来也不存在（Redis 挂了说明多实例都连不上，都不会执行）。如果需要更高可靠性，可以换 Redisson 的 RedLock。但对于寄样关闭这种"晚几分钟执行也没关系"的定时任务，Redis 单实例锁完全够用。

**Q: 寄样超时关闭的天数为什么做成配置？**
A: 业务规则可能随时调整。做配置的好处：运营人员可以通过管理后台修改，不需要改代码 + 发版 + 重启。比如淡季可能延长到 45 天，旺季缩短到 15 天。
