# 商品状态机与主链路闭环

## 一、商品主链路完整流程

```
选品候选
    ↓ 申请入库
PENDING_AUDIT（待审核）
    ↓ 审核通过
APPROVED（已通过）
    ↓ 分配负责人
ASSIGNED（已分配）
    ↓ 生成推广链接
LINKED（已关联）
    ↓ 达人跟进
FOLLOWING（跟进中）
```

状态说明：

| 状态 | 说明 | 可执行动作 |
|---|---|---|
| `PENDING_AUDIT` | 待审核，新入库商品默认状态 | 审核通过/拒绝 |
| `APPROVED` | 审核通过，待分配 | 分配负责人 |
| `ASSIGNED` | 已分配负责人，待转链 | 生成推广链接 |
| `LINKED` | 已生成推广链接 | 达人跟进/推进 |
| `FOLLOWING` | 达人跟进中 | - |
| `REJECTED` | 审核拒绝 | - |
| `BOUND` | 历史兼容状态（旧数据） | 读取时归一化为 `APPROVED` |

## 二、状态机守门规则

### 2.1 核心守则

```java
// ProductBizStatusService.java
public class ProductBizStatusService {

    /**
     * 状态机守门规则：所有状态流转必须经过此校验
     */
    public void validateTransition(ProductBizStatus from, ProductBizStatus to) {
        Map<ProductBizStatus, Set<ProductBizStatus>> ALLOWED_TRANSITIONS = Map.of(
            PENDING_AUDIT, Set.of(APPROVED, REJECTED),
            APPROVED,    Set.of(ASSIGNED, BOUND),   // BOUND 仅兼容旧数据
            ASSIGNED,    Set.of(LINKED),
            LINKED,      Set.of(FOLLOWING),
            FOLLOWING,   Set.of()                   // 最终态
        );

        Set<ProductBizStatus> allowed = ALLOWED_TRANSITIONS.getOrDefault(from, Set.of());
        if (!allowed.contains(to)) {
            throw new IllegalStateException(
                String.format("Invalid status transition: %s → %s", from, to));
        }
    }

    /**
     * APPLY_APPROVED 实际上是入库 + 审核合并动作
     * 审核通过 = 直接进入 APPROVED 状态并加入商品库
     */
    public void applyAuditResult(Long productId, boolean approved, String reason) {
        Product product = productMapper.selectById(productId);
        validateTransition(product.getBizStatus(), approved ? APPROVED : REJECTED);

        product.setBizStatus(approved ? APPROVED : REJECTED);
        productMapper.updateById(product);

        if (approved) {
            // 审核通过自动加入共享商品库
            productLibraryService.addToLibrary(productId);
        }

        // 写操作日志
        operationLogService.log(productId, "AUDIT", approved ? "APPROVE" : "REJECT", reason);
    }
}
```

### 2.3 旧状态兼容

```java
// ProductBizStatus 枚举读取时归一化
public ProductBizStatus normalize(String dbValue) {
    if ("SYNCED".equals(dbValue)) {
        // 历史 SYNCED 状态统一归一为 PENDING_AUDIT，避免旧数据读取报错
        return PENDING_AUDIT;
    }
    if ("BOUND".equals(dbValue)) {
        // BOUND 仅作为历史兼容读取，不再作为新流转目标
        return APPROVED;
    }
    return ProductBizStatus.valueOf(dbValue);
}
```

## 三、商品主链路完整闭环

### 3.1 从选品到推广的七步闭环

```
Step 1: 选品
  POST /api/colonel/activities/{activityId}/products/{productId}/library-entry
  → 商品进入 PENDING_AUDIT 状态

Step 2: 审核
  PUT /api/colonel/activities/{activityId}/products/{productId}/audit-result
  → 审核通过 → APPROVED，自动加入商品库

Step 3: 分配
  PUT /api/colonel/activities/{activityId}/products/{productId}/assignee
  → 分配负责人 → ASSIGNED

Step 4: 转链
  POST /api/colonel/activities/{activityId}/products/{productId}/promotion-links
  → 生成 pick_source 推广链接 → LINKED
  → 同时写入 pick_source_mapping 归因映射

Step 5: 达人分发
  → 达人获取推广链接，推广给用户

Step 6: 订单同步
  → 订单回流，通过 pick_source 匹配归因

Step 7: 寄样跟进
  → 寄样命中订单自动完成
```

### 3.2 关键设计决策：审核通过 = 自动入库

```java
// 旧流程：审核 → 入库 → 分配（两步）
// 新流程：审核通过 → 自动入库 + 分配合并
public void audit(Long productId, AuditRequest request) {
    Product product = productMapper.selectById(productId);

    if (request.isApproved()) {
        product.setBizStatus(BizStatus.APPROVED);
        // 入库动作在审核时一起完成
        productLibraryService.addToLibrary(productId);
        // 分配也在同一步完成（如果有默认负责人）
        if (request.getAssigneeId() != null) {
            product.setAssigneeId(request.getAssigneeId());
            product.setBizStatus(BizStatus.ASSIGNED);
        }
    } else {
        product.setBizStatus(BizStatus.REJECTED);
    }

    productMapper.updateById(product);
}
```

**为什么合并？**

因为"审核通过"和"加入商品库"是业务上的同一个决策点——审核人员认为这个商品可以推，就意味着它应该立即进入共享商品库可见。拆成两步只会增加操作负担和状态不一致的风险。

## 四、N+1 查询治理

### 4.1 批量装配模式

商品列表页最常见的 N+1 问题：

```java
// 旧版本（N+1）
List<Product> products = productMapper.selectPage(query);
for (Product p : products) {
    ProductSnapshot snapshot = snapshotMapper.selectById(p.getSnapshotId()); // N 次
    SysUser user = sysUserMapper.selectById(p.getAssigneeId()); // N 次
    ProductOperationState state = stateService.getState(p.getId()); // N 次
    // 总共 3N 次额外查询
}

// 新版本（批量装配）
List<Product> products = productMapper.selectPage(query);

List<Long> snapshotIds = products.stream().map(Product::getSnapshotId).toList();
List<Long> userIds = products.stream().map(Product::getAssigneeId).toList();
List<Long> productIds = products.stream().map(Product::getId).toList();

Map<Long, ProductSnapshot> snapshotMap = snapshotMapper.selectBatchIds(snapshotIds);
Map<Long, SysUser> userMap = sysUserMapper.selectBatchIds(userIds);
Map<Long, ProductOperationState> stateMap = stateService.getStateBatch(productIds);

// 组装
for (Product p : products) {
    p.setSnapshot(snapshotMap.get(p.getSnapshotId()));
    p.setAssignee(userMap.get(p.getAssigneeId()));
    p.setOperationState(stateMap.get(p.getId()));
}
// 总共 3 次额外查询（批量 IN 查询）
```

### 4.2 效果对比

| 场景 | 旧版查询次数 | 新版查询次数 | 100条商品节省 |
|---|---|---|---|
| 列表页 | 3N+1 | 4 | ~296 次 |

## 五、面试高频追问

**Q: 状态机和状态模式（State Pattern）有什么区别？**
A: 状态机强调"状态流转规则的枚举和校验"，适合规则固定的场景。状态模式把每个状态封装成对象，让状态自己决定下一个状态，适合状态行为复杂且需要动态扩展的场景。本项目用状态机（枚举 + 校验方法）就够了，因为状态规则相对固定，不需要运行时动态切换状态行为。

**Q: 为什么审核通过要自动入库，而不是分开两步？**
A: 业务上"审核通过"和"可以共享"是同一个决策点。拆开两步会增加操作负担，更重要的是增加状态不一致风险——审核通过了但入库失败了怎么办？合并为一个原子动作，避免中间态。

**Q: 商品库和活动商品是什么关系？**
A: `商品库` = 已审核通过的共享商品池（`/products`），所有业务人员可见。`活动商品` = 某个活动下的具体商品（`/colonel/activities/{id}/products`），只在那个活动的上下文中流转。一件商品可以同时出现在商品库和多个活动商品列表中。

**Q: FOLLOWING 状态之后是什么？会回到前面的状态吗？**
A: 当前设计里 `FOLLOWING` 是终态。商品一旦进入跟进中就不再流转。如果需要EOUT，可通过操作日志追溯，但不需要专门的状态回退。后续如果有"重新推广"需求，应该用新的推广链接（新的 LINKED）来处理，而不是回退状态。
