# 计算机网络核心知识点

---

## 一、HTTP 核心

### 1.1 HTTP 1.0 / 1.1 / 2.0 / 3.0 对比

| 版本 | 关键特性 | 问题 |
|---|---|---|
| HTTP 1.0 | 短连接，每请求重建 TCP | 连接开销大 |
| HTTP 1.1 | 长连接（keep-alive）、管道化 | 队头阻塞（HOL Blocking） |
| HTTP 2.0 | 多路复用、Header 压缩、Server Push | TCP 队头阻塞（TCP 层的 HOL） |
| HTTP 3.0 | QUIC（UDP 实现）、无队头阻塞 | 部署复杂、穿越防火墙难 |

**HTTP 2.0 多路复用**：

```
HTTP 1.1: 并发需要多个 TCP 连接（浏览器通常限制 6 个）
           请求A ──────────────────────────→

HTTP 2.0: 同一 TCP 连接上并行传输多个请求
           帧A1 ─────→ 帧B1 ─────→ 帧A2 ────→ 帧B2 ────→
           响应A1 ───→ 响应B1 ───→ 响应A2 ───→ 响应B2 ──→
```

**为什么 HTTP 3.0 用 UDP？**

TCP 有严格的可靠性保证和拥塞控制，但在弱网（丢包率高）环境下，TCP 的拥塞控制会主动降速，导致性能急剧下降。QUIC（HTTP 3.0 的传输层）基于 UDP，自己实现了可靠性、流量控制、拥塞控制，针对性解决了"一丢包全局降速"的问题。

### 1.2 GET vs POST 对比

| 维度 | GET | POST |
|---|---|---|
| 语义 | 获取资源 | 提交/修改资源 |
| 参数位置 | URL query string | 请求体（Body） |
| 长度限制 | 受 URL 长度限制（浏览器约 2KB） | 无限制（受服务器配置） |
| 缓存 | 可被缓存 | 通常不缓存 |
| 幂等 | 幂等 | 非幂等 |
| 安全性 | 参数在 URL 中，明文可见 | 参数在 Body，可加密 |

**POST 不等于安全**：POST 参数在 Body 里，不在 URL 中显示，但抓包或代理同样可见。安全靠 HTTPS，不是靠 GET/POST。

### 1.3 HTTP 状态码

```
2xx 成功：200 OK, 201 Created, 204 No Content
3xx 重定向：301 Moved Permanently, 302 Found, 304 Not Modified
4xx 客户端错误：400 Bad Request, 401 Unauthorized, 403 Forbidden, 404 Not Found, 429 Too Many Requests
5xx 服务端错误：500 Internal Server Error, 502 Bad Gateway, 503 Service Unavailable, 504 Gateway Timeout
```

**304 Not Modified**：缓存相关的响应。服务器告诉浏览器"这个资源你没有更新过，直接用你本地的缓存吧"。浏览器从本地缓存读取，节省带宽。

**429 Too Many Requests**：接口限流。本项目抖店 SDK 联调时遇到过，需要按退避策略重试。

### 1.4 HTTP 缓存

**强制缓存**（Cache-Control / Expires）：

```
Cache-Control: max-age=3600  ← 3600秒内直接用缓存，不发请求
Expires: Mon, 01 Jan 2027 00:00:00 GMT  ← 到时间点才发请求
```

**协商缓存**（Last-Modified / ETag）：

```
首次请求：
  服务器返回：ETag: "abc123"
             Last-Modified: Mon, 01 Jan 2026 12:00:00 GMT

后续请求：
  浏览器发送：If-None-Match: "abc123"
             If-Modified-Since: Mon, 01 Jan 2026 12:00:00 GMT

  服务器判断：
    ETag 匹配 → 304 Not Modified → 浏览器用本地缓存
    ETag 不匹配 → 200 OK + 新资源 + 新 ETag
```

**Cache-Control 指令**：

| 指令 | 说明 |
|---|---|
| `no-cache` | 每次都协商（发请求给服务器确认） |
| `no-store` | 不缓存（最严格，敏感数据） |
| `public` | 可被任何节点缓存（CDN、浏览器） |
| `private` | 只能被浏览器缓存 |

### 1.5 HTTPS 建立连接过程

```
1. 客户端 → 服务器：Client Hello（支持的TLS版本、加密套件、随机数）
2. 服务器 → 客户端：Server Hello + 证书（公钥 + 证书链）
3. 客户端：验证证书（浏览器内置根证书）
4. 客户端 → 服务器：Pre-Master Secret（用证书公钥加密）
5. 双方：用 Pre-Master Secret + 两个随机数生成对称密钥
6. 双向加密通信开始
```

**为什么不用非对称加密全程通信？**

非对称加密（RSA/ECDHE）计算成本是对称加密（AES）的 100-1000 倍。HTTPS 握手后用对称密钥加密数据传输，性能差距巨大。

## 二、TCP 三次握手与四次挥手

### 2.1 三次握手

```
客户端                服务器
  │                    │
  │─── SYN=1, seq=x ──→│ 客户端发送SYN，请求建立连接，发送序号x
  │                    │
  │←── SYN=1, ACK=1 ───│  服务器返回SYN+ACK，确认收到x+1，请求序号y
  │    ack=x+1          │
  │                    │
  │─── ACK=1, seq=x+1 ─→│ 客户端确认收到y+1，连接建立
  │    ack=y+1          │
  │                    │
  ╞═══ 连接建立完成 ═══╡
```

**为什么是三次？**

防止历史连接初始化导致的混乱。例如：客户端发了 SYN=1 请求（旧连接），但网络延迟导致这个包之后才到服务器；服务器收到后返回 SYN+ACK，客户端发现 seq 不对（期望 y，实际是旧的），发送 RST 拒绝连接。只有第三次握手时客户端确认了"这个连接是我发的"，双方才能开始可靠通信。

### 2.2 四次挥手

```
客户端                服务器
  │                    │
  │─── FIN=1, seq=u ──→│ 客户端发送FIN，请求关闭客户端→服务器方向
  │←── ACK=1, ack=u+1 ──│ 服务器确认，客户端→服务器方向关闭
  │                    │（服务器可能还在发数据）
  │                    │
  │←── FIN=1, seq=w ───│ 服务器发送FIN，请求关闭服务器→客户端方向
  │                    │
  │─── ACK=1, ack=w+1 ─→│ 客户端确认，所有方向关闭
  │                    │
  │    （等待 2MSL）   │ 客户端等 2MSL 才真正关闭
```

**为什么等 2MSL？**

MSL（Maximum Segment Lifetime）是一个 IP 分片在网络中能存在的最长时间，通常 60 秒。等待 2MSL 的原因：
1. 保证服务器收到最后的 ACK（否则服务器会重发 FIN，客户端已关闭无法接收）
2. 让本次连接产生的所有报文都从网络中消失，防止影响下一个连接

**本项目中的实际应用**：

```yaml
# application.yml 中配置了 HTTP 连接超时
restTemplate:
  connectTimeout: 3000  # 3秒，TCP握手超时
  readTimeout: 5000     # 5秒，读取数据超时
```

### 2.3 TCP 滑动窗口与拥塞控制

**滑动窗口**：解决"发送快但接收方处理慢"的问题。窗口大小告诉对方"我已经发出去了，在收到确认之前最多还能发多少"。

**拥塞控制**：解决"网络本身处理不过来"的问题。

```
慢启动 → 拥塞避免 → 乘法减小
cwnd（拥塞窗口）从小慢慢增大，出现丢包（拥塞）时 cwnd 减半重新开始
```

**本项目中遇到的问题**：

抖店接口返回 429 Too Many Requests 时，本项目实现了指数退避重试：
```java
// 抖店接口限流重试
int maxRetries = 3;
for (int i = 0; i < maxRetries; i++) {
    try {
        return douyinApiClient.post(method, params);
    } catch (RateLimitException e) {
        long waitMs = (long) Math.pow(2, i) * 1000; // 1s, 2s, 4s
        Thread.sleep(waitMs);
    }
}
```

## 三、TCP 与 UDP 的区别

| 维度 | TCP | UDP |
|---|---|---|
| 连接性 | 面向连接（三次握手） | 无连接 |
| 可靠性 | 可靠（确认+重传+序号） | 不可靠（可能丢包/乱序） |
| 拥塞控制 | 有 | 无 |
| 传输效率 | 相对低（有确认机制） | 高（无额外开销） |
| 头部大小 | 20-60 字节 | 8 字节 |
| 使用场景 | HTTP, 文件传输, 邮件 | DNS, VoIP, 视频, QUIC |

**为什么 DNS 用 UDP？**

DNS 查询通常很小（几十字节），UDP 头部只有 8 字节，TCP 至少 20 字节，UDP 传输效率更高。而且 DNS 查询是"问-答"模式，不需要双向持续通信，丢了重发即可。

**为什么 QUIC（HTTP 3.0）选 UDP？**

因为 TCP 的拥塞控制是 OS 内核实现的，无法定制化。而 QUIC 在用户态实现，可以在应用层灵活控制，解决 HTTP 2.0 的 TCP 队头阻塞问题。

## 四、常见面试问题

### Q: HTTP 和 TCP 的区别？

HTTP 是应用层协议，定义了"浏览器和服务器怎么说话"（请求格式、响应格式、状态码）。TCP 是传输层协议，定义了"字节怎么可靠地从 A 传到 B"。HTTP 依赖 TCP 传输数据，但两者解决的问题不同。

### Q: 为什么有了 HTTPS 还需要 Token？

HTTPS 只加密传输通道，防止中间人窃听和篡改。Token（JWT）是身份凭证，证明"这个请求是谁发的"。两者解决不同问题：HTTPS 防网络攻击，Token 防身份伪造。

### Q: 队头阻塞是什么？

**TCP 队头阻塞**：HTTP 2.0 在同一个 TCP 连接上传输多个请求/响应。如果其中一个 TCP 包丢了，TCP 的重传机制会让这个包之后的所有数据都等着，即使后面的包本身没丢也要等重传完成。这就是 TCP 层的队头阻塞（HOL Blocking）。

**HTTP 队头阻塞**：HTTP 1.1 的管道化（pipelining）也有类似问题，如果第一个请求响应慢，后面请求都会被卡住。浏览器默认关闭管道化就是因为这个问题。

### Q: 连接复用是什么？

HTTP 1.1 的 keep-alive 使得一个 TCP 连接可以发送多个 HTTP 请求/响应，避免每次请求都重新三次握手。HTTP 2.0 的多路复用更进一步，同一连接上可以并行传输多个请求/响应，不需要按顺序返回。

### Q: 什么是断点续传？

客户端下载大文件时，先问服务器"这个文件支持范围请求吗"（发送 Range 请求头）。服务器返回 `206 Partial Content` 和文件片段范围。断点续传时，客户端从断点位置继续请求。实现依赖 HTTP 的 Range 请求头和 206 状态码。
