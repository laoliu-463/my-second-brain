# 订单归因系统设计

## 一、什么是订单归因？

订单归因要回答的问题是：**这一单是怎么来的？**

团长 SaaS 场景中，商品通过推广链接分发到达人，达人推广后用户下单。这个订单到底算谁的业绩？

- 如果用户是通过某个达人的推广链接下单 → 归因到该达人（已归因）
- 如果用户是自然流量下单（没有通过任何推广链接）→ 无法归因（未归因）

归因是佣金结算的基础：**只有归因成功的订单才能计入达人和团长的业绩。**

## 二、抖店归因的两种通道

### 2.1 通道一：pick_source（外部推广链接归因）

用户通过外部推广链接购买时，URL 中带有 `pick_source` 参数：

```
https://haohuo.jinritemai.com/...?pick_source=abc123
```

订单同步时会把这个 `pick_source` 记下来，然后在 `pick_source_mapping` 表里查对应哪个达人/活动/商品。

```sql
-- 推广链接生成时写入映射
INSERT INTO pick_source_mapping (pick_source, activity_id, product_id, colonel_buyin_id, talent_id)
VALUES ('abc123', 3916506, 3810562766247428542, 7351155267604218149, 'talent_001');
```

```java
// 归因时查询映射
PickSourceMapping mapping = pickSourceMappingService.findByPickSource(order.getPickSource());
if (mapping != null) {
    orderService.attributeToTalent(order, mapping);
}
```

### 2.2 通道二：colonel_order_info（抖店原生归因）

用户通过视频/直播间下单时，抖店原生归因通过 `colonel_order_info` 字段传递：

```json
{
  "colonel_order_info": {
    "colonel_buyin_id": "7351155267604218149",
    "activity_id": 3916506
  }
}
```

这是抖店官方的归因通道，不需要用户点击外部推广链接。订单 raw JSON 中 `colonel_order_info` 包含 `activity_id` 和 `colonel_buyin_id`（团长在抖店的唯一标识）。

### 2.3 两种通道对比

| 维度 | pick_source（外部链接） | colonel_order_info（抖店原生） |
|---|---|---|
| 触发条件 | 用户点击外部推广链接 | 用户通过视频/直播间下单 |
| 归因依据 | URL 中的 pick_source 参数 | 订单中的 colonel_buyin_id |
| 数据来源 | 链接跳转 | 抖店直接透传 |
| 适用场景 | 外部流量（小红书/微信等） | 抖店内自然流量/直播 |

### 2.4 为什么会有未归因订单？

202 个 `NO_PICK_SOURCE` 订单 = 用户通过抖店内自然流量下单，但本地业务系统没有对应的活动/商品映射。

抖店原生归因了（`colonel_order_info` 有数据），但本地系统不知道这个 `activity_id + colonel_buyin_id` 对应哪个商品，所以无法计入业绩。

**这是业务问题，不是接口问题。**

## 三、归因实现代码

### 3.1 AttributionService.resolveAttribution()

```java
@Service
public class AttributionService {
    private final PickSourceMappingService pickSourceMappingService;
    private final ColonelsettlementActivityMapper activityMapper;

    public AttributionResult resolveAttribution(OrderRawData raw) {
        // 通道一：pick_source 归因
        String pickSource = raw.getPickSource();
        if (StringUtils.hasText(pickSource)) {
            PickSourceMapping mapping = pickSourceMappingService.findExactMatch(
                pickSource,
                raw.getActivityId(),
                raw.getProductId(),
                raw.getUserId()
            );
            if (mapping != null) {
                return AttributionResult.attributed(mapping.getTalentId(), mapping.getColonelBuyinId());
            }
            return AttributionResult.unattributed("PICK_SOURCE_MAPPING_NOT_FOUND");
        }

        // 通道二：colonel_order_info 抖店原生归因
        ColonelOrderInfo colonelInfo = raw.getColonelOrderInfo();
        if (colonelInfo != null && StringUtils.hasText(colonelInfo.getColonelBuyinId())) {
            // 用 activity_id + colonel_buyin_id 查本地映射
            String colonelBuyinId = colonelInfo.getColonelBuyinId();
            Long activityId = colonelInfo.getActivityId();

            // 先尝试二级团长（抖店可能返回 colonel_order_info_second）
            if (raw.getColonelOrderInfoSecond() != null) {
                colonelBuyinId = raw.getColonelOrderInfoSecond().getColonelBuyinId();
                activityId = raw.getColonelOrderInfoSecond().getActivityId();
            }

            // 查询本地是否有这个活动+团长组合
            ColonelsettlementActivity activity = activityMapper
                .findByActivityIdAndColonelBuyinId(activityId, colonelBuyinId);

            if (activity != null) {
                return AttributionResult.attributed(activity.getTalentId(), colonelBuyinId);
            }
            return AttributionResult.unattributed("COLONEL_MAPPING_NOT_FOUND");
        }

        return AttributionResult.unattributed("NO_PICK_SOURCE");
    }
}
```

### 3.2 pick_source_mapping 复合键设计

```sql
-- 复合唯一键：(pick_source, activity_id, product_id, user_id)
-- 同一 pick_source 可用于不同活动+商品组合
CREATE UNIQUE INDEX idx_pick_source_composite
ON pick_source_mapping (pick_source, activity_id, product_id, user_id);
```

旧版用 `(pick_source)` 全局唯一键，导致同一链接复用于不同商品时归因失败。改为复合键后，一个 `pick_source` 可以对应多个 `(activity_id, product_id)` 组合。

### 3.3 归因重算机制

历史订单可以重新归因：

```java
// POST /api/orders/replay-attribution
public ReplayResult replayAttribution(Long orderId) {
    Order order = orderMapper.selectById(orderId);
    OrderRawData raw = parseRawPayload(order.getExtraData());
    AttributionResult result = attributionService.resolveAttribution(raw);

    if (result.isAttributed()) {
        orderMapper.updateAttribution(orderId, result.getTalentId(), result.getColonelBuyinId());
    }
    return new ReplayResult(result);
}
```

## 四、未归因订单排查

### 4.1 归因状态机

```
订单同步
    ↓
是否有 pick_source?
    ├─ 有 → 查 pick_source_mapping → 命中? → 已归因
    │         └─ 未命中 → PICK_SOURCE_MAPPING_NOT_FOUND
    └─ 无 → 是否有 colonel_order_info?
              ├─ 有 → 查 colonelsettlement_activity
              │       ├─ 命中 → 已归因
              │       └─ 未命中 → COLONEL_MAPPING_NOT_FOUND
              └─ 无 → NO_PICK_SOURCE（抖店自然流量）
```

### 4.2 排查接口

```http
GET /api/orders/order-attribution-unattributed
```

返回所有未归因订单及其原因，供运营排查：

| 原因 | 含义 | 下一步 |
|---|---|---|
| `NO_PICK_SOURCE` | 抖店自然流量，无任何归因标识 | 业务问题，需确认推广策略 |
| `PICK_SOURCE_MAPPING_NOT_FOUND` | 链接被点击但映射未建立 | 检查链接生成时序 |
| `COLONEL_MAPPING_NOT_FOUND` | 抖店归因了但本地没有这个活动+团长组合 | 补录活动映射 |

### 4.3 常见归因失败原因

1. **商品未入库**：推广链接生成时商品还没入库，`pick_source_mapping` 没有这条记录
2. **活动已结束**：用户点击链接时活动已结束，归因通道关闭
3. **colonel_buyin_id 不匹配**：抖店返回的 `colonel_buyin_id` 和本地活动记录不一致（可能用了二级团长）
4. **自然流量**：用户没有通过任何推广链接，纯自然购买

## 五、面试高频追问

**Q: 归因失败和归因延迟有什么区别？**
A: 归因失败是"找不到归因通道"（映射不存在），归因延迟是"找到了但还没落库"（比如 Webhook 回调还没到）。处理方式也不同：失败要修复映射数据，延迟只需要等。

**Q: pick_source_mapping 表为什么用复合键而不是单一 pick_source 键？**
A: 因为同一个推广链接可以被复用到不同活动的不同商品上。如果用单一键，同一 `pick_source` 的第二次使用就会冲突，导致归因失败。实际业务中团长经常复用链接模板。

**Q: colonel_buyin_id 和 short_id 是什么关系？**
A: `colonel_buyin_id` 是抖店给团长的唯一标识（19位数字），`short_id` 是用户可读的短 ID（8-10位）。两者可以互相转换，但 `short_id` 可能不唯一，跨团场景下必须用 `colonel_buyin_id` 做精确匹配。

**Q: 归因系统在架构上属于哪个层次？**
A: 属于业务 Service 层。它依赖 Gateway 层同步的订单数据（OrderGateway），调用 Repository 层读写 `pick_source_mapping`，最终输出到 `colonelsettlement_order` 表。不属于 Gateway 层，因为归因规则是业务逻辑，不是协议适配。
