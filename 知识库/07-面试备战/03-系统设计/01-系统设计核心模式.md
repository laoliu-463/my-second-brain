## 一、认证授权系统设计

### 1.1 JWT 双令牌方案

**需求**：用户登录后获得 Token，有效期结束时自动续签，登出后 Token 立即失效。

```
用户登录 → 颁发 Access Token(2h) + Refresh Token(7d)
    ↓
前端请求带 Access Token
    ↓
Token 过期? → 用 Refresh Token 换新 Access Token
    ↓
登出 → Redis 黑名单标记，Token 立即失效
```

**核心实现**：

- Access Token 携带 `type=access` claim，每次请求验证签名和过期时间
- Refresh Token 携带 `type=refresh` claim，仅用于续签
- Refresh Token 用 SHA-256 哈希存储在 Redis，不存明文
- 登出时两级吊销：Access Token 和 Refresh Token 同时加入黑名单

**为什么不存 Session 在 Redis？**

| 方案 | 优点 | 缺点 |
|---|---|---|
| Session 存 Redis | 可以随时删除，精确控制 | 多一次 Redis 查询，分布式部署需 Session 同步 |
| JWT | 无状态，验证快，天然适合分布式 | 签发后无法撤销（只能黑名单兜底） |

本项目选 JWT + Redis 黑名单，平衡了性能和精确控制需求。

### 1.2 数据权限设计

**需求**：不同角色看到不同的数据范围（本人/本组/全部）。

**实现**：AOP 切面 + MyBatis 拦截，动态注入 SQL 条件：

```java
@DataScope(deptAlias = "u", userAlias = "u")
public interface OrderMapper {
    List<Order> selectOrderList(OrderQuery query);
}
```

```java
// 生成的 SQL 自动注入数据权限过滤
SELECT * FROM orders o, sys_user u
WHERE o.user_id = u.id
  AND o.dept_id = u.dept_id
  AND u.data_scope = 'PERSONAL'  -- 本人
  AND o.user_id = #{currentUserId}
```

### 1.3 水平权限漏洞防护

**常见攻击**：用户 A 通过修改 URL 参数 ID（如 `/orders/12345`）越权查看用户 B 的订单。

**防护**：

```java
// Controller 层做对象级权限校验
@GetMapping("/orders/{orderId}")
public ApiResult<OrderVO> getOrder(@PathVariable Long orderId) {
    Order order = orderService.getById(orderId);

    // 数据权限校验：即使知道 ID，非本人/本组也查不到
    if (!dataScopeChecker.canAccess(currentUser, order)) {
        throw new AccessDeniedException("无权访问该订单");
    }

    return ApiResult.ok(orderConverter.toVO(order));
}
```

**防御原则**：对象级校验必须在查询后立即执行，不能依赖"列表页筛选不到就安全"。

## 二、Gateway 模式与多环境切换

### 2.1 核心问题

接入第三方支付/短信/推送时，最常见的烂代码：

```java
// ❌ 环境污染：业务代码里充斥着环境判断
if ("test".equals(env)) {
    return mockData;
} else {
    return douyinApiClient.call(params);
}
```

### 2.2 Gateway 契约方案

```
业务层（Service/Controller）
    ↓ 只依赖接口
Gateway 接口（抽象层）
    ↓
Test Gateway 实现 ←→ Real Gateway 实现（运行时注入）
```

**为什么不用 Mock Server？**

| 方案 | 优点 | 缺点 |
|---|---|---|
| Mock Server（WireMock） | 独立进程，可视化配置 | 额外启动，接口变更要同步更新，运行时才知道是否匹配 |
| Gateway + Profile | 编译期类型安全，接口变更编译即报错 | 需要维护两套实现代码 |

### 2.3 依赖注入切换示例

```java
// 接口定义（契约）
public interface DouyinOrderGateway {
```

## 三、订单归因系统设计

### 3.1 归因的业务意义

团长 SaaS 的核心商业模式：

```
团长发布商品链接
    ↓
达人转发推广
    ↓
用户下单 → 订单归因 → 团长/达人分佣
```

**归因要回答的问题**：这一单是通过谁的链接下的？

### 3.2 两通道归因设计

**通道一：外部链接归因（pick_source）**

- 团长给达人发专属链接，带 `pick_source` 参数
- 用户点击链接后下单，URL 参数会随订单透传
- 后端通过 `pick_source` 查 `pick_source_mapping` 表确定业绩归属

**通道二：抖店原生归因（colonel_order_info）**

- 抖店官方通道，用户通过视频/直播间下单时，平台自动传递团长身份信息
- 不需要外部链接点击，直接透传 `colonel_buyin_id`

### 3.3 未归因订单处理

```java
public AttributionResult resolve(OrderRawData raw) {
    // 优先 pick_source 归因
    if (StringUtils.hasText(raw.getPickSource())) {
        Mapping mapping = mappingService.find(raw.getPickSource(),
            raw.getActivityId(), raw.getProductId());
        if (mapping != null) {
            return AttributionResult.attributed(mapping.getTalentId());
        }
        return AttributionResult.unattributed("PICK_SOURCE_NOT_FOUND");
    }

    // 其次抖店原生归因
    if (raw.getColonelInfo() != null) {
        Activity activity = activityService.findByBuyinId(
            raw.getColonelInfo().getColonelBuyinId(),
            raw.getColonelInfo().getActivityId()
        );
        if (activity != null) {
            return AttributionResult.attributed(activity.getTalentId());
        }
        return AttributionResult.unattributed("COLONEL_MAPPING_NOT_FOUND");
    }

    return AttributionResult.unattributed("NO_SOURCE");
}
```

## 四、商品状态机设计

### 4.1 什么是状态机？

状态机 = **状态枚举** + **合法流转规则** + **守卫校验**。

```java
public enum ProductStatus {
    PENDING_AUDIT,  // 待审核
    APPROVED,       // 已通过
    REJECTED,       // 已拒绝
    ASSIGNED,       // 已分配
    LINKED,         // 已生成推广链接
    FOLLOWING       // 跟进中
}

// 状态流转规则
public class ProductStatusMachine {
    private static final Map<ProductStatus, Set<ProductStatus>> TRANSITIONS = Map.of(
        PENDING_AUDIT, Set.of(APPROVED, REJECTED),
        APPROVED,      Set.of(ASSIGNED),
        ASSIGNED,      Set.of(LINKED),
        LINKED,        Set.of(FOLLOWING)
    );

    public void transition(ProductStatus from, ProductStatus to) {
        if (!TRANSITIONS.getOrDefault(from, Set.of()).contains(to)) {
            throw new IllegalStateException(from + " → " + to + " 不合法");
        }
        // 执行流转
    }
}
```

### 4.2 为什么要用状态机？

- **防错**：不通过状态机无法流转，所有流转都经过校验
- **可追溯**：每次流转都记录操作日志
- **易扩展**：新增状态只需改枚举和规则，不改业务逻辑

### 4.3 订单/寄样状态机的区别

| 维度 | 商品状态机 | 寄样状态机 |
|---|---|---|
| 状态数 | 6个 | 7个（含CLOSED超时态） |
| 终态 | FOLLOWING | COMPLETED |
| 特殊机制 | - | 订单命中自动完成、定时超时关闭 |
| 状态回退 | 不允许 | 不允许 |

## 五、缓存设计

### 5.1 缓存策略选择

**Cache-Aside（旁路缓存）**：应用自己负责读写缓存，数据库是数据的最终来源。最常用。

```
读：cache → 有则返回 → 无则查DB → 写cache → 返回
写：更新DB → 删除cache（不是更新cache）
```

**为什么删除而不是更新？**

因为缓存更新的同时如果有并发读，可能读到脏数据。删除缓存后，下次读会从数据库加载到最新数据，不会有脏读问题。

### 5.2 缓存三大问题

**缓存雪崩**：大量 key 同时过期 → 大量请求打到 DB

```java
// 解决：过期时间加随机偏移量
Duration ttl = Duration.ofHours(1).plusMinutes(random.nextInt(30));
```

**缓存击穿**：热点 key 过期 → 大量请求同时查 DB

```java
// 解决：分布式锁 + 双重检查
if (cache.get(key) == null) {
    if (redisLock.tryLock(lockKey)) {
        try {
            // 双重检查
            data = cache.get(key);
            if (data == null) {
                data = db.query(key);
                cache.set(key, data);
            }
        } finally {
            redisLock.unlock(lockKey);
        }
    }
}
```

**缓存穿透**：查询不存在的数据 → 每次都打到 DB

```java
// 解决：布隆过滤器 或 短 TTL 空值缓存
if (data == null) {
    cache.set(key, "EMPTY", Duration.ofMinutes(5)); // 空结果也缓存
}
```

### 5.3 分布式锁

**核心问题**：多实例部署时，定时任务每个实例都会执行。需要保证只有一个实例执行。

```java
public boolean tryLock(String key, Duration ttl) {
    // SET key value NX PX ttl — 原子操作
    return redisTemplate.opsForValue()
        .setIfAbsent(key, UUID.randomUUID().toString(), ttl);
}

public void unlock(String key) {
    // Lua 脚本：只有持有者才能删除
    String script = """
        if redis.call('get', KEYS[1]) == ARGV[1] then
            return redis.call('del', KEYS[1])
        end
        """;
    redisTemplate.execute(new RedisScript<>(script), List.of(key), lockValue);
}
```

**为什么不用数据库表做分布式锁？**

数据库锁是悲观锁，会阻塞事务，高并发下性能差。Redis 分布式锁基于单线程命令执行，性能高几个数量级。

## 六、分库分表与数据库设计

### 6.1 为什么需要分区表？

订单表数据随时间线性增长，查询历史订单时全表扫描慢。

**解决方案**：按月分区，每个分区独立存储和索引。

```sql
CREATE TABLE orders (
    id BIGSERIAL,
    order_id VARCHAR(64),
    create_time TIMESTAMP,
    ...
) PARTITION BY RANGE (create_time);

-- 按月分区
CREATE TABLE orders_2026_05 PARTITION OF orders
    FOR VALUES FROM ('2026-05-01') TO ('2026-06-01');
```

**查询时利用分区裁剪**：

```sql
SELECT * FROM orders WHERE create_time BETWEEN '2026-05-01' AND '2026-05-31';
-- PostgreSQL 自动只扫描 orders_2026_05 分区，不扫描其他分区
```

### 6.2 历史数据清理

```java
// 删除过期分区（DDL，秒级）
public void cleanupOldPartitions(int retentionDays) {
    LocalDate cutoff = LocalDate.now().minusDays(retentionDays);
    List<String> toDrop = getMonthlyPartitionsBefore(cutoff);
    for (String partition : toDrop) {
        jdbcTemplate.execute("DROP TABLE IF EXISTS " + partition);
    }
}
```

**为什么用 DROP TABLE 而不是 DELETE？**

| 操作 | 原理 | 百万条数据耗时 |
|---|---|---|
| `DELETE WHERE` | DML，写 undo/redo，触发触发器 | 几十分钟 |
| `DROP TABLE` | DDL，物理删除文件 | 秒级 |

## 七、高并发场景估算

### 7.1 估算公式

```
QPS = 并发用户数 × 每个用户请求数 / 时间窗口
```

假设：
- 日活 1000 人
- 每人每天平均操作 20 次
- 集中在 4 小时高峰（14:00-18:00）

```
日总请求 = 1000 × 20 = 20,000
QPS = 20,000 / (4 × 3600) ≈ 1.4
峰值 QPS ≈ 1.4 × 5 = 7（留 5 倍 buffer）
```

### 7.2 数据库连接数估算

```
最大并发连接数 = QPS × 平均响应时间(秒)
             = 100 × 0.1 = 10
```

PostgreSQL 默认最大 100 连接，MySQL 默认 151。本项目预估峰值 10-20 并发，足够。

## 八、面试高频追问

**Q: 为什么不用 synchronized 实现分布式锁？**
A: `synchronized` 是 JVM 锁，只在同一 JVM 内有效。多实例部署时有多个 JVM，无法跨进程同步。必须用 Redis/MySQL/ZooKeeper 等中间件实现分布式锁。

**Q: 状态机和 if-else 的区别？**
A: if-else 的流转规则散落在代码各处，新增状态要改多个地方。状态机把所有规则集中在规则表/枚举里，添加新状态只需要加一个枚举值和对应规则，侵入性小。

**Q: 缓存和数据库的一致性怎么保证？**
A: 先更新数据库，再删除缓存（不是更新缓存）。删除缓存后，下次读会从数据库加载最新数据。如果删除缓存失败，可以重试 2-3 次，或者通过 Canal 监听 binlog 异步同步。

**Q: 订单归因失败率高怎么排查？**
A: 先看归因失败原因分布（NO_SOURCE / PICK_SOURCE_NOT_FOUND / COLONEL_MAPPING_NOT_FOUND）。如果是 COLONEL_MAPPING_NOT_FOUND 占比高，说明抖店归因了但本地没有对应活动映射，需要补录活动数据。如果 NO_SOURCE 占比高，说明大量自然流量订单，这是业务问题不是系统问题。
