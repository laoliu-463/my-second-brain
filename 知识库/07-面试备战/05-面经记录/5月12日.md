# **项目 1：AI 面试助手 / 基于 RAG 的简历分析**

---

### **问题 1：项目整体是什么？为什么要做？**

**面试官问法**：

> “能详细介绍一下你的 AI 面试助手吗？它主要解决什么问题，为什么要做这个项目？”

**顺口回答**：

> 我的项目是一个基于 RAG 的 AI 简历分析和面试辅助系统。用户上传 PDF 或 DOCX 简历后，系统会先解析文本、清洗，然后按段落或固定长度切分成多个 chunk，并通过 embedding 模型生成向量，存入 PostgreSQL + pgvector。  
> 当用户提出问题时，系统会先把问题向量化，检索最相关的简历片段，再把片段和问题一起作为 Prompt 输入大模型生成回答。
> 
> 我做这个项目是为了让 AI 回答尽量精准，基于真实简历内容提供面试建议，避免凭空生成信息，同时展示我对后端微服务、缓存、异步任务处理的理解和实践。

**关键技术点 / 场景说明**：

- RAG 流程：检索增强生成
    
- 向量化存储 + pgvector
    
- 异步任务 + Redis Stream
    
- 项目场景：简历分析 + AI 回答
    

---

### **问题 2：RAG 是什么？项目里具体怎么用？**

**面试官问法**：

> “你在简历里写了 RAG，这个概念是什么？你是怎么在项目里落地的？”

**顺口回答**：

> RAG 是 Retrieval-Augmented Generation，中文叫检索增强生成。它的核心思想是先从知识库检索相关内容，再把检索结果与问题一起输入大模型生成回答。
> 
> 在我的项目中，知识库就是用户上传的简历。用户提问时，我先把问题向量化，然后检索出最相关的几个 chunk，把这些内容与问题拼成 Prompt 输入 AI 模型，这样生成的回答更精准，也能给用户具体的面试改进建议。

**关键技术点 / 场景说明**：

- 避免 AI 幻觉
    
- 精准回答问题
    
- 向量检索 + Prompt 拼接
    
- TopK 控制检索数量
    

---

### **问题 3：为什么用 pgvector？**

**面试官问法**：

> “你为什么用 PostgreSQL + pgvector，而不是 Milvus 或 Elasticsearch 这样的向量数据库？”

**顺口回答**：

> pgvector 是 PostgreSQL 的向量扩展，可以直接存储向量并支持相似度检索。  
> 我项目数据量属于中等规模，用 pgvector 可以轻量落地，不需要额外搭建 Milvus 或 Elasticsearch，减少复杂度，同时和我已有 PostgreSQL 数据库兼容，方便管理。

**关键技术点 / 场景说明**：

- 技术选型能力
    
- 轻量可落地
    
- 避免不必要复杂组件
    

---

### **问题 4：文本切分为什么要做？**

**面试官问法**：

> “你项目里把简历文本分成 chunk，为什么要这样处理？如果不切分会怎样？”

**顺口回答**：

> 简历可能很长，如果直接把整篇文本向量化，粒度太粗，检索时很难准确命中用户想问的内容。  
> 切分成段落或固定长度的 chunk，可以提高向量检索的准确性，同时保留上下文信息。  
> 在项目里，我结合 chunk 大小和 topK 数量控制，确保检索结果最相关，提高回答准确度。

**关键技术点 / 场景说明**：

- 检索精度
    
- RAG 实现关键步骤
    
- Prompt 输入内容控制
    

---

### **问题 5：Redis Stream 在项目里干什么？**

**面试官问法**：

> “你项目里用了 Redis Stream，它具体解决了什么问题？为什么不用 MQ？”

**顺口回答**：

> 用户上传简历后，文件解析、文本切分、向量化、报告生成等步骤都比较耗时，如果同步执行，接口响应会很慢。
> 
> 所以我用 Redis Stream 做异步任务处理，消费者异步消费任务，用户接口能快速返回，保证良好体验。
> 
> 选择 Redis Stream 而不是 MQ，是因为项目本身已经使用 Redis，轻量足够处理中小规模任务，减少额外组件复杂度。

**关键技术点 / 场景说明**：

- 异步任务处理
    
- 技术选型（轻量 vs MQ）
    
- 提升用户体验
    

---

### **问题 6：如果 AI 回答不准怎么办？**

**面试官问法**：

> “AI 回答可能不准，你是怎么控制输出的？”

**顺口回答**：

> 我从三个方面控制：
> 
> 1. 先向量检索，确保模型回答基于相关简历片段
>     
> 2. Prompt 中明确要求“只能根据提供的简历内容回答，不确定就说明无法判断”
>     
> 3. 输出结构化，要求模型给出优势、风险点、改进建议
>     
> 
> 这样既提高准确度，也能提供可操作的面试建议。

**关键技术点 / 场景说明**：

- 输出控制
    
- RAG + Prompt 约束
    
- 结构化输出

### **问题 7：项目难点是什么？怎么解决？**

**面试官问法**：

> “项目里遇到过什么技术难点？你是怎么解决的？”

**顺口回答**：

> 难点一是向量检索精度不够，最初 chunk 太大或 topK 设置不合理，检索结果不精准。解决方法是按段落切分，并控制 topK 数量，保证检索片段相关性。  
> 难点二是接口耗时问题，解析 + 向量化 + 报告生成任务慢。我通过 Redis Stream 异步任务 + 缓存优化解决，同时接口快速返回。
> 
> 这样既保证 AI 回答精准，也提升系统响应速度。

**关键技术点 / 场景说明**：

- 问题定位能力
    
- 异步任务处理
    
- 检索策略调整

### **Step 2b：核心微服务组件作用**

|组件|作用（面试顺口版）|
|---|---|
|**Nacos**|服务注册与发现，以及配置中心，服务启动后注册信息，其他服务通过服务名调用|
|**Gateway**|统一网关，处理请求入口、路由转发、鉴权、限流等公共逻辑|
|**OpenFeign**|声明式 HTTP 客户端，服务间调用就像调用本地方法|
|**Sentinel**|流量控制、熔断降级，保护系统不被瞬时高流量或下游异常拖垮|
|**Seata**|分布式事务管理，保证跨服务操作的一致性，例如报名课程同时创建订单和扣减课时|

> 你只要理解每个组件“在项目里解决什么问题”，不需要讲底层原理太深。

---
