AI面试项目

RAG 是 Retrieval-Augmented Generation，也就是检索增强生成。在我的项目里，用户上传简历后，系统会先解析 PDF 或 DOCX 文件，将文本清洗并按段落或固定长度切分成多个 chunk。

每个 chunk 会通过 embedding 模型生成向量，然后存入 PostgreSQL + pgvector 中，方便后续做向量相似度检索。

当用户提出问题时，系统会先将问题向量化，然后检索最相关的简历片段，再把这些内容和问题一起作为 Prompt 输入大模型生成回答。

这样，AI 的回答不仅精准，而且基于用户实际简历内容，避免了胡说。

在项目中，用户上传简历后，会有几个耗时操作，比如文件解析、文本切分、向量化以及报告生成。

如果这些步骤都同步执行，接口响应会很慢，用户体验不好。

所以我使用 Redis Stream，把这些任务放到异步队列里处理。消费者异步消费任务，完成向量化和报告生成，同时不阻塞用户请求。

我之所以选择 Redis Stream 而不是专门的消息队列，是因为项目本身已经使用 Redis，它轻量且足够处理中小规模异步任务，减少了额外组件复杂度。
我从三个方面保证 AI 回答尽量准确：

1. 先做向量检索，让大模型基于相关简历片段回答，而不是凭空生成；
2. 在 Prompt 中明确要求模型“只能根据提供的简历内容回答，不确定就说明无法判断”；
3. 对输出做结构化约束，比如要求返回优势、风险点和改进建议，这样生成内容更可控。

### **Step 2b：核心微服务组件作用**

|组件|作用（面试顺口版）|
|---|---|
|**Nacos**|服务注册与发现，以及配置中心，服务启动后注册信息，其他服务通过服务名调用|
|**Gateway**|统一网关，处理请求入口、路由转发、鉴权、限流等公共逻辑|
|**OpenFeign**|声明式 HTTP 客户端，服务间调用就像调用本地方法|
|**Sentinel**|流量控制、熔断降级，保护系统不被瞬时高流量或下游异常拖垮|
|**Seata**|分布式事务管理，保证跨服务操作的一致性，例如报名课程同时创建订单和扣减课时|

> 你只要理解每个组件“在项目里解决什么问题”，不需要讲底层原理太深。
